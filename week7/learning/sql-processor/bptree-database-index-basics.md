## 2. 먼저 인덱스가 왜 필요한가

### 2.1 책 맨 뒤의 색인을 떠올리면 쉽다

데이터베이스 인덱스는 책 맨 뒤의 `색인`과 비슷하다.

예를 들어 1000페이지짜리 책에서 `가상 메모리`라는 단어를 찾고 싶다고 하자.

인덱스가 없으면 아래처럼 해야 한다.

- 1페이지를 본다
- 2페이지를 본다
- 3페이지를 본다
- ...
- 원하는 단어가 나올 때까지 계속 넘긴다

이 방식은 가능은 하지만 매우 비효율적이다.

반면 책 뒤의 색인이 있으면 아래처럼 찾을 수 있다.

- 색인에서 `가상 메모리`를 찾는다
- 해당 단어가 나오는 페이지 번호를 확인한다
- 그 페이지로 바로 이동한다

즉, 인덱스는 `원본 전체를 처음부터 끝까지 읽지 않고도, 원하는 위치로 더 빨리 가게 해 주는 보조 구조`다.

### 2.2 데이터베이스에서도 문제는 같다

예를 들어 `users` 테이블에 1,000,000개의 레코드가 있다고 하자.

이때 아래 질의를 실행한다고 생각해 보자.

```sql
SELECT name FROM users WHERE id = 800000;
```

인덱스가 없으면 데이터베이스는 극단적으로 말해 아래처럼 행동할 수 있다.

- 첫 번째 레코드의 `id`를 본다
- 두 번째 레코드의 `id`를 본다
- 세 번째 레코드의 `id`를 본다
- ...
- `id = 800000`을 찾을 때까지 계속 확인한다

이것이 `선형 탐색` 또는 `풀 스캔`에 가까운 방식이다.

데이터가 적을 때는 괜찮아 보일 수 있다.
하지만 데이터가 많아질수록 시간이 급격히 늘어난다.

그래서 인덱스가 필요하다.

## 3. 인덱스가 하는 일을 한 문장으로 정리하면

인덱스는 아래 한 줄로 이해할 수 있다.

> `찾을 때 자주 사용하는 값`을 기준으로, `실제 데이터 위치를 빠르게 찾아가는 길잡이 구조`를 따로 만들어 둔 것이다.

여기서 중요한 점은 두 가지다.

- 인덱스는 `실제 데이터 그 자체`가 아니라, 데이터를 찾기 위한 `별도의 구조`다
- 인덱스는 보통 `키 -> 데이터 위치` 또는 `키 -> 데이터 참조` 형태로 생각할 수 있다

예를 들어 아래처럼 볼 수 있다.

```text
800000 -> users 테이블의 어떤 위치
```

즉, 인덱스는 "800000이라는 ID가 있나?"만 알려 주는 것이 아니라,
"그 레코드가 어디 있는가?"까지 연결해 주는 역할을 한다.

## 4. 왜 단순한 구조로는 부족한가

### 4.1 배열

배열은 인덱스로 쓰기에 무조건 나쁜 구조는 아니다.
정렬되어 있다면 이진 탐색도 가능하다.

하지만 데이터베이스처럼 레코드 삽입과 삭제가 자주 일어나면 문제가 생긴다.

- 정렬 상태를 유지해야 한다
- 중간 삽입 시 많은 데이터를 밀어야 할 수 있다
- 데이터가 매우 크면 재배치 비용이 커진다

즉, "찾기"는 빠르게 만들 수 있어도 "계속 데이터가 바뀌는 환경"에서는 불편해질 수 있다.

### 4.2 연결 리스트

연결 리스트는 중간 삽입은 편할 수 있다.
하지만 검색은 거의 처음부터 따라가야 한다.

- 첫 노드 확인
- 다음 노드 확인
- 또 다음 노드 확인

즉, 찾는 속도가 느리다.

### 4.3 해시 테이블

해시 테이블은 `정확히 같은 값 찾기`에는 매우 강력하다.
예를 들어 `id = 100` 같은 단건 조회에는 잘 맞을 수 있다.

하지만 데이터베이스 인덱스는 이것보다 더 다양한 요구를 받는다.

- `id = 100` 찾기
- `id >= 100 AND id < 200` 같은 범위 찾기
- 정렬된 순서대로 읽기
- 다음 값, 이전 값으로 자연스럽게 이동하기

해시 테이블은 범위 조회나 정렬 순서 유지에 불리하다.

### 4.4 이진 탐색 트리

이진 탐색 트리는 왼쪽 자식과 오른쪽 자식 두 방향으로만 내려간다.
기본 아이디어는 좋지만, 균형이 깨지면 한쪽으로 길게 늘어진 구조가 될 수 있다.

예를 들어 정렬된 값이 계속 들어오면 아래처럼 기울 수 있다.

```text
1
 \
  2
   \
    3
     \
      4
```

이렇게 되면 결국 거의 선형 탐색처럼 느려질 수 있다.

그래서 데이터베이스는 대량 데이터와 빈번한 변경에도 안정적으로 동작하는,
더 균형 잡힌 구조를 원한다.

## 5. 트리 자료구조를 아주 짧게 복습하기

트리는 부모-자식 관계로 연결된 자료구조다.

예를 들면 아래와 같다.

```text
        A
      / | \
     B  C  D
```

트리에서 중요한 개념은 아래 정도다.

- 루트(root): 맨 위의 시작 노드
- 리프(leaf): 더 이상 자식이 없는 노드
- 높이(height): 루트에서 리프까지 몇 단계 내려가는가

검색 구조로서 트리가 좋은 이유는,
루트에서 시작해 비교를 하며 아래로 내려가면
전체 데이터를 다 보지 않고도 점점 범위를 좁힐 수 있기 때문이다.

## 6. B-Tree와 B+ Tree는 무엇이 다른가

이 둘은 이름이 비슷해서 자주 헷갈린다.

### 6.1 B-Tree

B-Tree는 내부 노드와 리프 노드 모두 데이터를 가질 수 있다.

즉, 검색하다가 중간 내부 노드에서 바로 원하는 데이터를 찾을 수도 있다.

### 6.2 B+ Tree

B+ Tree는 보통 아래처럼 생각하면 된다.

- 내부 노드: 길 안내 표지판
- 리프 노드: 실제 데이터 참조가 모여 있는 곳

즉, 내부 노드는 주로 "어느 방향으로 내려갈지" 결정하는 용도다.
실제 키에 대응하는 값이나 레코드 참조는 리프 노드에 모아 둔다.

### 6.3 왜 B+ Tree가 더 많이 언급되는가

데이터베이스 인덱스에서는 B+ Tree가 특히 잘 맞는다.
이유는 아래와 같다.

- 실제 데이터 참조를 리프에 모아 구조가 깔끔하다
- 리프 노드들을 순서대로 연결하기 쉽다
- 범위 조회와 순차 조회에 유리하다
- 내부 노드는 더 많은 키를 담아 검색 경로를 줄이기 좋다

즉, B+ Tree는 "검색용 길 안내"와 "실제 데이터 위치 정보"를 분리해 관리하기 좋은 구조다.

## 7. B+ Tree를 아주 쉽게 그려 보면

예를 들어 아래와 같은 B+ Tree가 있다고 생각해 보자.

```text
             [30 | 70]
            /    |    \
   [10 | 20] [30 | 50] [70 | 90]
```

이 그림을 아주 단순화해서 해석하면 아래와 같다.

- 루트 노드의 `30`, `70`은 "탐색 방향을 나누는 기준"이다
- 왼쪽 리프는 30보다 작은 값들
- 가운데 리프는 30 이상 70 미만 값들
- 오른쪽 리프는 70 이상 값들

만약 `50`을 찾고 싶다면,
루트에서 `50`이 `30` 이상 `70` 미만이라는 것을 보고
가운데 리프로 내려간다.

즉, 한 번의 비교로 전체 범위를 크게 줄인다.

여기서 `30`, `70` 같은 값이 내부 노드와 리프에 둘 다 보일 수 있는데,
이상한 것이 아니다.
내부 노드의 키는 실제 레코드 자체라기보다
`아래 자식들이 어떤 범위를 담당하는지 알려 주는 경계 값`으로 이해하면 된다.

## 8. B+ Tree의 핵심 구성 요소

### 8.1 노드(node)

B+ Tree는 여러 개의 노드로 이루어진다.
노드는 크게 두 종류로 나뉜다.

- 내부 노드
- 리프 노드

### 8.2 내부 노드(internal node)

내부 노드는 보통 아래 정보를 가진다.

- 여러 개의 키
- 자식 노드로 가는 포인터

내부 노드의 역할은 실제 데이터를 저장하는 것이 아니라,
`어느 자식 쪽으로 내려가야 하는지 결정하는 것`이다.

즉, 내부 노드는 지도에서 갈림길 표지판과 비슷하다.

### 8.3 리프 노드(leaf node)

리프 노드는 실제 키에 대응하는 값 또는 레코드 참조를 가진다.

예를 들면 아래처럼 생각할 수 있다.

```text
[100 -> row#15, 120 -> row#18, 150 -> row#22]
```

즉, 리프는 "실제 검색 결과에 가장 가까운 정보"를 가지고 있다.

### 8.4 차수(order)와 fan-out

B+ Tree에서는 한 노드가 가질 수 있는 키 수와 자식 수에 제한이 있다.
이 제한을 설명할 때 `차수`, `order`, `fan-out`, `최대 자식 수`, `최대 키 수` 같은 표현이 나온다.

다만 주의할 점이 있다.
책이나 자료마다 `order`를 정의하는 방식이 조금씩 다르다.
어떤 자료는 `자식 수 기준`, 어떤 자료는 `키 수 기준`으로 설명한다.

반면 `fan-out`은 보통
`한 내부 노드가 몇 개의 자식으로 뻗을 수 있는가`라는 분기 규모를 뜻한다고 이해하면 된다.

비전공자 입장에서는 지금 단계에서 정확한 공식보다 아래 감각이 더 중요하다.

- 한 노드는 키를 1개만 담는 것이 아니다
- 여러 개의 키와 여러 개의 자식을 가진다
- 그래서 한 번 내려갈 때 많은 범위를 한꺼번에 줄일 수 있다
- 그래서 데이터가 커져도 트리 높이를 낮게 유지하기 좋다

이 점이 B+ Tree가 이진 트리보다 DB 인덱스에 더 잘 맞는 중요한 이유 중 하나다.

### 8.5 리프 노드 연결

B+ Tree의 큰 특징 중 하나는 리프 노드들이 순서대로 연결되기 쉽다는 점이다.

예를 들어 아래처럼 볼 수 있다.

```text
[10 | 20] <-> [30 | 50] <-> [70 | 90]
```

이 연결 덕분에 아래가 쉬워진다.

- 작은 값부터 큰 값 순서대로 읽기
- 특정 범위의 값을 연속으로 읽기

이것이 나중에 범위 조회에서 큰 장점이 된다.

## 9. B+ Tree에서 검색은 어떻게 일어나는가

예를 들어 아래 트리에서 `90`을 찾는다고 하자.

```text
             [30 | 70]
            /    |    \
   [10 | 20] [30 | 50] [70 | 90]
```

검색 흐름은 아래와 같다.

1. 루트에서 시작한다.
2. `90`은 `70`보다 크거나 같으므로 오른쪽 자식으로 간다.
3. 오른쪽 리프 노드에서 `90`을 찾는다.
4. `90`에 연결된 레코드 참조를 얻는다.

핵심은 아래다.

- 전체 데이터를 다 보지 않는다
- 비교를 통해 내려갈 방향만 결정한다
- 트리 높이가 낮으면 매우 적은 단계로 찾을 수 있다

즉, 데이터가 많아져도 검색 경로가 급격히 길어지지 않게 만들 수 있다.

## 10. 삽입은 왜 더 복잡한가

검색은 내려가서 찾으면 끝나지만,
삽입은 구조를 유지해야 하므로 더 어렵다.

### 10.1 기본 삽입 흐름

삽입은 보통 아래 순서로 생각한다.

1. 새 키가 들어갈 리프 노드를 찾는다
2. 정렬 순서를 유지하며 삽입한다
3. 노드가 넘치지 않으면 끝낸다
4. 노드가 가득 찼다면 split한다
5. 부모에 새로운 경계 키를 올린다
6. 부모도 넘치면 다시 split한다

### 10.2 split이란 무엇인가

split은 노드가 너무 꽉 차서 둘로 나누는 작업이다.

예를 들어 리프가 아래처럼 꽉 찼다고 해 보자.

```text
[10 | 20 | 30 | 40]
```

여기에 `50`을 넣어야 하는데 공간이 없다면,
둘로 나눌 수 있다.

```text
[10 | 20]   [30 | 40 | 50]
```

그리고 부모는 "이제 30을 기준으로 두 리프로 갈라진다"는 사실을 알아야 한다.

즉, split은 단순히 배열을 둘로 쪼개는 것이 아니라,
`부모와 자식의 관계까지 함께 갱신하는 작업`이다.

### 10.3 왜 split이 중요한가

split 덕분에 B+ Tree는 한쪽으로 길게 무너지지 않고,
높이를 비교적 낮고 균형 있게 유지할 수 있다.

즉, 삽입이 계속 일어나도 검색 속도가 안정적으로 유지되는 이유가 여기에 있다.

## 11. B+ Tree가 데이터베이스 인덱스로 잘 맞는 이유

이제 핵심 질문으로 들어가 보자.

> 왜 많은 데이터베이스가 인덱스 구조로 B+ Tree 계열을 사용해 왔는가?

답은 하나가 아니라 여러 이유가 겹친 결과다.

### 11.1 트리 높이가 낮아지기 쉽다

B+ Tree는 한 노드가 여러 개의 키를 담는다.
즉, 한 단계에서 많은 범위를 나눌 수 있다.

예를 들어 이진 트리는 한 번 내려갈 때 갈림길이 최대 2개지만,
B+ Tree는 한 번 내려갈 때 훨씬 더 많은 갈림길을 가질 수 있다.

이 말은 곧 아래 뜻이다.

- 데이터가 많아도 트리 높이가 너무 커지지 않는다
- 검색 시 내려가는 횟수가 줄어든다

데이터베이스는 레코드 수가 매우 많기 때문에,
이 특성이 특히 중요하다.

### 11.2 디스크와 페이지 구조에 잘 맞는다

실제 DBMS는 데이터를 메모리뿐 아니라 디스크와 페이지 단위로 다룬다.
여기서 페이지(page)는 데이터를 한 덩어리로 묶어 읽고 쓰는 기본 단위라고 생각하면 된다.

비전공자 기준으로는 이렇게 이해하면 된다.

- 디스크 접근은 메모리 접근보다 훨씬 비싸다
- 그래서 "몇 번 내려가야 하느냐"가 매우 중요하다
- B+ Tree는 한 노드에 많은 키를 담아 트리 높이를 줄이기 좋다

즉, B+ Tree는 `디스크에서 적은 횟수로 원하는 위치에 도달하도록` 돕는 구조다.

이번 주 과제는 메모리 기반 단순화 버전이지만,
원래 DB에서 B+ Tree가 사랑받는 이유는 이 디스크 친화성과 강하게 연결된다.

### 11.3 범위 조회에 유리하다

데이터베이스는 딱 한 건만 찾는 것이 전부가 아니다.

예를 들어 아래 같은 질의도 많다.

```sql
SELECT * FROM users WHERE id >= 100 AND id < 200;
```

B+ Tree는 리프가 순서대로 연결되기 쉬워서,
시작 위치만 찾은 뒤 그다음 값들을 순서대로 읽기 좋다.

즉, 아래가 자연스럽다.

- 시작 키를 찾는다
- 해당 리프에서 읽는다
- 다음 리프로 넘어간다
- 필요한 범위까지 계속 읽는다

해시 테이블보다 B+ Tree가 DB 인덱스에서 자주 선택되는 큰 이유 중 하나다.

### 11.4 정렬된 순서를 유지하기 쉽다

리프 노드의 키들은 정렬된 순서로 저장된다.

그래서 아래 작업에 유리하다.

- 오름차순 조회
- 범위 조회
- 다음 키, 이전 키 탐색

즉, `정확한 값 검색`과 `순서 기반 조회`를 함께 만족시키기 좋다.

### 11.5 삽입과 삭제가 있어도 균형을 유지한다

데이터베이스는 조회만 하는 시스템이 아니다.
삽입, 삭제, 수정도 계속 일어난다.

B+ Tree는 split, merge, redistribution 같은 동작을 통해
구조를 균형 있게 유지하려고 한다.

즉, 데이터가 바뀌더라도 검색 성능이 한쪽으로 심하게 무너지지 않도록 설계된 구조다.

## 12. 데이터베이스에서 인덱스는 실제로 무엇을 저장하는가

비전공자가 자주 헷갈리는 부분이 여기다.

> 인덱스는 레코드 전체를 저장하는가, 아니면 위치만 저장하는가?

답은 시스템에 따라 다를 수 있지만,
입문 단계에서는 아래처럼 이해하면 충분하다.

- 인덱스 키
- 그 키에 대응하는 데이터 위치 정보 또는 데이터 참조

예를 들어 아래처럼 생각할 수 있다.

```text
id = 100 -> users 테이블의 37번째 행
id = 120 -> users 테이블의 파일 오프셋 2048
id = 150 -> users 테이블의 페이지 7, 슬롯 3
```

여기서 `슬롯(slot)`은
`한 페이지 안에서 몇 번째 레코드 칸인가`를 가리키는 번호 정도로 이해하면 된다.

즉, 인덱스는 보통 아래 역할을 한다.

- 키를 빠르게 찾는다
- 실제 데이터가 있는 위치로 안내한다

이번 주 과제에서도 이 감각이 중요하다.
B+ Tree가 `id`를 찾고,
그 결과로 실제 레코드를 다시 얻을 수 있어야 한다.

이번 주처럼 메모리 기반으로 단순화한 구현에서는
`배열 인덱스`, `행 번호`, `메모리 상 포인터` 같은 방식을 써도 괜찮다.
다만 실제 DBMS 설명과 학습용 과제 구현은 구분해서 생각하는 편이 좋다.
특히 `메모리 상 포인터`는 이번 주 같은 학습용 메모리 기반 구현에서만 자연스러운 예시이고,
일반적인 DBMS 인덱스의 대표 형태로 받아들이면 안 된다.

### 12.1 리프 노드가 실제 행을 담는 경우와 위치 참조만 담는 경우

시스템에 따라 리프 노드가 담는 정보는 다를 수 있다.
아주 단순화하면 아래 두 방식으로 이해할 수 있다.

- `키 + 행 전체`
- `키 + 행 위치 정보(또는 참조)`

`키 + 행 전체` 방식은 해당 인덱스만 보고 결과를 얻기 쉬울 수 있다.
하지만 인덱스 크기가 커질 수 있다.
이 그림은 나중에 더 공부할 때 `clustered`에 가까운 감각으로 연결해서 볼 수 있다.

`키 + 행 위치 정보` 방식은 인덱스가 상대적으로 가볍다.
대신 실제 레코드를 다시 읽는 단계가 한 번 더 필요할 수 있다.
이쪽은 대략 `secondary index`에 가까운 그림으로 생각해 볼 수 있다.

실제 DBMS는 저장 구조에 따라 여러 변형을 사용한다.
이번 주 과제에서는 우선
`id -> 레코드를 다시 찾을 수 있는 정보` 정도로 이해하면 충분하다.

## 13. `WHERE id = ?`가 왜 빨라지는가

이제 이번 주 과제와 직접 연결되는 부분이다.

예를 들어 아래 질의를 생각해 보자.

```sql
SELECT name FROM users WHERE id = 800000;
```

### 13.1 인덱스가 없는 경우

가능한 흐름은 아래와 같다.

```text
첫 번째 레코드의 id 확인
-> 두 번째 레코드의 id 확인
-> 세 번째 레코드의 id 확인
-> ...
-> 800000을 찾을 때까지 반복
```

즉, 데이터가 많을수록 오래 걸릴 수 있다.

### 13.2 B+ Tree 인덱스가 있는 경우

흐름은 아래처럼 달라진다.

```text
B+ Tree 루트에서 시작
-> 비교를 통해 적절한 자식으로 내려감
-> 리프에서 id = 800000 찾음
-> 해당 레코드 위치 정보 획득
-> 실제 레코드 읽음
```

핵심은 아래다.

- 모든 레코드를 직접 훑지 않는다
- 비교를 통해 후보 범위를 빠르게 줄인다
- 최종적으로 매우 적은 단계만 거쳐 목표에 도달한다

그래서 `WHERE id = ?` 같은 키 기반 조회가 빨라진다.

---

## 14. 그럼 왜 모든 조회가 다 빨라지는 것은 아닌가

중요한 점은, 인덱스는 `인덱스를 만든 기준 컬럼`에 특히 강하다는 것이다.

예를 들어 `id`에 인덱스를 만들었다면 아래는 유리하다.

```sql
SELECT name FROM users WHERE id = 100;
```

하지만 아래는 다르다.

```sql
SELECT name FROM users WHERE name = 'jungle';
```

`name`에 대한 인덱스가 없다면,
여전히 많은 레코드를 확인해야 할 수 있다.

즉, 이번 주 과제에서 `WHERE id = ?` 조회와
다른 필드 기준 조회를 비교하는 이유가 여기에 있다.

인덱스는 "아무 조회나 다 빠르게 만드는 마법"이 아니라,
`특정 키 기준 탐색을 빠르게 만드는 구조`다.

### 14.1 왜 `id`가 인덱스 키로 자주 쓰이는가

이번 주 과제에서 `id`가 핵심인 이유는 아래와 같다.

- 보통 각 레코드를 유일하게 구분하기 쉽다
- `WHERE id = ?`처럼 정확히 한 건을 찾는 질의와 잘 맞는다
- 숫자형 증가 값으로 관리하면 비교와 정렬이 자연스럽다
- 자동 증가 정책과 연결하기 쉽다
- 값이 레코드를 잘 구분해 주는 편이라 선택도(selectivity) 관점에서도 유리한 경우가 많다

즉, `id`는 `잘 구분되고`, `잘 비교되고`, `잘 찾을 수 있는` 키라서
입문용 인덱스 과제에 특히 적합하다.

물론 실제 서비스에서는 `email`, `created_at`, 복합 키 등
다른 인덱스가 더 중요할 수도 있다.
하지만 이번 주 과제처럼 `자동 ID + WHERE id = ?`를 연결해 보기에는
`id`가 가장 직관적이다.

## 15. B+ Tree 인덱스의 장점과 비용

### 15.1 장점

- 정확한 키 조회가 빠르다
- 범위 조회가 유리하다
- 정렬 순서를 유지하기 좋다
- 데이터가 커져도 비교적 안정적인 검색 성능을 기대할 수 있다
- 디스크 기반 DB 구조와 잘 맞는다

### 15.2 비용

- 인덱스 자체를 저장할 추가 공간이 필요하다
- 삽입 시 데이터만 쓰는 것이 아니라 인덱스도 갱신해야 한다
- 삭제와 수정도 더 복잡해질 수 있다
- 구현 난도가 높다

즉, 인덱스는 공짜가 아니다.
읽기 성능을 얻는 대신, 공간과 갱신 비용을 함께 부담한다.

이 감각은 실제 데이터베이스를 이해할 때도 매우 중요하다.

### 15.3 왜 모든 컬럼에 인덱스를 만들지 않는가

처음 보면 `"빠르면 좋은데 모든 컬럼에 인덱스를 만들면 되지 않나?"`라고 생각하기 쉽다.
하지만 인덱스는 추가 구조이기 때문에, 쓰기 작업마다 함께 갱신해야 한다.

예를 들어 `users` 테이블에 아래 인덱스를 모두 만들었다고 하자.

- `id`
- `name`
- `email`
- `city`
- `created_at`

이 상태에서 새 행 1개를 `INSERT`하면
실제 데이터만 저장하는 것이 아니라 여러 인덱스도 함께 갱신해야 할 수 있다.

즉, 인덱스가 많아질수록 아래 비용이 커질 수 있다.

- 저장 공간이 더 든다
- `INSERT`, `UPDATE`, `DELETE`가 느려질 수 있다
- 관리해야 할 구조가 많아진다

그래서 실제 데이터베이스에서는
`자주 검색하거나 정렬하거나 조인에 쓰는 컬럼` 중심으로 인덱스를 고른다.
또한 값이 너무 비슷한 컬럼보다,
`값이 레코드를 잘 구분해 주는 컬럼`이 인덱스 효과가 더 큰 편이다.

그리고 인덱스를 만들었다고 해서 항상 그 인덱스를 사용하는 것도 아니다.
테이블이 아주 작거나, 조건에 맞는 행이 너무 많아서 대부분의 레코드를 읽어야 한다면,
DB는 인덱스 대신 전체 스캔이 더 낫다고 판단할 수도 있다.

---

## 16. 실제 DBMS의 B+ Tree와 이번 과제의 B+ Tree는 무엇이 다른가

이번 주 과제 공지에는 `실제 DBMS에서 사용하는 디스크 기반 B+ Tree를 단순화한 메모리 기반 방식`이라고 적혀 있다.

이 말은 매우 중요하다.

### 16.1 실제 DBMS는 더 복잡하다

실제 DBMS에서는 아래 요소들이 함께 얽힌다.

- 디스크 페이지 관리
- 버퍼 풀
- 페이지 분할
- 동시성 제어
- 복구 로그
- 크래시 안전성

즉, 상용 DB의 인덱스는 자료구조 하나만 잘 짠다고 끝나지 않는다.

### 16.2 이번 과제는 학습용 단순화 버전이다

이번 과제에서는 우선 아래에 집중하면 된다.

- B+ Tree의 핵심 삽입/검색 이해
- `id` 기반 인덱스 구성
- 기존 SQL 처리기와 연결
- 인덱스 조회와 선형 탐색의 속도 차이 확인

즉, 이번 주는 "상용 DB 엔진 완성"이 아니라,
`인덱스가 왜 필요한지 직접 구현으로 체감하는 것`이 목표다.

---

## 17. 이번 주 과제와 연결해서 이 문서에서 가져가야 할 것

이번 과제와 연결해서 이 문서에서 가져가야 할 핵심은 아래 정도면 충분하다.

- B+ Tree는 `id` 같은 키를 기준으로 레코드 위치를 빠르게 찾기 위한 인덱스 구조다
- 리프 노드에는 레코드 전체보다 `레코드를 다시 찾을 수 있는 정보`를 두는 그림이 자연스럽다
- `WHERE id = ?`가 빨라지는 이유를 선형 탐색과 비교해 설명할 수 있어야 한다
- 실제 DBMS는 더 복잡하지만, 이번 과제는 그 핵심 아이디어를 단순화해 체험하는 단계다

즉, 이 문서는 `왜 B+ Tree가 인덱스로 쓰이는가`를 이해하는 데 집중하고,
구체적인 구현 요구사항과 설계 순서는 `bptree-index-assignment-guide.md`에서 이어서 보는 편이 자연스럽다.

## 18. 비전공자가 특히 헷갈리기 쉬운 포인트

### 18.1 "트리가 있으니 데이터도 전부 트리 안에 들어 있는 것 아닌가?"

반드시 그렇지는 않다.
입문 단계에서는 `트리에는 키와 데이터 위치 정보가 들어 있다`고 생각하는 편이 이해하기 쉽다.

즉, 트리는 전체 저장소를 대신하는 것이 아니라,
저장소를 더 빨리 찾게 해 주는 보조 구조다.

### 18.2 "B+ Tree면 무조건 최고인가?"

아니다.
해시 인덱스가 더 적합한 경우도 있고,
워크로드에 따라 다른 구조가 더 나을 수도 있다.

하지만 데이터베이스가 자주 요구하는 아래 조건을 함께 만족시키기 좋아서
B+ Tree가 매우 널리 쓰인다.

- 정확한 값 조회
- 범위 조회
- 순서 기반 조회
- 대용량 데이터 처리

### 18.3 "메모리 기반 구현이면 DB 인덱스 학습과 상관없는 것 아닌가?"

그렇지 않다.
메모리 기반 구현은 실제 시스템을 단순화한 학습 단계라고 보면 된다.

핵심은 아래를 직접 경험하는 것이다.

- 키 기반 탐색 구조
- 삽입 시 인덱스 갱신
- 인덱스 검색과 선형 탐색의 차이
- 자료구조와 SQL 실행 흐름의 연결

즉, 단순화되었어도 핵심 아이디어는 충분히 배울 수 있다.

---

## 19. B+ Tree를 이해할 때 꼭 기억할 핵심 문장

아래 문장만 기억해도 큰 흐름을 잡는 데 도움이 된다.

- 인덱스는 데이터를 빨리 찾기 위한 별도의 길잡이 구조다.
- B+ Tree는 내부 노드로 방향을 정하고, 리프 노드에 실제 데이터 참조를 모은다.
- 한 노드가 여러 키를 담을 수 있어서 트리 높이를 낮게 유지하기 좋다.
- 리프 노드가 순서대로 이어져 범위 조회에 유리하다.
- 그래서 B+ Tree는 데이터베이스 인덱스에 잘 맞는다.

---

## 20. 용어 빠른 정리

### 20.1 키(key)

찾기의 기준이 되는 값이다.
이번 과제에서는 주로 `id`가 키가 된다.

### 20.2 레코드(record)

테이블의 한 행(row)이라고 생각하면 된다.

### 20.3 인덱스(index)

키를 기준으로 데이터 위치를 빠르게 찾게 해 주는 보조 구조다.

### 20.4 내부 노드(internal node)

탐색 방향을 정하는 노드다.

### 20.5 리프 노드(leaf node)

실제 키와 값 또는 데이터 참조가 모여 있는 노드다.

### 20.6 split

노드가 꽉 찼을 때 둘로 나누는 작업이다.

### 20.7 범위 조회(range query)

예를 들어 `100 이상 200 미만`처럼 값의 구간을 찾는 조회다.

---

## 21. 마무리 정리

비전공자 기준으로 B+ Tree를 가장 단순하게 설명하면 아래와 같다.

> B+ Tree는 많은 키와 그에 대응하는 데이터 참조를 정렬된 상태로 관리하면서, 원하는 키를 빠르게 찾고, 범위까지 효율적으로 읽기 좋게 만든 트리 구조다.

그리고 데이터베이스 인덱스로 자주 쓰이는 이유는 아래로 정리할 수 있다.

- 검색 경로를 짧게 만들기 쉽다
- 디스크 기반 구조와 잘 맞는다
- 범위 조회와 순차 조회에 유리하다
- 데이터가 많고 계속 바뀌는 상황에서도 균형을 유지하기 좋다

이번 주 과제에서는 이 이론을 아주 거대한 DBMS 수준으로 구현하는 것이 아니라,
`id`를 기준으로 빠르게 찾을 수 있는 구조를 직접 만들고,
그 효과를 실제 조회 속도 차이로 확인하는 것이 핵심이다.

즉, 이번 문서를 다 읽고 나서 최종적으로 잡아야 할 감각은 이것이다.

> "B+ Tree는 단순한 트리 자료구조가 아니라, 데이터베이스가 대량 데이터를 더 빨리 찾기 위해 쓰는 실전형 인덱스 구조다."
