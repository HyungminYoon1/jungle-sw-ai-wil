## 2. 먼저 한 문장으로 정리하는 7주차 과제

7주차 수요 코딩회의 핵심은 다음 한 문장으로 요약할 수 있다.

> 6주차에 만든 기본 SQL 처리기에 `자동 ID 부여`, `B+ 트리 인덱스`, `WHERE id = ?` 조회 최적화, `대용량 성능 비교`를 추가하는 것이다.

이때 중요한 포인트는 단순히 `B+ 트리 자료구조만 따로 구현`하는 것이 아니라,
그 인덱스를 실제 SQL 처리 흐름 속에 연결해야 한다는 점이다.

즉, 아래가 모두 이어져야 한다.

```text
INSERT 발생
-> 레코드 저장
-> ID 자동 생성
-> 해당 ID를 인덱스에 등록
-> 이후 SELECT ... WHERE id = ? 에서 인덱스를 사용
```

## 3. 6주차 과제와 7주차 과제의 차이

### 3.1 6주차에서 일반적으로 요구된 것

6주차 수요 코딩회의 핵심은 `SQL 처리기` 구현이었다.

최소 범위는 대체로 아래와 같다.

- 텍스트 파일이나 CLI 입력으로 SQL을 받기
- SQL 문장을 파싱하기
- 최소한 `INSERT`, `SELECT` 지원하기
- 데이터를 파일에 저장하고 다시 읽기
- 각 테이블을 파일 단위로 관리하기
- 기본 테스트를 통해 동작 확인하기

즉, 6주차는 "입력된 SQL을 읽고, 해석하고, 저장소에 반영하는 흐름"을 만드는 문제였다.

### 3.2 7주차에서 새롭게 추가된 것

7주차는 그 기본 처리기에 `인덱스` 개념이 들어온다.

공지 기준으로 새로 추가된 핵심 요구사항은 아래와 같다.

- 메모리 기반 방식으로 `B+ 트리 인덱스` 구현
- 레코드 추가 시 `ID 자동 부여`
- 부여된 `ID`를 B+ 트리에 등록
- `WHERE id = ?` 조회 시 B+ 트리 사용
- 다른 필드 기준 조회는 선형 탐색과 비교
- 최소 `1,000,000`개 이상의 레코드 삽입 후 성능 테스트
- 기존 SQL 처리기와의 연동

즉, 7주차는 단순한 "SQL 처리" 과제가 아니라,
`저장 + 조회 + 인덱싱 + 성능 검증`까지 포함한 작은 DB 엔진 확장 과제에 가깝다.

### 3.3 비교표

| 항목 | 6주차 일반 과제 | 7주차 과제 |
|------|------------------|-------------|
| 핵심 목표 | SQL 처리 흐름 구현 | SQL 처리기에 인덱스 추가 |
| 최소 지원 SQL | `INSERT`, `SELECT` | 기존 기능 + `WHERE id = ?` 활용 |
| 저장 방식 | 파일 기반 저장 | 파일 저장 + 메모리 기반 인덱스 연동 |
| 조회 방식 | 전체 스캔 중심 | `id` 조건은 인덱스, 다른 필드는 선형 탐색 |
| 레코드 식별 | 직접 입력한 값 중심일 수 있음 | `ID 자동 부여` 필수 |
| 성능 요구 | 기능 위주 | 대용량 삽입 및 성능 비교 필수 |
| 핵심 자료구조 | 배열/문자열 처리, 파일 처리 | B+ 트리, 노드 분할, 인덱스 매핑 |

## 4. 7주차 과제가 정확히 요구하는 것

공지 내용을 구현 관점으로 다시 풀어쓰면, 최종 결과물은 아래를 만족해야 한다.

### 4.1 기존 SQL 처리기를 버리지 않고 이어서 써야 한다

7주차 공지에는 `이전 차수에서 구현한 SQL 처리기를 그대로 사용하여 연동`한다고 되어 있다.

이 말은 아래 뜻에 가깝다.

- 6주차의 기본 입력/파싱/실행/저장 흐름을 기반으로 확장해야 한다
- 별도의 완전히 새로운 프로그램을 만드는 것보다, 기존 처리기에 인덱스 기능을 연결해야 한다
- 기존 `INSERT`, `SELECT`가 계속 동작해야 한다

즉, 7주차는 "기존 구조를 유지하면서 기능을 증설하는 연습"이기도 하다.

### 4.2 `INSERT` 시 ID를 자동 부여해야 한다

사용자가 직접 `id`를 넣는 방식이 아니라,
레코드가 추가될 때 시스템이 자동으로 고유한 ID를 만들어야 한다.

여기서 필요한 설계 질문은 아래와 같다.

- 현재 마지막 ID가 무엇인지 어디서 관리할 것인가
- 프로그램이 다시 실행되어도 ID가 이어져야 하는가
- 이미 같은 ID가 존재하는 상황을 어떻게 막을 것인가

7주차의 핵심은 "ID가 단순 필드 하나"가 아니라,
인덱스의 키 역할을 한다는 점이다.

### 4.3 자동 부여한 ID를 B+ 트리에 등록해야 한다

레코드를 저장만 하면 끝이 아니다.
저장과 동시에 인덱스도 함께 갱신되어야 한다.

즉, `INSERT`는 아래 두 작업이 묶여야 한다.

1. 실제 데이터 저장소에 레코드 반영
2. `id -> 해당 레코드를 찾을 수 있는 정보`를 인덱스에 등록

여기서 중요한 것은 B+ 트리가 보통 `레코드 전체`를 들고 있기보다,
`레코드를 가리키는 참조 정보`를 들고 있는 구조로 생각하는 편이 자연스럽다는 점이다.

예를 들면 아래와 같은 식이다.

- `id -> 파일 오프셋`
- `id -> 행 번호`
- `id -> 다시 계산하거나 복원 가능한 레코드 위치 정보`

어떤 방식을 택하든, 핵심은 `id로 빠르게 레코드를 찾을 수 있어야 한다`는 것이다.
이번 과제처럼 6주차의 파일 기반 저장 흐름을 이어받는 경우에는,
프로그램을 다시 실행해도 설명 가능한 형태의 참조를 잡아 두는 편이 자연스럽다.
예를 들어 `파일 오프셋`, `행 번호`, `페이지/슬롯에 해당하는 위치 정보` 같은 방식이 그렇다.
반면 학습용 메모리 전용 구현에서는 같은 실행 중에만 유효한 포인터나 배열 인덱스를 사용할 수도 있다.

### 4.4 `WHERE id = ?` 조회에서 인덱스를 사용해야 한다

이 요구사항이 7주차의 핵심이다.

즉, 아래 두 조회는 같은 방식으로 처리하면 안 된다.

```sql
SELECT name FROM users WHERE id = 100;
SELECT name FROM users WHERE name = 'jungle';
```

첫 번째는 `id` 조건이므로 B+ 트리를 사용할 수 있어야 한다.
두 번째는 다른 필드 조건이므로 일반적으로 전체 레코드를 훑는 선형 탐색이 된다.

즉, 실행 시점에 최소한 아래 분기가 필요하다.

```text
WHERE 절이 있는가?
-> 있다면 where column이 id인가?
-> id라면 B+ 트리 검색
-> 아니면 선형 탐색
```

### 4.5 인덱스와 선형 탐색의 성능 차이를 보여야 한다

공지에서 단순 구현만 요구한 것이 아니다.
`최소 1,000,000개 이상의 레코드를 Insert`하고,
`ID 기준 SELECT`와 `다른 필드 기준 SELECT`의 속도를 비교해 보라고 했다.

즉, 결과물에는 아래가 포함되어야 한다.

- 대량 데이터 생성 방식
- 실제 삽입 수행
- 조회 시간 측정 방식
- 인덱스 사용 경로와 선형 탐색 경로 비교 결과

여기서 중요한 것은 "빠를 것 같다"가 아니라
`실제로 측정한 숫자`를 보여 주는 것이다.

## 5. 자주 생기는 오해

### 5.1 완전한 DBMS를 만드는 과제가 아니다

이번 과제는 학습용 구현이다.
즉, 실제 상용 DBMS 수준의 디스크 페이지 관리, 버퍼 풀, 쿼리 최적화기, 동시성 제어까지 모두 구현하라는 뜻은 아니다.

공지에도 `실제 DBMS에서 사용하는 디스크 기반 B+ Tree를 단순화한 메모리 기반 방식`이라고 되어 있다.

따라서 우선순위는 아래와 같다.

- B+ 트리의 핵심 동작을 구현한다
- 기존 SQL 처리기에 연결한다
- 성능 차이를 보여 준다

### 5.2 모든 WHERE를 최적화하라는 뜻이 아니다

공지에서 분명하게 강조한 것은 `WHERE id = ?` 조건이다.
따라서 우선순위는 `id` 단일 equality 조회 최적화에 있다.

예를 들면 아래는 꼭 필요하지 않을 수 있다.

- `WHERE age > 10`
- `WHERE id = 1 AND name = 'x'`
- `WHERE name = 'x' OR city = 'y'`

초기 구현에서는 지원 범위를 좁게 잡는 것이 더 현실적이다.

### 5.3 B+ 트리만 구현하고 SQL 처리기와 따로 놀면 부족하다

자료구조 파일 하나에서 B+ 트리 삽입/검색만 동작한다고 해서 과제가 끝난 것이 아니다.

반드시 아래 연결이 필요하다.

- SQL `INSERT`가 발생하면 인덱스 등록
- SQL `SELECT ... WHERE id = ?`가 발생하면 인덱스 조회
- SQL `SELECT ... WHERE name = ?` 같은 경우는 선형 탐색

즉, 자료구조 구현과 시스템 통합이 같이 필요하다.

## 6. 구현 전에 반드시 알아야 할 지식

이번 주 과제를 밀고 나가려면, "B+ 트리 개념만" 알아서는 부족하다.
아래 지식이 함께 필요하다.

여기서 6주차와 7주차의 학습 흐름을 같이 보면 더 이해하기 쉽다.

- 6주차에서는 `C 문법`, `포인터`, `동적 메모리`, `기본 자료구조`, `파일 기반 SQL 처리 흐름`을 익혔다
- 7주차에서는 `malloc lab`과 메모리 관점을 통해 포인터와 동적 메모리 감각을 더 깊게 다룬다
- 그리고 그 위에 `B+ 트리 노드 관리`, `split`, `자동 ID`, `인덱스와 저장소 연동`을 얹는다

즉, 6주차가 `저장하고 읽는 처리기`를 만드는 단계였다면,
7주차는 그 처리기에 `빠르게 찾는 경로`를 붙이는 단계라고 볼 수 있다.

### 6.1 C 언어 기초

최소한 아래는 익숙해야 한다.

- 구조체 선언과 사용
- 포인터와 이중 포인터
- 배열과 문자열 처리
- 함수 인자 전달 방식
- 헤더 파일과 소스 파일 분리
- `malloc`, `calloc`, `realloc`, `free`

이유는 명확하다.
B+ 트리 노드는 대부분 동적 메모리로 생성되고,
노드 내부에는 키 배열, 자식 포인터 배열, 리프 데이터 참조 등이 들어가기 때문이다.

예를 들어 아래 감각이 중요하다.

- 구조체 안에 배열을 둘 것인가
- 포인터 배열을 둘 것인가
- 자식 노드 연결을 어떻게 표현할 것인가
- 메모리 해제를 누가 책임질 것인가

### 6.2 포인터와 동적 메모리

7주차 학습 과제와 탐험 준비에서도 강조한 것처럼,
이번 주는 메모리와 포인터 감각이 특히 중요하다.

왜냐하면 B+ 트리는 연결 구조를 다루기 때문이다.

필수로 이해해야 할 질문은 아래와 같다.

- 루트 노드는 어디에 저장되는가
- 리프 노드와 내부 노드의 구조는 같은가
- 노드 split 이후 부모 포인터나 자식 포인터는 어떻게 갱신되는가
- 프로그램 종료 전 메모리 해제를 어떻게 할 것인가

단순히 "포인터 문법을 안다" 수준으로는 부족하고,
`어떤 주소가 어떤 노드를 가리키는지`를 추적할 수 있어야 한다.

### 6.3 파일 입출력과 레코드 저장 방식

6주차 SQL 처리기 과제는 파일 기반 저장이 핵심이었다.
7주차도 인덱스는 추가되지만, 데이터 저장소 자체를 완전히 버리는 것은 아니다.

따라서 아래를 알아야 한다.

- 파일 열기/닫기
- 텍스트 기반 저장 형식 다루기
- CSV 또는 자체 포맷에서 레코드 읽기/쓰기
- 대량 삽입 시 파일 append 비용
- 레코드 위치를 어떻게 다시 찾을 것인가

인덱스를 설계할 때 중요한 질문은 이것이다.

> 인덱스가 찾아낸 값으로 최종 레코드를 어떻게 복원할 것인가?

이 질문에 답하려면 저장 포맷과 레코드 참조 방식이 먼저 정해져야 한다.

### 6.4 SQL 처리 흐름

팀마다 구조는 다를 수 있지만,
일반적으로 SQL 처리기는 아래 단계들을 가진다.

```text
입력 받기
-> SQL 해석하기
-> 내부 명령 구조로 바꾸기
-> 명령 종류에 따라 실행하기
-> 저장소/인덱스를 사용하기
-> 결과 출력하기
```

7주차에서는 특히 아래가 중요하다.

- `SELECT`에 `WHERE`를 어떻게 추가할 것인가
- 실행 단계에서 `id 조건`과 `비 id 조건`을 어떻게 분기할 것인가
- 기존 코드와 새 인덱스 코드를 어디서 연결할 것인가

즉, B+ 트리 자체보다도
`이 자료구조를 SQL 처리 흐름 속 어느 지점에 끼울 것인가`가 실전적으로 더 중요하다.

### 6.5 인덱스의 의미

이 문서에서는 인덱스를 `키를 기준으로 레코드 위치를 더 빨리 찾기 위한 별도 구조` 정도로만 잡고 가면 충분하다.
자세한 개념 설명은 `bptree-database-index-basics.md`를 참고하면 된다.

이번 과제에서 중요한 것은 아래 실행 분기다.

- `WHERE id = ?` 는 인덱스를 사용
- `WHERE other_field = ?` 는 선형 탐색

즉, 인덱스가 왜 필요한지 결과로 보여 주는 과제라고 볼 수 있다.

### 6.6 B-Tree와 B+ Tree의 차이

구현 전에 최소한 아래 한 줄은 구분해야 한다.

- `B+ Tree`는 실제 데이터 또는 데이터 참조가 주로 리프에 모이고, 내부 노드는 탐색 경로용 키를 가진다

자세한 비교와 그림 설명은 `bptree-database-index-basics.md` 쪽이 더 적합하다.

### 6.7 B+ 트리의 핵심 구조

구현 관점에서 최소한 아래 이름들이 코드로 어떻게 대응되는지만 잡으면 된다.

- 차수(order)와 노드가 담을 수 있는 키/자식 수의 한계
- 내부 노드와 리프 노드의 차이
- 키 배열
- 자식 포인터 배열
- 리프 노드의 값 또는 레코드 참조
- 리프 노드 간 연결
- 루트 노드

여기서 `order`는 자료마다 정의가 조금 다를 수 있으므로,
이 문서에서는 `노드가 가질 수 있는 키/자식 수의 한계를 정하는 값` 정도로 이해해도 충분하다.
개념 그림과 자세한 구조 설명은 `bptree-database-index-basics.md`를 참고하면 된다.

### 6.8 삽입 시 split이 왜 필요한가

B+ 트리는 항상 균형을 유지해야 한다.
그래서 어떤 노드가 가득 찬 상태에서 새 키가 들어오면 `split`이 일어난다.

즉, 아래 흐름을 이해해야 한다.

1. 적절한 리프 노드를 찾는다
2. 키를 삽입한다
3. 노드가 넘치면 둘로 나눈다
4. 부모에 분할 정보를 올린다
5. 부모도 넘치면 다시 split한다
6. 루트가 넘치면 새 루트를 만든다

실제 구현에서 가장 많이 막히는 곳이 바로 이 split 로직이다.

### 6.9 자동 ID와 고유성

이번 과제에서 `id`는 단순 일반 컬럼이 아니라 인덱스 키다.
따라서 보통 아래 조건이 필요하다.

- 각 레코드는 서로 다른 `id`를 가져야 한다
- `id`는 증가하는 방식으로 부여하는 것이 단순하다
- 중복 ID는 허용하지 않는 편이 자연스럽다

즉, `id` 생성 정책은 곧 인덱스 키 정책이다.

### 6.10 레코드 참조 방식

인덱스는 키만 있다고 끝나지 않는다.
키에 대응하는 "실제 데이터 위치"가 필요하다.

가능한 방식은 아래와 같다.

- 파일 오프셋 저장
- 행 번호 저장
- 다시 계산 가능한 레코드 위치 정보 저장
- 학습용 메모리 구현이라면 배열 인덱스나 포인터 저장

어떤 방식을 택하든 아래를 만족해야 한다.

- `id`로 검색했을 때 원하는 레코드를 다시 찾을 수 있어야 한다
- 저장소 구조와 충돌하지 않아야 한다
- 대량 데이터에서도 비교적 안정적으로 동작해야 한다

특히 이 문서는 `6주차의 파일 기반 SQL 처리기`를 출발점으로 삼고 있으므로,
설계 설명에서는 실행이 끝난 뒤에도 의미를 유지할 수 있는 참조 방식이 더 일반적이다.
예를 들어 파일 오프셋, 행 번호, 페이지/슬롯 개념으로 치환 가능한 위치 정보가 그렇다.
포인터 기반 참조는 학습용 메모리 구현에서는 가능하지만,
프로그램 재시작 후에도 그대로 유지되는 값은 아니라는 점을 같이 이해해야 한다.

### 6.11 성능 측정 기초

이번 과제는 성능 비교가 필수이므로,
단순 기능 구현과 달리 "어떻게 측정할 것인가"도 알아야 한다.

최소한 아래를 정해야 한다.

- 삽입 시간 측정 범위
- 조회 시간 측정 범위
- 몇 번 반복 측정할 것인가
- 어떤 데이터 분포를 쓸 것인가
- 결과를 평균으로 볼 것인가

예를 들면 아래와 같은 함정을 피해야 한다.

- 파일 생성 시간과 조회 시간을 섞어 버리기
- 인덱스 재구성 시간과 조회 시간을 섞어 버리기
- 한 번만 측정하고 결론 내리기
- 데이터가 너무 적어 차이가 거의 안 보이는 상태에서 비교하기

### 6.12 테스트 관점

자료구조 과제에서 자주 생기는 문제는 "대충 돌아가 보이지만 특정 케이스에서 깨지는 것"이다.
따라서 기능 시연만으로는 부족하다.

아래 테스트 감각이 필요하다.

- 정상 입력 테스트
- 빈 결과 테스트
- 없는 키 조회 테스트
- 루트 split 테스트

- 대량 삽입 테스트
- 잘못된 WHERE 처리 테스트

## 7. B+ 트리를 이해할 때 꼭 잡아야 하는 핵심 동작

이 절도 개념 자체를 길게 다시 설명하기보다,
구현 체크 관점에서 꼭 빠뜨리면 안 되는 동작을 짚는 데 집중한다.

### 7.1 검색

B+ 트리 검색은 아래 흐름이다.

1. 루트에서 시작한다
2. 현재 노드의 키와 비교해 어느 자식으로 갈지 정한다
3. 리프 노드에 도달할 때까지 반복한다
4. 리프에서 목표 키를 찾는다

핵심은 "모든 레코드를 보지 않고, 비교를 통해 내려갈 방향만 결정한다"는 점이다.

### 7.2 삽입

삽입은 검색보다 복잡하다.

1. 들어갈 리프를 찾는다
2. 정렬 순서를 유지하며 삽입한다
3. 노드가 가득 차면 split한다
4. 부모 키를 조정한다
5. 필요하면 위로 계속 전파한다

특히 아래를 놓치기 쉽다.

- split 후 두 노드의 키 분배
- 부모에 올릴 기준 키 선택
- 리프 노드 연결 갱신
- 루트 변경

### 7.3 조회 경로 최적화

이번 과제에서는 B+ 트리 이론 전체보다,
아래 연결이 실제로 동작하는지가 더 중요하다.

```text
SELECT ... WHERE id = 123
-> WHERE 절 파싱
-> id 조건임을 확인
-> B+ 트리 search(123)
-> 레코드 참조 획득
-> 실제 레코드 읽기
-> SELECT 컬럼에 맞춰 출력
```

즉, 검색 결과가 단순히 "있다/없다"로 끝나면 안 되고,
최종 출력까지 이어져야 한다.

## 8. 시스템을 어떻게 나누면 좋은가

구현 방식은 팀마다 달라도 되지만,
책임 분리는 어느 정도 공통 원칙이 있다.

### 8.1 공통적으로 필요한 책임

보통 아래 역할은 어딘가에 존재해야 한다.

- SQL 입력 처리
- SQL 파싱
- 내부 명령 표현
- 실행 분기
- 실제 데이터 저장/조회
- 자동 ID 생성
- B+ 트리 인덱스 관리
- 성능 측정 및 테스트

즉, 파일명이 다르더라도 본질적으로는 위 책임들이 필요하다.

### 8.2 권장되는 일반적 분리

아래는 한 가지 가능한 예시일 뿐 정답은 아니다.

```text
입력 계층
-> 파싱 계층
-> 실행 계층
   -> 저장소 계층
   -> 인덱스 계층
```

이 구조에서 핵심은 아래다.

- 파서는 SQL 문자열을 구조화한다
- 실행 계층은 어떤 경로로 처리할지 결정한다
- 저장소는 실제 레코드를 다룬다
- 인덱스 계층은 `id` 기반 빠른 탐색을 제공한다

### 8.3 왜 인덱스 계층을 분리하는 것이 좋은가

실행 로직이 B+ 트리 내부 상세 구현까지 직접 알게 되면 코드가 쉽게 엉킨다.

예를 들어 아래처럼 역할을 나누면 읽기 쉽다.

- 실행 계층: `id 조건이면 인덱스를 써라`만 결정
- 인덱스 계층: `insert/search` 등 B+ 트리 동작 제공
- 저장소 계층: 실제 레코드 읽기/쓰기 담당

이렇게 하면 이후 구조를 바꾸기도 쉬워진다.

### 8.4 중요한 원칙: 비즈니스 판단과 자료구조 구현을 섞지 않기

아래는 가능한 한 분리하는 편이 좋다.

- SQL 문법 판단
- `WHERE id = ?` 여부 판단
- 실제 B+ 트리 노드 split 구현
- 파일 입출력 구현

이것들이 한 함수에 섞이면 디버깅이 매우 어려워진다.

## 9. 구현 관점에서 요구사항을 다시 분해하기

### 9.1 최소 SQL 확장

7주차 요구사항을 만족하려면 보통 `SELECT`에 아래 정도는 추가되어야 한다.

```sql
SELECT id, name FROM users WHERE id = 123;
SELECT name FROM users WHERE name = 'jungle';
```

초기 구현에서는 보통 아래 범위면 충분하다.

- 단일 `WHERE`
- 단일 비교 연산자 `=`
- `SELECT`에 대해서만 우선 지원
- `id` 조건과 비 `id` 조건 구분

### 9.2 INSERT 경로 확장

`INSERT`는 이제 단순 append가 아니라 아래 단계가 된다.

1. 새 ID 생성
2. 레코드에 ID 포함
3. 저장소에 쓰기
4. 인덱스 등록

즉, 삽입 시점에 저장소와 인덱스가 함께 갱신되어야 한다.

### 9.3 SELECT 경로 확장

조회는 아래 3가지로 분기될 수 있다.

1. `WHERE` 없음 -> 전체 조회 또는 기존 방식 유지
2. `WHERE id = ?` -> B+ 트리 사용
3. `WHERE other_column = ?` -> 선형 탐색

이번 과제에서 중요한 것은
2번과 3번이 실제로 다른 경로를 탄다는 점을 코드와 결과로 보여 주는 것이다.

### 9.4 인덱스 초기화 전략

프로그램이 시작할 때 인덱스를 어떻게 준비할지도 정해야 한다.

가능한 방식은 예를 들면 아래와 같다.

- 시작 시 기존 파일을 읽어 전체 인덱스 재구성
- 실행 중 삽입되는 데이터만 인덱싱

학습용 과제에서는 첫 번째 방식이 더 단순하고 설명하기 쉽다.
기존 레코드까지 포함해 조회해야 하기 때문이다.

## 10. 추천 구현 순서

과제를 빠르게 밀고 나가려면 아래 순서가 실용적이다.

### 10.1 기존 6주차 기능을 먼저 안정화하기

먼저 확인할 것:

- 기존 `INSERT`, `SELECT`가 깨지지 않는가
- 저장 포맷이 일관적인가
- 테스트를 다시 돌려 기본 기능이 정상인가

기존 기반이 흔들리면 인덱스 통합 과정에서 디버깅 비용이 크게 늘어난다.

### 10.2 `WHERE` 없는 상태에서 B+ 트리만 먼저 검증하기

처음부터 SQL과 합치면 복잡해진다.
먼저 독립적인 단위 테스트 수준에서 아래를 검증하는 편이 좋다.

- 키 삽입
- 키 검색
- 없는 키 검색
- split
- 루트 split

즉, 자료구조 자체를 먼저 믿을 수 있게 만들어야 한다.

### 10.3 자동 ID 정책 정하기

예를 들어 아래를 먼저 결정한다.

- 시작 ID는 1인가
- 마지막 ID를 어디서 계산하는가
- 기존 데이터가 있으면 최대 ID 다음 값을 쓰는가

이 정책이 정해져야 인덱스 키가 일관된다.

### 10.4 INSERT와 인덱스 연결하기

그 다음 아래를 구현한다.

```text
INSERT
-> 새 ID 생성
-> 저장소 반영
-> 인덱스 등록
```

이 단계에서는 `SELECT`보다 `INSERT` 경로를 먼저 완성하는 편이 자연스럽다.
인덱스는 삽입 데이터가 있어야 의미가 생기기 때문이다.

### 10.5 `WHERE id = ?` 조회 연결하기

이제 `SELECT` 실행 단계에서 아래 분기를 추가한다.

- `WHERE` 없음
- `WHERE id = ?`
- `WHERE other = ?`

여기서부터 7주차 과제의 핵심 모습이 드러난다.

### 10.6 성능 테스트 스크립트 또는 도구 만들기

기능이 돌아간 뒤에는 반드시 측정 도구를 만든다.

예를 들면 아래가 필요하다.

- 100만 건 데이터 생성기
- 대량 삽입 실행기
- `id` 조회 반복 측정
- 비 `id` 필드 조회 반복 측정
