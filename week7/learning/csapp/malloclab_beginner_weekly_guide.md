# 1. 이번 주 과제에서 실제로 해야 하는 일

이번 주 목표는 이미 있는 `malloc`을 사용하는 것이 아니라,
**직접 간단한 `malloc`, `free`, `realloc`을 구현해 보는 것**이다.

과제의 핵심은 `mm.c` 안의 아래 함수들을 구현하는 것이다.

```c
int mm_init(void);
void *mm_malloc(size_t size);
void mm_free(void *ptr);
void *mm_realloc(void *ptr, size_t size);
```

공지 기준으로 보면 이번 주의 기본 흐름은 아래와 같다.

1. `make`가 되는지 확인한다.
2. `./mdriver`를 실행해 본다.
3. starter code 상태에서 어떤 문제가 나는지 본다.
4. `implicit free list` 방식으로 기본 allocator를 구현한다.
5. 기본 trace가 통과되도록 만든다.
6. 여유가 있으면 점수를 더 높일 방법을 검토한다.

즉, 이번 주의 기본 목표는 **최고 성능 allocator**가 아니라,
**기본 동작이 맞는 allocator를 만드는 것**이다.

# 2. 이 과제를 왜 하는가

이 과제는 함수 4개를 채우는 문제처럼 보이지만,
실제로는 아래 감각을 익히는 것이 핵심이다.

- 힙을 블록 단위로 관리하는 방법
- 포인터를 이용해 블록을 순회하는 방법
- 정렬과 단편화가 왜 중요한지
- 빈 블록을 다시 쓰는 방법
- 인접한 빈 블록을 병합하는 방법

즉, 이번 주는 **메모리와 포인터를 코드로 체감하는 주**라고 보면 된다.

# 4. 실습 전에 알아둘 최소 개념

이 문서에서는 실습에 필요한 개념만 짧게 정리한다.

## 4.1 힙과 스택

- 스택은 함수 호출과 지역 변수처럼 자동으로 관리되는 공간이다.
- 힙은 프로그램이 실행 중에 직접 요청해서 쓰는 공간이다.
- `malloc`은 힙에서 공간을 할당하고, `free`는 그 공간을 반납한다.

## 4.2 왜 힙 관리가 어려운가

힙은 사용 패턴이 일정하지 않다.

- 요청 크기가 매번 다르다.
- 반납 순서도 제각각이다.
- 사용 중 블록과 빈 블록이 계속 섞인다.

그래서 allocator는 아래를 계속 판단해야 한다.

- 빈 블록이 어디 있는가
- 어떤 블록을 써야 하는가
- 반납된 블록을 주변 블록과 합칠 수 있는가

## 4.3 `mem_sbrk`

- 적당한 빈 블록이 없으면 힙 자체를 더 늘려야 한다.
- Malloc Lab에서는 이 과정을 보통 `mem_sbrk` 같은 함수로 단순화해서 다룬다.
- 즉, allocator는 빈 블록을 재사용하기도 하고, 필요하면 힙을 확장하기도 한다.

## 4.4 블록, 정렬, 단편화

- allocator는 힙을 **블록(block)** 단위로 관리한다.
- 블록에는 크기와 상태를 나타내는 정보가 들어간다.
- 요청 크기는 보통 정렬 단위에 맞게 반올림된다.
- 그래서 내부 단편화와 외부 단편화가 생긴다.

실습할 때 한 가지 더 기억하면 좋다.

- `malloc`이 반환하는 포인터는 보통 **사용자가 실제로 쓰는 영역(payload)의 시작 주소**다.
- 즉, allocator 내부의 header/footer 같은 정보는 사용자가 직접 만지는 영역이 아니다.

이번 주에는 아래 네 단어만 확실히 잡으면 된다.

- block
- alignment
- fragmentation
- coalescing

## 4.5 `implicit free list`

- 힙 전체를 앞에서부터 순서대로 훑으면서 free block을 찾는 방식이다.
- 별도의 복잡한 자료구조보다, 먼저 이 기본 방식이 어떻게 동작하는지 이해하는 것이 중요하다.

먼저 확인할 것:

- 어떻게 테스트하는지

## 5.2 `mm.h`

먼저 확인할 것:

- 구현해야 하는 함수가 무엇인지

# 6. 구현을 어떤 순서로 진행할 것인가

이번 주에는 아래 순서로 가는 것이 가장 현실적이다.

## 6.1 1단계: 환경 확인

- `make`가 되는지 확인한다.
- `./mdriver`가 실행되는지 확인한다.
- starter code 상태에서 어떤 에러가 나는지 본다.

## 6.2 2단계: 최소 구조 이해

아래 질문에 답할 수 있으면 출발할 준비가 된 것이다.

- 블록은 무엇인가
- free block은 어떻게 찾는가
- 왜 정렬이 필요한가
- 왜 `free` 후 병합이 필요한가

## 6.3 3단계: `mm_init`

먼저 allocator가 시작할 수 있는 최소 상태를 만든다.

이 단계에서는 보통 아래가 중요하다.

- 힙 초기화
- 순회의 시작점 준비
- 필요 시 초기 free block 준비

완료 기준 예시:

- `mm_init`이 실패 없이 호출된다.
- 이후 `mm_malloc`이 시작할 최소 상태가 준비된다.

## 6.4 4단계: `mm_malloc`

가장 먼저 맞춰야 할 핵심 함수다.

흐름은 보통 아래와 같다.

1. 요청 크기 보정
2. free block 탐색
3. 적당한 블록이 있으면 할당
4. 남는 공간이 크면 split
5. 없으면 힙 확장

완료 기준 예시:

- 작은 trace에서 `malloc` 호출이 바로 깨지지 않는다.
- `out of memory` 상태를 기본적으로 벗어난다.
- 첫 번째 free block 탐색과 할당 흐름이 동작한다.

## 6.5 5단계: `mm_free`

`free`는 단순히 "지운다"가 아니다.

중요한 것은 아래 두 가지다.

- 블록을 free 상태로 바꾸기
- 가능하면 인접 free block과 병합하기

완료 기준 예시:

- `free` 후 같은 공간이 다시 재사용될 가능성이 생긴다.
- 인접 free block 병합이 기본적으로 동작한다.

## 6.6 6단계: `mm_realloc`

처음에는 가장 단순한 방식으로 가는 것이 맞다.

```text
새 블록 malloc
-> 기존 데이터 복사
-> old block free
```

처음부터 최적화하려고 하지 말고,
먼저 맞게 동작하게 만드는 것이 중요하다.

완료 기준 예시:

- 새 블록 할당 -> 복사 -> 기존 블록 반납 흐름이 동작한다.
- 기존 데이터가 기본적으로 보존된다.

# 7. 기본 구현에서 꼭 붙잡아야 할 흐름

이번 주에는 아래 네 흐름을 이해하는 것이 핵심이다.

## 7.1 free block 탐색

- 앞에서부터 순서대로 본다.
- 들어갈 수 있는 첫 블록을 찾는다.
- 처음에는 `first fit` 기준으로 이해하면 충분하다.

## 7.2 split

- free block이 너무 크면 나눈다.
- 한쪽은 할당 블록
- 나머지는 free block으로 남긴다

## 7.3 coalesce

- `free`된 블록 옆에 free block이 있으면 합친다.
- 병합을 하지 않으면 외부 단편화가 커진다.

## 7.4 힙 확장

- 기존 free block으로 해결이 안 되면 힙을 늘린다.
- 즉, allocator는 재사용만 하는 것이 아니라 필요하면 새 공간도 확보한다.

# 8. 테스트 방법

기본적으로는 아래처럼 빌드하고 실행한다.

```bash
make
./mdriver
```

처음부터 전체 점수를 보는 것보다,
**작은 trace 하나를 먼저 통과시키는 것**이 훨씬 중요하다.

특정 trace만 테스트하고 싶으면 예를 들어:

```bash
./mdriver -f traces/binary2-bal.rep
```

verbose 출력이 필요하면 예를 들어:

```bash
./mdriver -V -f short1-bal.rep
```

실전에서는 아래 순서가 효율적이다.

1. 전체 실행이 되는지 확인
2. 작은 trace 하나로 디버깅
3. 기본 동작 안정화
4. 그다음 전체 trace 확인

# 10. 비전공자가 자주 막히는 지점

1. 정렬 때문에 요청 크기와 실제 블록 크기가 다르다는 점을 놓친다.
2. split 후 남은 블록을 free block으로 다시 처리하는 것을 잊는다.
3. `free` 후 coalesce를 하지 않아 단편화가 커진다.
4. `realloc`에서 복사 크기를 잘못 잡는다.
5. 전체 trace를 한 번에 돌리다 어디서 깨지는지 못 찾는다.

이 문제를 줄이려면 아래 습관이 중요하다.

- 작은 trace부터 검증하기
- 블록 레이아웃을 손으로 그려 보기
- 한 함수씩 맞는지 확인하기

# 11. 자주 하는 오해

## 11.1 "`free`를 하면 운영체제에 바로 반환된다"

기본 학습 단계에서는 그렇게 이해하지 않는 편이 좋다.
우선은 allocator 내부에서 다시 사용할 수 있는 블록으로 바뀐다고 보면 된다.

## 11.2 "빈 블록만 있으면 어떤 요청이든 처리할 수 있다"

그렇지 않다.
빈 블록이 흩어져 있으면 큰 요청을 처리하지 못할 수 있다.

## 11.3 "`realloc`은 그냥 `malloc`과 `free`를 합친 것이다"

가장 단순한 구현은 비슷하지만,
실제로는 기존 데이터 보존까지 고려해야 해서 더 어렵다.

## 11.4 "처음부터 점수를 높여야 한다"

아니다.
이번 주는 correctness 확보가 먼저다.
기본 동작이 맞지 않으면 점수 최적화는 의미가 없다.
