## 자주 나오는 약어 빠른 정리

처음 읽을 때 가장 많이 막히는 부분은 개념 자체보다 약어다.
아래 표를 먼저 보고 가면 문서를 훨씬 덜 끊기면서 읽을 수 있다.

| 약어 | 뜻 | 아주 짧은 설명 |
|------|----|----------------|
| `VA` | Virtual Address | 프로그램이 사용하는 가상 주소 |
| `PA` | Physical Address | 실제 RAM 상의 물리 주소 |
| `VPN` | Virtual Page Number | 가상 주소에서 "어느 페이지인가"를 나타내는 부분 |
| `VPO` | Virtual Page Offset | 그 페이지 안의 몇 번째 바이트인지를 나타내는 부분 |
| `PFN` | Physical Frame Number | 물리 메모리에서 어느 프레임인지를 나타내는 번호 |
| `PTE` | Page Table Entry | 페이지 테이블의 한 칸. 매핑과 권한 정보를 담음 |
| `TLB` | Translation Lookaside Buffer | 최근 주소 변환 결과를 저장하는 작은 캐시 |
| `MMU` | Memory Management Unit | 주소 변환을 수행하는 하드웨어 구성 요소 |
| `COW` | Copy-on-Write | 처음에는 공유하고, 쓰기 시점에만 복사하는 기법 |
| `BSS` | Block Started by Symbol | 초기값이 0인 전역/정적 변수를 위한 영역 |

이 문서를 읽다가 약어가 헷갈리면, 먼저 이 표로 돌아와 뜻을 다시 확인하면 된다.

# 1. 먼저 큰 그림: 가상 메모리는 왜 필요한가?

## 1.1 현실의 문제

컴퓨터의 실제 RAM은 한정되어 있다. 그런데 우리는 동시에 여러 프로그램을 실행한다.

예를 들어:

- 브라우저
- 메신저
- 코드 에디터
- 음악 재생 프로그램
- 운영체제 자체

이 모든 프로그램이 동시에 메모리를 필요로 한다. 이때 단순히 “각 프로그램이 RAM의 일부를 직접 사용하라”고만 하면 여러 문제가 발생한다.

### 문제 1: 메모리가 부족하다
실제 물리 메모리는 제한되어 있는데, 프로그램이 요구하는 전체 메모리 양은 이보다 더 클 수 있다.

### 문제 2: 보호가 필요하다
한 프로그램이 다른 프로그램의 메모리를 읽거나 덮어쓰면 심각한 오류가 발생한다.

### 문제 3: 주소 배치가 복잡하다
프로그램이 실행될 때마다 정확히 어느 물리 주소에 올라갈지 미리 알기 어렵다.

### 문제 4: 프로그래밍이 불편하다
프로그래머가 실제 물리 메모리의 위치까지 모두 신경 써야 한다면 개발이 매우 어려워진다.

---

## 1.2 가상 메모리의 핵심 아이디어

가상 메모리는 각 프로세스에게 다음과 같은 **착시**를 제공한다.

> “너는 아주 크고, 연속적이고, 독립적인 메모리 공간을 가진 것처럼 생각해도 된다.”

즉, 프로그램은 자신만의 주소 공간을 사용한다고 믿는다. 하지만 실제로는 운영체제와 하드웨어가 그 주소를 물리 메모리로 매핑한다.

이 착시 덕분에 다음이 가능해진다.

- 각 프로세스가 서로 분리된다
- 실제 RAM보다 더 큰 주소 공간을 제공할 수 있다
- 필요한 부분만 물리 메모리에 올려 효율적으로 사용한다
- 프로그래머는 연속적인 주소 공간을 전제로 코드를 작성할 수 있다

---

## 1.3 한 문장 정의

**가상 메모리(Virtual Memory)** 는 각 프로세스에게 독립적이고 연속적인 주소 공간을 제공하는 추상화이며, 실제로는 가상 주소를 물리 주소로 변환하여 메모리를 관리하는 시스템이다.

---

# 2. 주소 공간이란 무엇인가?

## 2.1 주소 공간의 의미

주소 공간(address space)은 프로그램이 사용할 수 있다고 보는 주소들의 집합이다.

프로그램 입장에서는 메모리가 다음처럼 생긴 것처럼 보인다.

```text
낮은 주소
+-------------------+
| 코드(text)        |
+-------------------+
| 초기화된 데이터   |
+-------------------+
| 초기화 안 된 데이터|
+-------------------+
| 힙(heap)          |  <- 높은 주소 방향으로 성장
|                   |
|                   |
|                   |
|                   |
|                   |
|                   |
|                   |
|                   |
| 스택(stack)       |  <- 낮은 주소 방향으로 성장
+-------------------+
높은 주소
```

이 구조는 프로세스마다 거의 비슷하게 보인다. 중요한 점은 이것이 **프로세스가 보는 관점**이라는 것이다. 실제 물리 메모리가 이렇게 연속적으로 배치되어 있다는 뜻은 아니다.

---

## 2.2 왜 프로세스마다 같은 모양의 주소 공간을 줄까?

이렇게 하면 프로그래머 입장에서 프로그램을 만들기 쉬워진다.

- 코드 영역은 코드
- 데이터 영역은 전역 변수
- 힙은 동적 할당 공간
- 스택은 함수 호출과 지역 변수

이 구조는 익숙하고 예측 가능하다. 운영체제는 이 추상화를 유지한 채, 실제 물리 메모리의 배치를 따로 관리한다.

---

# 3. VM as a tool: 가상 메모리가 주는 세 가지 가치

CS:APP는 가상 메모리를 단순한 저장기법이 아니라 **도구(tool)** 로 본다. 가상 메모리가 제공하는 가치는 크게 세 가지다.

## 3.1 캐싱 도구

가상 메모리는 디스크에 있는 데이터를 RAM으로 가져와 캐시처럼 사용한다.

- 디스크: 크지만 느림
- RAM: 작지만 빠름

필요한 페이지만 RAM에 올리고, 나머지는 디스크에 둔다. 따라서 가상 메모리는 일종의 **주기억장치 캐시 시스템**으로 볼 수 있다.

---

## 3.2 메모리 관리 도구

가상 메모리는 각 프로세스에게 균일하고 연속적인 주소 공간을 제공한다.

프로그래머는 다음만 생각하면 된다.

- 포인터가 가리키는 가상 주소
- 배열, 구조체, 지역 변수의 배치
- 힙과 스택의 성장

실제 물리 메모리의 조각난 배치나 다른 프로세스와의 충돌은 운영체제가 숨긴다.

---

## 3.3 메모리 보호 도구

프로세스는 자신의 가상 주소 공간에만 접근하도록 제한된다.

예를 들어:

- 프로세스 A는 프로세스 B의 메모리를 직접 읽지 못한다
- 사용자 프로그램은 커널 메모리를 임의로 수정하지 못한다
- 읽기 전용 영역은 쓰기 금지로 보호할 수 있다

이러한 보호는 페이지 단위 권한 비트와 하드웨어 검사에 의해 구현된다.

---

# 4. 가상 메모리의 기본 단위: 페이지

## 4.1 왜 페이지 단위를 사용할까?

메모리를 바이트 단위로 모두 따로 관리하면 너무 비효율적이다. 그래서 가상 메모리는 고정 크기 블록인 **페이지(page)** 단위로 메모리를 관리한다.

- 가상 주소 공간은 페이지들로 나뉜다
- 물리 메모리도 페이지 크기와 같은 **페이지 프레임(page frame)** 들로 나뉜다

예를 들어 페이지 크기가 4KB라면:

- 가상 메모리의 각 페이지는 4KB
- 물리 메모리의 각 프레임도 4KB

운영체제는 “가상 페이지 번호 → 물리 프레임 번호”의 대응을 관리한다.

---

## 4.2 페이지 단위 관리의 장점

- 큰 주소 공간을 작은 조각으로 나누어 관리할 수 있다
- 필요한 부분만 가져올 수 있다
- 보호 권한을 페이지 단위로 설정할 수 있다
- 페이지를 파일과 연결하거나 공유하기 쉽다

---

## 4.3 페이지 관점에서 본 메모리

```text
가상 주소 공간:
[VP0][VP1][VP2][VP3][VP4]...

물리 메모리:
[PF7][PF2][빈칸][PF9][PF1]...

매핑 예:
VP0 -> PF2
VP1 -> PF9
VP2 -> 디스크에 있음
VP3 -> PF1
```

여기서 중요한 점은 **가상 페이지가 물리 메모리에서 연속적일 필요가 없다는 것**이다.

# 5. 가상 주소와 물리 주소

## 5.1 두 주소를 구분해야 한다

### 가상 주소(Virtual Address)
프로그램이 사용하는 주소이다.

### 물리 주소(Physical Address)
실제 RAM 상의 위치이다.

CPU가 명령어를 실행할 때 프로그램은 가상 주소를 사용한다. 그 주소는 하드웨어에 의해 물리 주소로 바뀐다.

---

## 5.2 주소 변환이 필요한 이유

만약 프로그램이 물리 주소를 직접 사용한다면:

- 실행 위치가 바뀔 때마다 프로그램 수정이 필요할 수 있다
- 서로 다른 프로그램이 같은 물리 주소를 덮어쓸 수 있다
- 보호와 재배치가 매우 어려워진다

가상 주소를 사용하면 프로그램은 자신의 독립된 주소 공간 안에서만 동작하면 된다.

---

## 5.3 주소 변환의 큰 흐름

```text
CPU가 가상 주소 생성
        |
        v
MMU가 주소 변환
        |
        v
물리 주소 생성
        |
        v
캐시 / 메모리 접근
```

이때 MMU(Memory Management Unit)는 페이지 테이블 정보를 사용해 번역한다.

---

## 5.4 가상 주소는 어떻게 나뉘는가?

비전공자가 여기서 가장 자주 막히는 부분은 다음이다.

> "주소를 페이지 테이블로 찾는다는 말은 알겠는데,  
> 주소를 도대체 어떻게 나눠서 보는 건가?"

핵심은 간단하다.

- 가상 주소 전체를 한 덩어리로 보는 것이 아니라
- **가상 페이지 번호(VPN)** 와 **페이지 오프셋(VPO)** 으로 나눠서 본다

예를 들어 페이지 크기가 4KB라면:

- 4KB = 4096 bytes
- 4096 = 2^12
- 따라서 **하위 12비트**는 페이지 안에서의 위치(offset)를 나타낸다
- 나머지 상위 비트는 어떤 페이지인지(VPN)를 나타낸다

아주 단순하게 그리면:

```text
가상 주소
+----------------------+------------+
| 가상 페이지 번호(VPN) | 오프셋(VPO) |
+----------------------+------------+
```

이 구조의 의미는 다음과 같다.

- VPN: "어느 페이지인가?"
- VPO: "그 페이지 안의 몇 번째 바이트인가?"

주소 변환에서는 보통:

1. VPN으로 페이지 테이블을 찾고
2. 물리 프레임 번호를 얻은 뒤
3. 원래의 오프셋을 그대로 붙여 최종 물리 주소를 만든다

즉, 오프셋은 페이지 안의 위치이므로
가상 주소에서나 물리 주소에서나 그대로 유지된다고 이해하면 된다.

```text
가상 주소 = [VPN | VPO]
               |
               v
페이지 테이블 조회로 PFN 획득
               |
               v
물리 주소 = [PFN | VPO]
```

이 감각이 잡히면,
"페이지 테이블은 페이지 단위로 번역하고, 페이지 안의 세부 위치는 오프셋이 담당한다"는 말이 훨씬 쉽게 이해된다.

### 숫자로 한 번만 직접 보면 더 쉽다

예를 들어 페이지 크기가 4KB라면, 오프셋은 항상 12비트다.

어떤 가상 주소가 `0x12345`라고 가정해 보자.

- 하위 12비트: 페이지 안의 위치(VPO)
- 나머지 상위 비트: 가상 페이지 번호(VPN)

16진수로 아주 단순하게 보면:

```text
0x12345 = [ 0x12 | 0x345 ]
           VPN     VPO
```

여기서 중요한 것은 다음이다.

- `0x345`는 "그 페이지 안에서 몇 번째 바이트인가"를 뜻한다
- 페이지 테이블은 `0x12`라는 페이지 번호를 보고
- 예를 들어 이것을 물리 프레임 `0x7A`에 대응시킬 수 있다

그러면 최종 물리 주소는 다음처럼 생각할 수 있다.

```text
가상 주소  = [ 0x12 | 0x345 ]
페이지 변환 =   0x12  ->  0x7A
물리 주소  = [ 0x7A | 0x345 ]
```

즉, **페이지 번호 부분만 바뀌고, 페이지 안의 위치는 그대로 간다** 는 감각이 핵심이다.

# 6. 페이지 테이블

## 6.1 페이지 테이블이란?

페이지 테이블은 가상 페이지가 어디에 있는지를 알려주는 자료구조다.

각 페이지 테이블 엔트리(PTE)는 대략 다음 정보를 가진다.

- 이 가상 페이지가 현재 물리 메모리에 있는가? (valid/present bit)
- 있다면 어느 물리 프레임에 매핑되는가?
- 읽기/쓰기/실행 권한은 무엇인가?
- 최근 접근 여부, 수정 여부 등의 상태 비트

### 자주 보는 PTE 비트들

비전공자는 페이지 테이블을 "주소를 찾는 표" 정도로만 이해하기 쉽다.
하지만 실제로는 **상태와 권한까지 함께 담는 표**에 가깝다.

- `valid/present bit`
  - 이 페이지가 현재 RAM에 있는지 나타낸다
  - 0이면 page fault가 날 수 있다
- `read/write bit`
  - 읽기만 가능한지, 쓰기도 가능한지 나타낸다
  - 읽기 전용 페이지에 쓰면 protection fault가 날 수 있다
- `user/supervisor bit`
  - 사용자 모드에서 접근 가능한지, 커널만 접근 가능한지 구분한다
- `accessed/reference bit`
  - 최근에 접근된 적이 있는지를 표시한다
  - 운영체제가 페이지 교체 판단에 참고할 수 있다
- `dirty bit`
  - 이 페이지가 메모리에서 수정된 적이 있는지를 표시한다
  - dirty page를 내보낼 때는 다시 써야 할 수 있다

즉, PTE는 단순히 `VP -> PF` 대응만 담는 것이 아니라
**존재 여부, 권한, 교체 판단에 필요한 상태**까지 함께 담는 엔트리라고 보면 된다.

---

## 6.2 페이지 테이블의 개념 예시

```text
가상 페이지 번호   페이지 테이블 엔트리
VP0              -> PF5, valid=1, read/write
VP1              -> PF2, valid=1, read-only
VP2              -> disk, valid=0
VP3              -> PF8, valid=1
```

CPU가 VP2에 속한 주소에 접근하면, 그 페이지는 현재 물리 메모리에 없으므로 페이지 폴트가 발생한다.

---

## 6.3 페이지 테이블은 어디에 있는가?

페이지 테이블 자체는 메모리에 존재한다. 즉, 주소 변환을 위해서도 메모리에 접근해야 한다. 이 사실은 성능 문제를 낳는다.

예를 들어:

1. 페이지 테이블 엔트리를 읽어야 하고
2. 실제 데이터도 읽어야 한다

이렇게 되면 메모리 접근이 두 번 필요할 수 있다. 이 비용을 줄이기 위해 TLB가 사용된다.

---

## 6.4 왜 다단계 페이지 테이블을 쓰는가?

비전공자가 다음 질문을 자주 한다.

> "페이지 테이블이 필요하다는 것은 알겠는데,  
> 왜 다단계 구조까지 쓰는가?"

이유는 **페이지 테이블 자체도 커질 수 있기 때문**이다.

가상 주소 공간이 매우 크다면, 모든 가상 페이지에 대해 엔트리를 하나씩 둔 단일 페이지 테이블은 메모리를 너무 많이 차지할 수 있다.

그래서 운영체제는 보통 **다단계 페이지 테이블(multi-level page table)** 을 사용한다.

아이디어는 단순하다.

- 페이지 테이블을 한 번에 전부 만들지 않는다
- 실제로 필요한 구간만 잘게 나눠서 만든다

비유하면:

- 단일 페이지 테이블: 거대한 주소록 한 권을 처음부터 끝까지 다 펼쳐 놓는 방식
- 다단계 페이지 테이블: 큰 분류표를 먼저 보고, 필요한 구역의 세부 주소록만 여는 방식

즉, 다단계 구조를 쓰면

- 사용하지 않는 가상 주소 구간에 대한 테이블은 만들지 않아도 되고
- 페이지 테이블 메모리 사용량을 줄일 수 있다

이번 문서에서는 다단계 페이지 테이블의 세부 구현보다,
**"큰 주소 공간 때문에 페이지 테이블도 커지고, 이를 줄이기 위해 계층 구조를 쓴다"** 는 감각만 잡으면 충분하다.

설명을 단순화한 2단계 예시 그림으로 보면 다음과 같다.

```text
가상 주소
[ 상위 인덱스 | 하위 인덱스 | 오프셋 ]
      |
      v
상위 페이지 테이블에서 하위 테이블 위치를 찾음
      |
      v
하위 페이지 테이블에서 실제 PTE를 찾음
      |
      v
물리 프레임 번호(PFN) 획득
      |
      v
최종 물리 주소 = [ PFN | 오프셋 ]
```

즉, 다단계 페이지 테이블은
"주소를 한 번에 전부 찾는 구조"가 아니라
"큰 분류를 먼저 찾고, 그 다음 세부 항목을 찾는 구조"라고 이해하면 된다.

---

# 7. TLB: 주소 변환 캐시

## 7.1 TLB란?

TLB(Translation Lookaside Buffer)는 최근의 페이지 테이블 변환 결과를 저장하는 작은 캐시다.

즉,

- 가상 페이지 번호를 입력하면
- 대응하는 물리 프레임 번호를 빠르게 알려준다

---

## 7.2 왜 필요한가?

페이지 테이블이 메모리에 있으므로 주소 변환 자체가 느릴 수 있다. TLB는 이 변환 결과를 캐시하여 대부분의 주소 변환을 매우 빠르게 처리한다.

---

## 7.3 TLB 동작 흐름

```text
가상 주소 생성
   |
   v
TLB 조회
   |------ hit ------> 바로 물리 주소 생성
   |
   |------ miss -----> 페이지 테이블 조회 -> TLB 갱신 -> 물리 주소 생성
```

TLB miss가 난다고 해서 반드시 페이지 폴트가 나는 것은 아니다.

- **TLB miss**: 변환 정보가 TLB에 없음
- **Page fault**: 해당 페이지 자체가 물리 메모리에 없음

이 둘은 서로 다른 사건이다.

---

## 7.4 자주 하는 혼동

많은 학습자가 다음을 헷갈린다.

> “TLB miss = page fault 아닌가요?”

아니다.

- TLB miss는 단지 **변환 캐시에 없음**을 뜻한다
- page fault는 **해당 페이지가 RAM에 없음**을 뜻한다

페이지가 RAM에는 있지만 TLB에만 없는 경우가 흔하다.

---

# 8. 페이지 폴트와 요구 페이징

## 8.1 페이지 폴트란?

프로세스가 접근한 가상 페이지가 현재 물리 메모리에 없을 때 발생하는 예외가 페이지 폴트다.

이 예외는 오류일 수도 있지만, 많은 경우 **정상적인 동작 과정의 일부**다.

---

## 8.2 요구 페이징(demand paging)

모든 페이지를 처음부터 메모리에 올리는 것은 비효율적이다. 그래서 필요한 순간에만 페이지를 적재한다.

이를 요구 페이징이라 한다.

예를 들어 큰 프로그램이 있어도 실제로는 코드의 일부, 데이터의 일부만 당장 사용할 수 있다. 따라서 처음부터 전부 RAM에 올릴 필요가 없다.

---

## 8.3 페이지 폴트 처리 과정

대략적인 흐름은 다음과 같다.

1. CPU가 가상 주소에 접근한다
2. MMU가 해당 페이지가 메모리에 없음을 발견한다
3. 페이지 폴트 예외를 발생시킨다
4. 커널이 실행된다
5. 운영체제가 해당 페이지를 준비한다
   - 파일/스왑 영역에서 RAM으로 읽어올 수도 있고
   - 0으로 채워진 새 페이지를 할당할 수도 있으며
   - 접근 자체가 잘못된 경우에는 예외 처리 후 프로세스를 종료할 수도 있다
6. 페이지 테이블을 갱신한다
7. 중단된 명령어를 다시 실행한다

이 메커니즘 덕분에 프로그램은 마치 처음부터 모든 데이터가 메모리에 있었던 것처럼 계속 실행된다.

---

## 8.4 왜 페이지 폴트는 느린가?

디스크 접근은 RAM보다 훨씬 느리다. 따라서 디스크 I/O를 동반하는 페이지 폴트가 자주 발생하면 시스템은 심각하게 느려질 수 있다.

다만 모든 페이지 폴트가 반드시 디스크 접근을 일으키는 것은 아니다. 예를 들어 새 익명 페이지를 0으로 초기화해 매핑하는 경우처럼, 운영체제가 비교적 가볍게 처리하는 fault도 있다.

이 현상이 심해지면 **thrashing**이 발생할 수 있다. 즉, CPU가 실제 계산보다 페이지 교체와 적재에 대부분의 시간을 쓰는 상태다.

## 8.5 RAM이 꽉 차 있으면 어떻게 하는가?

페이지 폴트를 이해할 때 한 단계 더 나가면 이런 질문이 나온다.

> "필요한 페이지를 가져와야 하는데,  
> 물리 메모리(RAM)가 이미 꽉 차 있으면 어떻게 하나요?"

이 경우 운영체제는 **기존 물리 페이지 중 하나를 내보내고(eviction)** 새 페이지를 들여와야 한다.
즉, 페이지 폴트 처리는 단순히 "가져오기"만의 문제가 아니라,
때로는 **"무엇을 밀어낼 것인가"** 의 문제이기도 하다.

그래서 페이지 교체 정책이 중요해진다.

예:

- 최근에 안 쓴 페이지를 내보내려 하기도 하고
- 자주 쓰는 페이지는 남겨 두려 하기도 한다

학습 단계에서는 구체 정책 이름보다 아래 감각이 중요하다.

- 페이지 폴트가 났다
- 필요한 페이지를 메모리에 올려야 한다
- 공간이 없으면 다른 페이지를 내보내야 한다
- 이 과정이 자주 반복되면 성능이 크게 나빠진다

여기서 한 가지를 더 알아두면 좋다.

- 내보내는 페이지가 **수정되지 않은(clean) 페이지** 라면 버리거나, 필요할 때 다시 읽어오거나 재생성할 수 있다
- 하지만 **수정된(dirty) 페이지** 라면 디스크나 스왑 영역에 다시 써야 할 수 있다

즉, 페이지 교체 비용은
"무엇을 내보내는가"뿐 아니라
"그 페이지가 수정된 상태인가"에도 영향을 받는다.

즉, page fault는 단순한 예외가 아니라
메모리 관리 정책과 성능 문제까지 연결되는 사건이다.

---

## 8.6 page fault와 protection fault는 다르다

비전공자는 종종 모든 메모리 접근 오류를 page fault로 뭉뚱그려 이해한다.
하지만 둘은 다르다.

### page fault

- 해당 페이지가 현재 물리 메모리에 없음
- 운영체제가 페이지를 준비하면 정상적으로 계속 실행될 수도 있음

### protection fault

- 페이지는 있을 수 있지만 접근 권한이 맞지 않음
- 예: 읽기 전용 페이지에 쓰기 시도
- 이런 경우는 "없는 페이지" 문제가 아니라 **권한 위반** 문제다

즉:

- page fault: "페이지를 가져와야 한다"
- protection fault: "그 페이지에 그런 방식으로 접근하면 안 된다"

C 학습자가 실제로 자주 보는 `segmentation fault`도 이와 연결해서 이해할 수 있다.

- 잘못된 주소 접근이나 권한 위반이 발생하면
- 운영체제가 프로세스를 종료시키고
- 사용자 입장에서는 `Segmentation fault` 같은 형태로 보일 수 있다

즉, 모든 `segmentation fault`를 단순한 page fault로 이해하면 틀리고,
"복구 가능한 페이지 부재"와 "잘못된 접근"을 구분하는 감각이 필요하다.

이 구분이 잡히면,
운영체제가 왜 어떤 접근은 복구하고 어떤 접근은 프로세스를 종료시키는지 이해하기 쉬워진다.

---

## 8.7 주소 변환 전체 흐름을 한 번에 따라가 보기

지금까지 배운 내용을 하나의 접근 예시로 이어 보면 다음과 같다.

가정:

- 프로그램이 어떤 가상 주소 `VA`를 읽으려 한다
- CPU가 이 가상 주소를 만들었다
- 해당 페이지는 TLB에 없을 수도 있고, RAM에도 없을 수도 있다

흐름:

```text
1. CPU가 가상 주소 VA 생성
2. MMU가 VPN 부분을 이용해 TLB 조회
3. TLB hit면 바로 PFN을 얻음
4. TLB miss면 페이지 테이블(PTE)을 확인
5. PTE가 valid면 PFN을 얻고 TLB를 갱신
6. PTE가 invalid면 page fault 발생
7. OS가 페이지를 RAM에 올리고 PTE를 갱신
8. 다시 명령을 재시작
9. 최종 물리 주소 [PFN | VPO]로 메모리에 접근
```

이 과정을 한 문장으로 줄이면 다음과 같다.

> 프로그램은 가상 주소를 쓰고,  
> 하드웨어와 운영체제가 협력해서 그것을 실제 접근 가능한 물리 주소로 바꾼다.

비전공자가 여기서 꼭 잡아야 하는 포인트는 세 가지다.

- 프로그램은 처음부터 끝까지 **가상 주소**만 사용한다
- TLB miss와 page fault는 다른 사건이다
- page fault가 나더라도 OS가 복구하면 원래 명령은 다시 실행될 수 있다

---

# 9. 메모리 계층과 locality

## 9.1 왜 locality가 중요한가?

가상 메모리가 잘 작동하려면 프로그램이 “당장 필요한 일부 데이터”에 집중해서 접근해야 한다. 이것이 locality의 개념이다.

### 시간적 지역성(temporal locality)
최근에 사용한 데이터를 곧 다시 사용할 가능성이 높다.

### 공간적 지역성(spatial locality)
어떤 위치를 사용하면 그 주변 위치도 곧 사용할 가능성이 높다.

---

## 9.2 locality와 페이지

프로그램이 locality를 잘 보이면:

- 적은 수의 페이지로도 실행이 가능하다
- 페이지 폴트가 적다
- TLB hit율도 높아진다

반대로 locality가 나쁘면:

- 페이지 폴트가 많아진다
- 성능이 크게 떨어진다

즉, 가상 메모리 성능은 프로그램의 접근 패턴과 깊게 연결된다.

### 배열을 순서대로 읽는 예시

비전공자는 locality를 추상 개념으로만 이해하면 금방 잊어버린다.
아래처럼 생각하면 더 쉽다.

```c
for (int i = 0; i < n; i++) {
    sum += arr[i];
}
```

이 코드는 배열을 앞에서부터 순서대로 읽는다.
그러면 보통 다음 일이 일어난다.

- 한 페이지를 메모리에 올리면 그 안에 `arr[i]`, `arr[i+1]`, `arr[i+2]` 같은 이웃 원소가 함께 들어 있다
- 그래서 한 번 페이지를 가져오면 그 다음 접근들도 같은 페이지 안에서 해결될 가능성이 높다
- 최근에 본 페이지의 주소 변환 결과도 TLB에 남아 있을 가능성이 높다

즉, 순차 접근은 보통

- **공간적 지역성**이 좋고
- **TLB hit율**도 좋아지며
- **페이지 폴트**도 줄이는 방향으로 작동한다

반대로 큰 배열을 듬성듬성 건너뛰며 접근하면 locality가 나빠져 성능이 떨어지기 쉽다.

---

# 10. VM과 파일: 메모리 매핑

## 10.1 메모리 매핑이란?

운영체제는 파일의 내용을 가상 메모리의 일부 영역에 연결할 수 있다. 이를 **메모리 매핑(memory mapping)** 이라고 한다.

즉, 프로그램은 파일을 `read`로 복사해 오는 대신, 파일이 메모리에 있는 것처럼 주소로 접근할 수 있다.

---

## 10.2 왜 유용한가?

- 파일 입출력과 메모리 접근을 통합된 방식으로 다룰 수 있다
- 코드 영역(text segment)도 파일에서 매핑된다
- 여러 프로세스가 같은 파일 페이지를 공유할 수 있다
- `mmap` 기반으로 효율적인 파일 처리 가능

---

## 10.3 파일 기반 매핑과 익명 매핑

메모리 매핑을 볼 때는 먼저 두 종류를 구분하면 좋다.

### 파일 기반 매핑(file-backed mapping)

- 어떤 파일의 내용을 메모리에 연결한다
- 코드 영역, 공유 라이브러리, 메모리 매핑 파일 입출력 등이 여기에 해당한다
- 필요 시 파일 내용을 페이지 단위로 읽어 온다

### 익명 매핑(anonymous mapping)

- 특정 파일과 직접 연결되지 않은 메모리 영역이다
- 큰 힙 확장, 일부 동적 메모리 할당, 스택 성격의 메모리 확보 등에 쓰일 수 있다
- 처음에는 0으로 채워진 새 페이지처럼 보일 수 있다

즉, 메모리 매핑은 항상 "파일을 메모리로 보는 기능"만이 아니라,
운영체제가 **새 메모리 영역을 준비하는 방식**이기도 하다.

---

## 10.4 shared mapping과 private mapping

또 하나 자주 나오는 구분은 다음이다.

### shared mapping

- 같은 매핑을 여러 프로세스가 함께 볼 수 있다
- 한쪽의 수정이 다른 쪽에도 보일 수 있다
- 공유 메모리나 일부 파일 공유에 유용하다

### private mapping

- 처음에는 같은 내용을 볼 수 있어도
- 수정이 일어나면 copy-on-write 방식으로 분리될 수 있다
- 수정 내용이 원본 파일에 반영되지 않고, 보통 copy-on-write로 분리된 사본에 적용된다고 이해하면 된다

이 구분은 나중에 `fork`, 공유 라이브러리, COW를 이해할 때 도움이 된다.

---

## 10.5 공유 객체(shared object)

라이브러리나 공유 파일은 여러 프로세스에 동시에 매핑될 수 있다. 이때 물리 메모리의 같은 페이지를 여러 가상 주소 공간이 함께 참조할 수 있다.

이 방식은 메모리를 절약하고, 라이브러리 로딩을 효율적으로 만든다.

---

# 11. 프로세스의 가상 주소 공간 구조

이제 프로세스 주소 공간을 조금 더 체계적으로 보자.

```text
낮은 주소
+-----------------------------+
| 코드(text)                  |
+-----------------------------+
| 초기화된 전역/정적 데이터   |
+-----------------------------+
| BSS(0으로 초기화되는 영역)  |
+-----------------------------+
| 힙(heap)                    |
|   malloc이 사용하는 영역    |
|   높은 주소 방향으로 성장   |
+-----------------------------+
|        매핑된 영역들        |
|   공유 라이브러리, mmap 등  |
+-----------------------------+
| 스택(stack)                 |
|   함수 호출 프레임          |
|   낮은 주소 방향으로 성장   |
+-----------------------------+
높은 주소
```

---

## 11.0 코드로 한 번 연결해 보기

문장만 읽으면 `데이터`, `BSS`, `힙`, `스택`이 머릿속에서 잘 분리되지 않을 수 있다.
아래처럼 아주 짧은 C 코드 예시로 보면 더 직관적이다.

```c
#include <stdlib.h>

int global_init = 10;       // 데이터 영역
int global_zero;            // BSS
static int static_init = 3; // 데이터 영역

int main(void) {
    int local = 7;                  // 스택
    int *heap_ptr = malloc(4 * sizeof(int)); // 힙

    heap_ptr[0] = global_init + static_init + local;

    free(heap_ptr);
    return 0;
}
```

이 예시에서 보면:

- `global_init`, `static_init`은 초기값이 있으므로 데이터 영역에 놓인다
- `global_zero`는 0으로 시작하는 전역 변수이므로 BSS에 놓인다
- `local`은 함수 안의 지역 변수이므로 스택에 놓인다
- `malloc`으로 받은 `heap_ptr`이 가리키는 메모리는 힙에 놓인다
- 단, `heap_ptr` 변수 자체는 지역 변수이므로 스택에 있다

비전공자가 가장 자주 헷갈리는 지점은
"포인터 변수"와
"포인터가 가리키는 메모리"
가 서로 다른 영역에 있을 수 있다는 점이다.

---

## 11.1 코드 영역

프로그램 명령어가 저장된다. 보통 읽기 전용이고 실행 가능하다.

## 11.2 데이터 영역

초기화된 전역 변수, 정적 변수가 들어간다.

## 11.3 BSS

초기값이 0인 전역/정적 변수를 위한 영역이다.

## 11.4 힙

실행 중 동적으로 크기가 변하는 메모리 영역이다. `malloc`, `new` 등이 이 영역을 사용한다.

## 11.5 스택

함수 호출마다 생성되는 스택 프레임이 쌓인다. 지역 변수, 반환 주소, 저장 레지스터 등이 놓인다.

---

# 12. 커널 가상 메모리 개요

사용자 프로세스뿐 아니라 커널도 가상 메모리를 사용한다. 커널은 다음과 같은 이유로 별도 관리가 필요하다.

- 커널 코드와 데이터 보호
- 사용자 공간과 커널 공간 분리
- I/O 버퍼와 자료구조 관리
- 페이지 테이블 자체 관리

사용자 모드에서 커널 메모리에 임의로 접근하지 못하도록 권한 검사가 수행된다.

---

# 13. 복사-시-쓰기(Copy-on-Write) 아이디어

운영체제는 `fork` 같은 동작을 효율적으로 만들기 위해 **copy-on-write** 기법을 사용한다.

처음에는 부모와 자식이 같은 물리 페이지를 공유한다. 이 페이지들은 읽기 전용처럼 표시된다. 둘 중 하나가 실제로 쓰기를 시도하면 그 순간 새로운 페이지를 복사하여 별도로 할당한다.

### 순서대로 보면 더 쉽다

```text
1. 부모 프로세스가 어떤 페이지를 사용 중이다
2. fork가 일어나 자식 프로세스가 만들어진다
3. 처음에는 부모와 자식이 같은 물리 페이지를 함께 본다
4. 이 페이지는 함부로 쓰지 못하도록 읽기 전용처럼 관리된다
5. 부모나 자식 중 한쪽이 쓰기를 시도한다
6. 운영체제가 그 시점에 새 페이지를 하나 복사해서 만들어 준다
7. 쓰기를 시도한 쪽은 새 페이지를 사용하고, 다른 쪽은 기존 페이지를 계속 본다
```

즉, COW의 핵심은
"처음부터 복사하지 말고, 진짜 수정이 필요할 때만 복사하자"이다.

장점은 명확하다.

- 처음부터 전체 주소 공간을 복사하지 않아도 된다
- 실제 수정이 일어날 때만 비용을 지불한다

이 역시 가상 메모리와 페이지 보호 비트 덕분에 가능한 기법이다.

---

# 14. 이제 핵심 심화: 9.9 동적 메모리 할당

여기부터는 9장의 중요한 응용인 **동적 메모리 할당(dynamic memory allocation)** 을 깊게 살펴본다.

많은 학습자가 `malloc`을 “필요한 만큼 메모리를 주는 함수” 정도로만 이해한다. 하지만 시스템 관점에서는 훨씬 더 중요하다.

동적 메모리 할당은 다음 질문과 연결된다.

- 프로그램 실행 중에 필요한 메모리 크기가 바뀌면 어떻게 할까?
- 힙은 누가 관리할까?
- `malloc`은 어떻게 빈 공간을 찾을까?
- `free` 이후 공간은 어떻게 재사용할까?
- 왜 메모리 단편화가 생길까?
- 좋은 할당기의 기준은 무엇일까?

---

## 14.1 왜 동적 할당이 필요한가?

정적 메모리만으로는 실행 중 크기가 달라지는 데이터를 다루기 어렵다.

예:

- 실행 중 입력 크기를 보고 배열 크기를 결정
- 연결 리스트, 트리, 해시 테이블 생성
- 객체 수가 예측되지 않는 프로그램
- 네트워크 서버의 요청별 버퍼 관리

이런 경우 필요한 시점에 필요한 크기만큼 메모리를 받아야 한다.

---

## 14.2 기본 인터페이스

C에서 대표적인 동적 메모리 할당 인터페이스는 다음과 같다.

```c
void *malloc(size_t size);
void free(void *ptr);
void *realloc(void *ptr, size_t size);
void *calloc(size_t nmemb, size_t size);
```

### `malloc(size)`
- 최소 `size` 바이트 이상을 담을 수 있는 블록을 반환한다
- 정렬(alignment) 조건을 만족해야 한다
- 실패하면 `NULL` 반환

### `free(ptr)`
- 이전에 할당받은 블록을 반환한다
- 반환된 블록은 다시 가용 공간이 된다
- `free(NULL)`은 아무 일도 하지 않는다

### `realloc(ptr, size)`
- 기존 블록의 크기를 조정한다
- 필요하면 새 블록을 할당하고 내용을 복사한 뒤 기존 블록을 반환한다
- `ptr == NULL`이면 보통 `malloc(size)`처럼 생각해도 된다

### `calloc(nmemb, size)`
- `nmemb * size` 바이트를 할당하고 0으로 초기화한다

---

## 14.3 힙과 할당기

동적 메모리 할당기는 보통 **힙(heap)** 영역을 관리한다.

프로그래머는 단순히 `malloc`/`free`를 호출하지만, 내부적으로 할당기는 다음을 수행한다.

- 힙 안에서 빈 블록을 찾는다
- 요청한 크기에 맞게 블록을 분할한다
- 반환된 블록을 다시 가용 상태로 만든다
- 인접한 빈 블록을 병합한다
- 필요하면 운영체제로부터 힙을 더 확장한다

즉, 할당기는 힙의 관리자다.

여기서 중요한 점은, `malloc`이 직접 물리 메모리 주소를 반환하는 것이 아니라는 사실이다. `malloc`이 돌려주는 것은 프로세스의 가상 주소 공간 안에 있는 힙 블록의 시작 주소이며, 실제 물리 페이지의 준비와 매핑은 운영체제와 가상 메모리 시스템이 담당한다.

---

# 15. 동적 메모리 할당기의 목표

좋은 할당기는 단순히 “동작하는 것”만으로는 부족하다. 일반적으로 다음 목표들을 균형 있게 만족해야 한다.

## 15.1 처리량(throughput)

`malloc`과 `free`를 빠르게 처리해야 한다.

## 15.2 메모리 이용률(utilization)

낭비되는 공간이 적어야 한다. 즉, 프로그램이 실질적으로 사용하는 데이터 대비 힙의 총 크기가 너무 커지지 않아야 한다.

## 15.3 단편화 최소화

작은 빈 조각이 많이 생기면 큰 블록을 할당하기 어려워진다. 이를 줄여야 한다.

## 15.4 구현 복잡도와 유지보수성

너무 복잡한 자료구조는 빠를 수는 있어도 구현 실수와 오버헤드를 낳을 수 있다.

이 목표들은 서로 충돌하기도 한다. 예를 들어 매우 빠른 할당을 위해 단순 정책을 쓰면 단편화가 커질 수 있다.

---

# 16. 블록(block) 관점으로 힙 이해하기

힙은 일반적으로 여러 개의 **블록(block)** 으로 나뉘어 관리된다.

각 블록은 다음 둘 중 하나다.

- 할당된 블록(allocated block)
- 가용 블록(free block)

간단히 그리면 다음과 같다.

```text
힙:
+--------+--------+--------+--------+--------+
| alloc  | free   | alloc  | free   | alloc  |
+--------+--------+--------+--------+--------+
```

할당기는 요청이 들어오면 가용 블록 중 하나를 골라 사용한다.

---

## 16.1 왜 블록마다 메타데이터가 필요한가?

할당기는 단순히 사용자 데이터만 저장해서는 안 된다. 블록의 크기, 상태 같은 관리 정보가 있어야 한다. 그래서 각 블록에는 보통 **헤더(header)** 가 붙는다.

예:

```text
+------------------+----------------------+
| 헤더             | payload(사용자 데이터)|
+------------------+----------------------+
```

헤더에는 보통 다음이 들어간다.

- 블록 크기
- 할당 여부
- 경우에 따라 이전 블록의 상태

이 정보가 있어야 다음 블록으로 이동하거나, 블록을 분할하거나, 병합할 수 있다.

---

## 16.2 최소 블록 크기와 센티넬 블록

실제 할당기 구현에서는 "아무 크기나 블록으로 쓰는 것"이 보통 불가능하다.

예를 들어:

- 헤더를 저장해야 하고
- 경우에 따라 footer도 필요하며
- explicit free list라면 포인터 두 개를 저장할 공간도 필요하다

그래서 할당기에는 보통 **최소 블록 크기(minimum block size)** 가 있다.
너무 작은 나머지 조각은 별도 블록으로 분할하지 않고, 현재 블록에 포함시키는 이유도 여기에 있다.

또 많은 교재와 구현에서는 **prologue block** 과 **epilogue block** 같은 센티넬(sentinel) 블록을 둔다.

- prologue block: 힙 시작 부분의 경계 처리를 단순하게 만들기 위한 가짜 할당 블록
- epilogue block: 힙 끝을 표시하기 위한 특별한 끝 블록

이런 장치는
"첫 블록인가?"
"마지막 블록인가?"
같은 예외 처리를 줄여 준다.

---

## 16.3 구현에서 자주 보는 불변식(invariant)

할당기 코드는 포인터 계산이 많아서, 코드 한 줄 한 줄보다
"항상 참이어야 하는 조건"을 잡고 보는 편이 안전하다.

대표적인 불변식은 다음과 같다.

- 모든 블록 시작 주소가 정렬 조건을 만족한다
- 모든 블록 크기는 최소 블록 크기 이상이다
- 헤더와 footer를 둘 다 쓰는 설계라면 두 값이 서로 일치한다
- immediate coalescing 설계라면 인접한 free 블록이 그대로 남아 있지 않는다
- free list를 따로 쓰는 설계라면 리스트에 있는 블록은 정말 free 상태여야 한다
- prologue / epilogue 같은 센티넬 블록이 깨지지 않아야 한다

그래서 실제 구현에서는 **heap checker** 같은 검증 함수를 두고
이 불변식들이 유지되는지 점검하기도 한다.

이 관점이 있으면,
버그가 났을 때 "어느 함수가 틀렸나?"보다
"어떤 불변식이 깨졌나?"를 먼저 확인할 수 있어 디버깅이 쉬워진다.

---

# 17. 정렬(alignment)

## 17.1 왜 정렬이 필요한가?

많은 시스템에서 데이터는 특정 바이트 경계에 맞춰 저장될 때 접근이 효율적이거나 필수적이다. 그래서 `malloc`이 반환하는 주소는 정렬 조건을 만족해야 한다.

예를 들어 8바이트 정렬이면, 반환 주소는 8의 배수 주소여야 한다.

---

## 17.2 정렬이 설계에 미치는 영향

사용자가 13바이트를 요청해도 실제 블록 크기는 다음처럼 커질 수 있다.

- 헤더 포함
- 정렬 패딩 포함

따라서 실제 할당 크기는 16바이트, 24바이트 등으로 반올림될 수 있다.

이것이 내부 단편화의 원인 중 하나다.

---

# 18. 단편화(fragmentation)

동적 메모리 할당을 이해할 때 반드시 잡아야 하는 개념이 단편화다.

## 18.1 내부 단편화(internal fragmentation)

할당된 블록 내부에서 사용하지 못하고 낭비되는 공간이다.

예:

- 사용자는 13바이트 요청
- 할당기는 24바이트 블록 제공
- 실제 payload로 13바이트만 사용

그러면 나머지 일부 공간은 블록 내부에서 낭비된다.

원인:

- 정렬 요구
- 최소 블록 크기
- 분할 후 너무 작은 나머지를 따로 쓰지 못함

---

## 18.2 외부 단편화(external fragmentation)

전체 가용 공간의 합은 충분하지만, 하나의 큰 연속 블록이 없어 요청을 처리하지 못하는 상태다.

예:

```text
[free 20][alloc][free 18][alloc][free 22]
```

총 60바이트가 비어 있어도 50바이트 연속 블록이 없으면 큰 요청을 만족시키지 못할 수 있다.

---

## 18.3 왜 중요한가?

동적 할당기의 성능 문제는 단순 속도만이 아니다. 시간이 지날수록 힙이 잘게 조각나면 메모리 이용률이 떨어지고, 더 자주 힙을 확장해야 한다.

---

# 19. 암시적 가용 리스트(implicit free list)

가장 기본적인 할당기 설계는 **암시적 가용 리스트** 이다.

## 19.1 아이디어

모든 블록이 힙에 순서대로 놓여 있다고 가정하고, 각 블록의 헤더를 읽으며 순차적으로 탐색한다.

즉, 별도의 free 리스트 포인터가 없어도 “블록 크기”를 이용해 다음 블록 위치를 계산할 수 있다.

```text
현재 블록 주소 + 현재 블록 크기 = 다음 블록 주소
```

---

## 19.2 블록 구조 예시

```text
+-----------+-------------------+
| 헤더      | payload           |
+-----------+-------------------+
```

헤더에는 다음 정보가 packed 형태로 들어갈 수 있다.

- 블록 크기 (정렬 때문에 하위 비트는 0)
- 할당 비트(alloc bit)

예를 들어 하위 1비트를 할당 비트로 사용하면,
나머지 상위 비트는 크기를 나타내는 데 사용 가능하다.

---

## 19.3 장점과 한계

### 장점
- 구현이 단순하다
- 개념 이해에 좋다

### 한계
- 가용 블록을 찾기 위해 전체 힙을 선형 탐색할 수 있다
- 큰 힙에서 느리다
- 단편화 관리가 어렵다

즉, 교육용 출발점으로는 좋지만 고성능 실무 할당기로는 부족하다.

---

# 20. 블록 배치 정책(place policy)

가용 블록 중 어떤 블록을 선택할지 결정하는 정책은 할당기의 핵심 설계 요소다.

## 20.1 First fit

앞에서부터 탐색하여 **처음으로 맞는 블록**을 선택한다.

### 특징
- 빠르고 단순
- 앞부분에 작은 조각들이 많이 쌓일 수 있다

---

## 20.2 Next fit

직전 탐색이 끝난 위치부터 다시 시작한다.

### 특징
- first fit의 탐색 비용을 줄이려는 시도
- 항상 더 좋은 결과를 보장하지는 않는다

---

## 20.3 Best fit

들어갈 수 있는 가용 블록 중 **가장 작은 적합 블록**을 선택한다.

### 특징
- 남는 공간을 최소화하려고 한다
- 탐색 비용이 크다
- 실제 단편화가 항상 줄어드는 것은 아니다

---

## 20.4 정책 비교의 핵심

좋은 정책은 상황에 따라 다르다. 요청 패턴, 블록 크기 분포, 힙의 상태에 따라 결과가 달라진다.

학습 포인트는 다음이다.

> 동적 할당에는 “항상 최선의 단 하나의 정책”이 없다.  
> 속도와 공간 효율 사이의 trade-off를 이해해야 한다.

---

# 21. 블록 분할(splitting)

큰 가용 블록이 있는데 요청 크기가 더 작다면, 전체를 다 주는 것은 낭비다. 그래서 할당기는 블록을 분할한다.

예:

```text
원래 free 블록: 64바이트
요청 크기(정렬/헤더 포함): 24바이트

분할 후:
[alloc 24][free 40]
```

단, 남는 조각이 너무 작으면 별도 free 블록으로 관리할 수 없으므로 분할하지 않고 통째로 할당할 수 있다.

이 판단 역시 내부 단편화와 외부 단편화 사이의 균형 문제다.

---

# 22. 블록 병합(coalescing)

`free`가 호출되면 블록이 가용 상태가 된다. 그런데 인접한 블록도 가용 상태라면 하나로 합치는 것이 유리하다. 이를 **병합(coalescing)** 이라 한다.

## 22.1 왜 필요한가?

병합이 없으면 작은 free 조각들이 많이 남아 외부 단편화가 커진다.

예:

```text
[free 16][free 24][alloc][free 32]
```

앞의 두 블록을 합쳐 `[free 40]` 으로 만들면 큰 요청을 처리하기 쉬워진다.

---

## 22.2 즉시 병합 vs 지연 병합

### 즉시 병합(immediate coalescing)
`free`가 호출될 때마다 바로 인접 free 블록과 병합한다.

장점:
- 외부 단편화 감소
- 큰 free 블록 확보 용이

단점:
- 자주 분할/병합이 반복되어 오버헤드가 생길 수 있다

### 지연 병합(deferred coalescing)
필요한 시점까지 병합을 미룬다.

장점:
- 어떤 workload에서는 오버헤드 감소

단점:
- 당장 단편화가 커질 수 있다

---

# 23. 경계 태그(boundary tag)

병합을 효율적으로 하려면 **이전 블록이 free인지** 빠르게 알아야 한다. 그런데 암시적 리스트에서는 이전 블록의 시작 위치를 찾기 어렵다.

이를 해결하기 위해 많은 할당기는 **푸터(footer)** 를 둔다.

## 23.1 구조

```text
+---------+-------------------+---------+
| header  | payload           | footer  |
+---------+-------------------+---------+
```

header와 footer 모두 블록 크기와 할당 여부를 저장한다.

---

## 23.2 왜 유용한가?

현재 블록의 시작만 알고 있어도,
이전 블록의 footer를 읽어 이전 블록 크기를 알 수 있다. 그러면 이전 블록 시작 주소를 계산할 수 있다.

즉, 양옆 인접 블록의 상태를 빠르게 알아내어 병합할 수 있다.

---

## 23.3 비용

footer는 추가 공간을 사용한다. 따라서 공간 효율이 떨어질 수 있다. 그래서 실제 고급 할당기에서는 모든 블록에 footer를 두지 않고 최적화하기도 한다.

---

# 24. 4가지 병합 경우

현재 블록을 `free`할 때, 인접 블록 상태는 네 가지 경우가 있다.

## 경우 1: 이전 alloc, 다음 alloc
그냥 현재 블록만 free로 표시

## 경우 2: 이전 alloc, 다음 free
현재 + 다음 병합

## 경우 3: 이전 free, 다음 alloc
이전 + 현재 병합

## 경우 4: 이전 free, 다음 free
이전 + 현재 + 다음 모두 병합

이 4가지는 동적 할당 구현에서 매우 중요한 기본 패턴이다.

---

# 25. 암시적 리스트 기반 할당기의 전체 흐름

## `malloc(size)`
1. 요청 크기를 정렬과 헤더 포함 형태로 조정한다
2. 가용 블록을 탐색한다
3. 적절한 블록이 있으면 배치한다
4. 필요하면 분할한다
5. 없으면 힙을 확장한다

## `free(ptr)`
1. 해당 블록을 free로 표시한다
2. 인접 블록들과 병합한다

이 흐름을 정확히 이해하면 `malloclab`의 기본 골격을 읽기 쉬워진다.

---

# 26. 명시적 가용 리스트(explicit free list)

암시적 리스트의 가장 큰 문제는 탐색 비용이다. 이를 줄이기 위해 **가용 블록들만 따로 연결 리스트로 관리**할 수 있다.

## 26.1 아이디어

free 블록의 payload 일부를 포인터 저장 공간으로 활용한다.

예:

```text
free block:
+---------+----------+----------+---------+
| header  | pred ptr | succ ptr | footer  |
+---------+----------+----------+---------+
```

이렇게 하면 free 블록들만 연결된 리스트를 유지할 수 있다.

---

## 26.2 장점

- 전체 힙이 아니라 free 블록만 탐색하면 된다
- 탐색 속도가 개선된다

## 26.3 비용과 주의점

- free 블록이 최소한 두 개의 포인터를 담을 만큼 커야 한다
- 삽입/삭제 로직이 복잡해진다
- 리스트 순서와 메모리 상의 물리 순서가 다를 수 있다

---

# 27. 명시적 리스트의 삽입 정책

free 블록을 리스트 어디에 넣을지도 결정해야 한다.

## 27.1 LIFO 삽입

방금 free된 블록을 리스트 맨 앞에 넣는다.

장점:
- 삽입이 O(1)
- 구현이 단순

단점:
- 어떤 workload에서는 단편화가 커질 수 있다

---

## 27.2 주소 순 삽입

메모리 주소 순으로 free 리스트를 유지한다.

장점:
- 병합과 locality 측면에서 유리할 수 있다

단점:
- 삽입이 더 복잡하고 느릴 수 있다

---

# 28. 분리 가용 리스트(segregated free lists)

더 나은 성능을 위해 free 블록을 크기별로 여러 리스트로 나눌 수 있다.

예:

- 1~16 바이트용 리스트
- 17~32 바이트용 리스트
- 33~64 바이트용 리스트
- 65~128 바이트용 리스트
- ...

이렇게 하면 요청 크기에 맞는 블록을 더 빨리 찾을 수 있다.

---

## 28.1 왜 효과적인가?

- 큰 블록 리스트에서 작은 요청을 탐색하느라 낭비하지 않는다
- best-fit에 가까운 선택을 더 적은 비용으로 구현할 수 있다
- 실제 많은 현대 할당기가 크기 클래스(size class)를 활용한다

---

## 28.2 trade-off

- 여러 리스트 관리가 복잡하다
- 크기 클래스 설계가 중요하다
- 너무 잘게 나누면 관리 비용이 커질 수 있다

---

# 29. 힙 확장과 시스템 호출

할당기가 힙 내부에서 더 이상 적절한 블록을 찾지 못하면 운영체제로부터 더 많은 메모리를 받아와야 한다.

전통적으로는 `sbrk` 같은 방식으로 프로그램의 데이터 세그먼트를 확장했다. 현대 시스템에서는 큰 할당에 대해 `mmap`을 사용하는 경우도 많다.

CS:APP의 교육용 동적 메모리 할당기 문맥에서는 `mem_sbrk` 같은 인터페이스를 통해 힙 확장을 단순화해서 다룬다. 즉, 책에서는 운영체제의 실제 복잡한 세부 구현을 모두 드러내기보다, 할당기 설계의 핵심 원리를 이해할 수 있도록 모델을 정리해서 제시한다.

학습 포인트는 다음이다.

> `malloc`은 무에서 메모리를 만드는 것이 아니다.  
> 필요할 때 운영체제에게 더 큰 힙 영역을 요청하고, 그 내부를 다시 잘게 관리하는 것이다.

---

# 30. `realloc`은 왜 어려운가?

`realloc`은 기존 블록 크기를 바꾸는 함수인데, 생각보다 복잡하다.

## 30.1 가능한 전략

### 경우 1: 현재 블록이 이미 충분히 크다
그대로 유지하고 필요한 경우 내부적으로 분할할 수 있다.

### 경우 2: 바로 옆 free 블록과 합치면 충분하다
인접 블록과 병합하여 제자리 확장 가능

### 경우 3: 제자리 확장이 불가능하다
새 블록 할당 → 기존 내용 복사 → old 블록 free

---

## 30.2 왜 비싼가?

가장 나쁜 경우 데이터 복사가 필요하다. 큰 객체를 자주 `realloc`하면 성능 비용이 커질 수 있다.

그래서 동적 배열 구현에서는 보통 크기를 조금씩 늘리지 않고 **2배씩 확장**하는 전략을 사용한다. 이렇게 하면 총 복사 비용을 amortized 관점에서 줄일 수 있다.

---

# 31. 안전성 관점에서의 동적 메모리

동적 메모리 할당은 편리하지만, 시스템 버그의 주요 원인이다.

## 31.1 흔한 오류들

### 메모리 누수(memory leak)
할당했지만 `free`하지 않음

### 이중 해제(double free)
같은 블록을 두 번 `free`

### 해제 후 사용(use-after-free)
`free`된 포인터를 계속 사용

### 버퍼 오버플로(buffer overflow)
할당된 크기를 넘어 쓰기

### 잘못된 포인터 해제
`malloc`으로 받은 적 없는 주소를 `free`

---

## 31.2 왜 위험한가?

힙은 동적으로 변하고 블록이 재사용되므로, 실수 하나가 다른 객체의 데이터를 망가뜨릴 수 있다. 이런 오류는 재현이 어렵고 보안 취약점으로 이어질 수 있다.

즉, 동적 메모리 할당은 단순히 자료구조 구현 문제가 아니라 **시스템 안정성과 보안의 핵심 문제**다.

---

# 32. 동적 할당기를 볼 때 꼭 잡아야 하는 관점

동적 메모리 할당 문제를 볼 때 다음 세 가지 관점을 동시에 잡아야 한다.

## 32.1 자료구조 관점

- 블록은 어떻게 표현되는가?
- free 블록은 어떻게 탐색되는가?
- 병합과 분할은 어떻게 수행되는가?

## 32.2 시스템 관점

- 힙은 가상 주소 공간의 일부다
- 필요시 운영체제로부터 확장된다
- 페이지 단위 VM 위에서 동작한다

## 32.3 성능 관점

- 탐색 비용은 어떤가?
- 내부/외부 단편화는 어떤가?
- locality는 어떤가?
- 처리량과 이용률의 균형은 어떤가?

이 셋을 함께 보는 것이 CS:APP식 사고다.

# 33. 간단한 예제로 보는 힙의 변화

가정:
- 초기 힙에 큰 free 블록 하나가 있다
- 정렬과 헤더/푸터를 단순화하여 표기한다

초기 상태:

```text
[ free 128 ]
```

### 1) `p = malloc(24)`

```text
[ alloc 32 ][ free 96 ]
```

### 2) `q = malloc(16)`

```text
[ alloc 32 ][ alloc 24 ][ free 72 ]
```

### 3) `free(p)`

```text
[ free 32 ][ alloc 24 ][ free 72 ]
```

### 4) `r = malloc(20)`

first fit이면 앞의 free 32가 다시 사용될 수 있다.

```text
[ alloc 32 ][ alloc 24 ][ free 72 ]
```

### 5) `free(q)`

```text
[ alloc 32 ][ free 24 ][ free 72 ]
```

### 6) 병합

```text
[ alloc 32 ][ free 96 ]
```

이 예제를 통해 볼 수 있는 것:

- 할당과 해제가 반복되면 힙의 모양이 계속 변한다
- 병합이 없으면 조각이 쌓인다
- 정책에 따라 결과가 달라진다

# 34. 동적 할당과 페이지 수준 VM의 연결

많은 학습자는 9.9를 별개의 내용처럼 느낀다. 하지만 실제로는 9장의 앞부분과 강하게 연결된다.

## 34.1 힙도 가상 메모리 위에 있다

`malloc`이 반환하는 주소는 물리 주소가 아니라 가상 주소다.

## 34.2 힙 확장은 결국 VM 영역 확장이다

할당기가 더 많은 공간을 요구하면 운영체제는 프로세스의 가상 주소 공간에서 힙 영역을 늘린다.

## 34.3 큰 힙 접근은 페이지 locality의 영향을 받는다

힙 객체를 흩어지게 배치하거나, 비효율적인 접근 패턴을 사용하면 페이지 폴트나 캐시/TLB 성능에도 영향을 줄 수 있다.

즉,

> 동적 메모리 할당은 힙 내부 자료구조 문제인 동시에,  
> 가상 메모리와 locality 위에서 동작하는 시스템 문제다.

# 35. 학습자가 가장 자주 헷갈리는 지점 정리

## 35.1 힙과 스택의 차이

### 스택
- 함수 호출에 따라 자동 관리
- 일반적으로 호출/복귀 구조에 잘 맞는 LIFO 패턴으로 사용됨
- 관리가 단순하고 빠름
- 수명 짧은 지역 변수에 적합

### 힙
- 프로그래머가 명시적으로 관리 (`malloc/free`)
- 수명과 크기가 유연함
- 관리 비용과 오류 가능성이 큼

---

## 35.2 가상 메모리와 힙은 같은가?

아니다.

- 가상 메모리는 **전체 주소 공간을 관리하는 큰 개념**
- 힙은 그 주소 공간 안의 **한 영역**
- 동적 메모리 할당기는 그 힙 영역 내부를 관리하는 소프트웨어 계층

---

## 35.3 `malloc`이 곧바로 물리 메모리를 주는가?

직접적으로 그렇게 이해하면 틀리다.

`malloc`은 프로세스의 가상 주소 공간 안에서 힙 블록을 관리한다. 실제 물리 페이지는 필요 시 운영체제가 매핑하고, 실제 접근 시점에 페이지 폴트로 적재될 수도 있다.

---

## 35.4 `free`하면 바로 OS에 반환되는가?

항상 그렇지는 않다.

많은 경우 `free`는 할당기 내부에서 그 블록을 다시 가용 상태로 표시할 뿐이다. 즉, 해당 메모리는 같은 프로세스 안에서 이후 `malloc` 요청에 재사용될 수 있다.

# 36. 시험/면접/과제에서 중요한 설명 포인트

다음 질문에 자기 말로 답할 수 있으면 9장의 핵심을 잘 잡은 것이다.

## 36.1 가상 메모리는 왜 필요한가?

- 큰 주소 공간의 환상 제공
- 프로세스 간 보호
- 효율적인 메모리 사용
- 파일/라이브러리 공유와 매핑 지원

## 36.2 페이지 테이블과 TLB의 차이는?

- TLB: 최근 번역 결과를 저장하는 캐시

## 36.3 페이지 폴트는 언제 발생하는가?

- 접근한 가상 페이지가 물리 메모리에 없을 때

## 36.4 동적 메모리 할당기의 핵심 문제는?

- 빠르게 적절한 free 블록 찾기
- 단편화 줄이기
- 분할/병합 관리
- 처리량과 이용률 균형 맞추기

# 37. 비전공자를 위한 핵심 비유

## 37.1 책상과 창고 비유

- 책상 위: RAM
- 창고: 디스크
- 자주 쓰는 자료만 책상 위에 둔다
- 필요 없으면 치우고, 필요하면 다시 가져온다

이 비유는 페이지와 페이지 폴트의 감각을 잡는 데 좋다.

---

## 37.2 호텔 방 번호 비유

- 손님이 받은 방 번호: 가상 주소
- 실제 배정 정보: 페이지 테이블
- 프런트 데스크: OS/MMU 협력

이 비유는 “보이는 주소와 실제 위치는 다르다”는 점을 설명하기 좋다.

---

## 37.3 창고 칸막이 관리자 비유 (`malloc`/`free`)

- 큰 창고를 여러 칸으로 나눔
- 손님이 작은 칸, 큰 칸을 요청
- 비어 있는 칸을 골라 배정
- 반납되면 다시 비움
- 붙어 있는 빈 칸은 합쳐 더 큰 칸으로 만듦

이 비유는 분할, 병합, 단편화를 설명하기 좋다.

# 38. 9장 전체 흐름을 한 번에 연결하기

지금까지의 내용을 하나의 서사로 다시 정리하면 다음과 같다.

1. 실제 RAM은 작고 공유되어야 한다
2. 그래서 각 프로세스에 큰 독립 주소 공간이라는 추상화를 제공한다
3. 이 공간은 페이지로 나뉘고, 페이지 테이블을 통해 물리 메모리에 매핑된다
4. TLB는 번역을 빠르게 한다
5. 페이지가 RAM에 없으면 page fault가 나고, 디스크에서 가져온다
6. 파일과 라이브러리는 메모리에 매핑될 수 있다
7. 프로세스의 주소 공간 안에는 코드, 데이터, 힙, 스택이 있다
8. 그중 힙은 동적 메모리 할당기가 관리한다
9. 할당기는 블록을 찾고, 분할하고, 해제하고, 병합한다
10. 이 과정 전체는 성능과 단편화의 trade-off 위에서 설계된다

이 10개의 연결고리를 이해하면 9장을 단편 지식이 아니라 **하나의 시스템 그림**으로 볼 수 있다.

# 40. 마무리 요약

CS:APP 9장은 표면적으로는 가상 메모리를 다루지만, 본질적으로는 **추상화와 현실의 차이를 관리하는 방법**을 다룬다.

- 프로그램은 연속적이고 독립적인 메모리를 가진 것처럼 보인다
- 실제로는 페이지 단위로 쪼개져 물리 메모리와 디스크 사이에서 관리된다
- 주소 변환, 보호, 캐싱, 매핑이 이 시스템을 구성한다
- 힙은 그 가상 주소 공간의 일부이며, 동적 메모리 할당기는 힙을 효율적으로 운영하는 소프트웨어다
- 좋은 할당기는 빠르면서도 단편화를 줄여야 하며, 그 과정에서 다양한 설계 trade-off를 가진다

9장의 진짜 핵심은 다음 문장으로 요약할 수 있다.

> 가상 메모리는 프로그램에게 단순한 세계를 보여주고,  
> 그 뒤에서 운영체제와 하드웨어가 복잡성을 관리한다.  
> 동적 메모리 할당은 그 가상 세계 안에서 다시 한 번 공간을 효율적으로 나누는 문제다.
