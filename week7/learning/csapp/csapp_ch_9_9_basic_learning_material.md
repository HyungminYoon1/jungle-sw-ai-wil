# 1. 왜 9.9절을 공부해야 하는가

우리가 C에서 다음과 같은 코드를 쓸 때를 생각해 보자.

```c
int *arr = malloc(sizeof(int) * 10);
free(arr);
```

이 코드는 매우 짧지만, 내부에서는 생각보다 많은 일이 일어난다.

- 필요한 메모리 크기를 계산한다.
- 힙 안에서 그 크기에 맞는 공간을 찾는다.
- 찾은 공간을 사용 중인 블록으로 표시한다.
- `free`가 호출되면 다시 사용 가능한 블록으로 바꾼다.
- 필요하면 인접한 빈 블록을 합쳐서 더 큰 빈 공간을 만든다.

즉, `malloc`과 `free`는 단순 함수 호출이 아니라, **힙이라는 공간을 관리하는 작은 관리자**를 부르는 것이다.

CS:APP 9.9절은 바로 이 관리자의 원리를 설명한다.
Malloc Lab은 그 원리를 실제 코드로 구현해 보는 과제다.

# 2. 9.9절을 읽기 전에 필요한 최소 개념

## 2.1 프로세스의 메모리 공간

프로그램은 대략 다음과 같은 메모리 구조를 가진다고 생각하면 된다.

```text
낮은 주소
+-------------------+
| 코드(text)        |
+-------------------+
| 전역 데이터        |
+-------------------+
| 힙(heap)          |  <- 위로 커짐
|                   |
|                   |
|                   |
|                   |
| 스택(stack)       |  <- 아래로 커짐
+-------------------+
높은 주소
```

여기서 이번 주에 중요한 것은 **힙(heap)** 이다.

- 스택은 함수 호출과 지역 변수를 위해 자동으로 관리된다.
- 힙은 프로그램이 실행 중에 직접 요청해서 쓰는 공간이다.
- `malloc`은 힙에서 공간을 빌려 주고, `free`는 그 공간을 반납받는다.

추가로 공지 키워드와 연결해서 보면:

- 코드(text): 실행할 명령이 들어 있는 영역
- 전역 데이터: 전역 변수, `static` 변수처럼 프로그램이 시작할 때부터 잡혀 있는 영역
- 힙(heap): 실행 중에 `malloc` 같은 방식으로 늘고 줄 수 있는 영역
- 스택(stack): 함수 호출에 따라 자동으로 쌓이고 빠지는 영역

여기서 공지에 나온 **데이터 세그먼트(data segment)** 는 보통
위 그림에서 `전역 데이터`에 해당하는 영역이라고 이해하면 된다.
즉, 데이터 세그먼트와 힙은 둘 다 메모리 공간이지만 역할이 다르다.

짧은 예시로 보면 더 쉽다.

```c
int global_count = 0;      // 전역 데이터(데이터 세그먼트 쪽)

void foo(void) {
    int local_count = 1;   // 스택
    int *arr = malloc(sizeof(int) * 10); // 힙
}
```

즉:

- 전역 변수, `static` 변수 -> 데이터 세그먼트 쪽
- 함수 안의 일반 지역 변수 -> 스택
- `malloc`으로 받은 메모리 -> 힙

## 2.2 가상 주소 공간과 힙의 관계

CS:APP 9장 앞부분을 보지 않으면 여기서 첫 번째로 막히기 쉽다.

프로그램은 실제 RAM을 직접 보는 것이 아니라,
**자기만의 가상 주소 공간(virtual address space)** 을 가진 것처럼 동작한다.

즉, 프로그램 입장에서는:

- "내가 쓰는 메모리 주소"가 있고
- 운영체제와 하드웨어가 그 주소를 실제 물리 메모리와 연결해 준다

라는 방식으로 이해하면 된다.

그림으로 아주 단순화하면 아래와 같다.

```text
프로그램이 보는 주소
[ 가상 주소 공간 ]
      |
      v
운영체제 + 하드웨어가 변환
      |
      v
[ 실제 물리 메모리(RAM) ]
```

즉, 프로그램은 직접 RAM의 어느 칸을 쓰는지 모르고,
그냥 자기 가상 주소 공간 안의 주소를 쓴다고 이해하면 된다.

여기서 중요한 감각 하나만 더 잡으면 된다.

- 프로그램이 쓰는 주소 = 가상 주소
- 실제 RAM 위치 = 물리 주소
- 둘은 같다고 보면 안 된다

이번 주에 꼭 필요한 핵심만 정리하면 아래와 같다.

- 프로그램은 **가상 주소 공간** 안에서 동작한다.
- 힙도 그 가상 주소 공간의 일부다.
- `malloc`은 "RAM에서 바로 공간을 꺼내 주는 함수"라기보다, **프로세스의 힙 영역 안에서 사용할 블록을 관리하는 함수**에 가깝다.
- 힙 공간이 모자라면, 그때 allocator가 운영체제 쪽 메커니즘과 연결된 확장 함수를 통해 더 큰 힙을 요청할 수 있다.

즉, 층위를 구분해서 보면 이해가 쉬워진다.

```text
프로그램 코드
   -> malloc / free 호출
      -> allocator가 힙 내부 블록 관리
         -> 필요하면 힙 확장 요청
            -> 운영체제/메모리 시스템과 연결
```

9.9절은 이 전체 중에서 **"allocator가 힙 내부 블록을 어떻게 관리하는가"** 에 집중하는 부분이다.

## 2.3 왜 힙 관리가 어려운가

힙은 스택과 다르게 사용 패턴이 일정하지 않다.

- 어떤 블록은 금방 반납된다.
- 어떤 블록은 오래 살아남는다.
- 요청 크기도 매번 다르다.

예를 들어 다음과 같은 요청이 들어오면 힙은 점점 복잡해진다.

```text
malloc(16)
malloc(32)
free(16)
malloc(8)
malloc(24)
free(32)
```

이런 요청이 반복되면, 힙에는 사용 중인 공간과 빈 공간이 섞여서 남는다.
그래서 할당기는 "빈 공간을 어떻게 찾고, 어떻게 다시 합칠 것인가"를 계속 결정해야 한다.

## 2.4 `sbrk`는 왜 언급되는가

Malloc Lab에서는 보통 실제 운영체제 대신 `mem_sbrk` 같은 교육용 함수가 등장한다.

의미는 단순하다.

- 힙 공간이 부족하면 더 큰 힙 공간을 요청한다.
- 그 요청을 아주 단순화해서 흉내 낸 것이 `mem_sbrk`다.

중요한 감각은 이것이다.

- `mem_sbrk`는 힙 끝을 늘린다.
- 그리고 보통 **새로 확보된 구간의 시작 주소**를 반환한다.
- allocator는 그 주소를 기준으로 "여기서부터 새 free block을 만들 수 있겠다"라고 생각한다.

이번 주에 꼭 기억할 점은 다음 정도면 충분하다.

- `malloc`은 항상 기존 빈 블록만 쓰는 것이 아니다.
- 필요한 경우 힙 자체를 더 늘릴 수도 있다.
- `mm_init`이나 `extend_heap`류의 함수는 이 과정을 담당한다.

## 2.5 시스템 콜과 `mmap`은 어디에 연결되는가

공지의 학습 키워드에는 `시스템 콜`, `sbrk/mmap`도 들어 있다.
9.9절만 읽으면 이 단어들이 갑자기 튀어나오는 것처럼 느껴질 수 있다.

비전공자 기준으로는 아래 정도로 이해하면 충분하다.

- **시스템 콜(system call)**:
  프로그램이 운영체제에게 기능을 요청하는 통로
- **`sbrk`**:
  전통적으로 힙 끝을 늘리는 방식과 연결되는 개념
- **`mmap`**:
  메모리 영역을 새로 매핑하는 방식과 연결되는 개념

즉, allocator는 평소에는 힙 내부 블록을 직접 관리하지만,
필요한 공간이 부족해지면 더 큰 메모리 영역을 확보해야 한다.
그때 운영체제와 연결되는 층위에서 `sbrk`나 `mmap` 같은 개념이 등장한다.

다만 이번 주 Malloc Lab에서는 이를 실제 운영체제 수준으로 깊게 구현하지 않는다.
보통은 `mem_sbrk` 같은 교육용 인터페이스로 단순화해서 다룬다.

따라서 이번 주 기준으로는 아래처럼 이해하면 충분하다.

```text
사용자 코드
-> malloc/free 호출
-> allocator가 힙 블록 관리
-> 공간이 부족하면 mem_sbrk 같은 단순화된 인터페이스 사용
-> 실제 운영체제 층위에는 system call, sbrk, mmap 같은 개념이 있다
```

## 2.6 공지 키워드 한 줄 정의

- 가상 주소 공간: 프로그램이 자기만의 메모리를 가진 것처럼 보는 주소 세계
- 데이터 세그먼트: 전역 변수와 `static` 변수처럼 프로그램 시작부터 존재하는 데이터 영역
- 시스템 콜: 프로그램이 운영체제 기능을 요청하는 통로
- `sbrk`: 힙 끝을 늘리는 방식과 연결되는 전통적 개념
- `mmap`: 메모리 영역을 새로 매핑해서 확보하는 방식과 연결되는 개념

# 3. 동적 메모리 할당이란 무엇인가

동적 메모리 할당은 프로그램 실행 중에 필요한 만큼 메모리를 요청하고 반납하는 방식이다.

대표 함수는 아래 네 가지다.

```c
void *malloc(size_t size);
void free(void *ptr);
void *realloc(void *ptr, size_t size);
void *calloc(size_t nmemb, size_t size);
```

이번 주에 핵심은 `malloc`, `free`, `realloc`이다.

## 3.1 `malloc(size)`

- `size` 바이트 이상 저장할 수 있는 공간을 요청한다.
- 성공하면 그 공간의 시작 주소를 반환한다.
- 실패하면 `NULL`을 반환한다.

## 3.2 `free(ptr)`

- 이전에 `malloc`으로 받은 공간을 반납한다.
- 반납했다고 해서 운영체제가 바로 메모리를 회수하는 것은 아니다.
- 우선은 **할당기 입장에서 다시 쓸 수 있는 블록**으로 표시하는 것이 핵심이다.

## 3.3 `realloc(ptr, size)`

- 기존 블록 크기를 바꾸고 싶을 때 사용한다.
- 더 큰 공간이 필요하면 새 블록을 만들고 복사할 수 있다.
- 운이 좋으면 기존 위치를 유지한 채 확장할 수도 있다.

초보자 관점에서는 다음 정도만 먼저 이해하면 충분하다.

- `realloc`은 생각보다 어렵다.
- 가장 단순한 구현은 `malloc -> memcpy -> free` 흐름이다.
- 이후 성능을 높이려면 제자리 확장 같은 최적화를 고민한다.

# 4. 힙 안에서는 무엇을 관리하는가

할당기는 힙을 보통 **블록(block)** 단위로 나누어 관리한다.

블록은 크게 두 부분으로 생각할 수 있다.

- 메타데이터
- 실제 사용자가 쓰는 payload

간단한 개념도는 아래와 같다.

```text
+-----------+----------------------+
| metadata  | payload              |
+-----------+----------------------+
```

여기서 아주 중요한 점이 하나 있다.

- `malloc`이 반환하는 포인터는 보통 **payload의 시작 주소**다.
- 사용자는 payload만 사용한다.
- metadata는 allocator가 내부적으로만 관리한다.

즉, 사용자는 보통 이런 식으로 쓰게 된다.

```text
[ header | payload | footer ]
           ^
        malloc이 주는 포인터
```

이 감각이 없으면, 이후에 header/footer가 왜 필요한지와
왜 사용자 코드가 그것을 직접 건드리면 안 되는지가 잘 연결되지 않는다.

즉, 사용자가 `arr[0] = 10;`처럼 쓰는 것은 payload 안쪽이고,
header/footer는 allocator가 내부적으로만 읽고 쓰는 정보다.

메타데이터에는 보통 이런 정보가 들어간다.

- 블록 크기
- 현재 블록이 사용 중인지 여부
- 경우에 따라 이전 블록 상태나 free list 연결 정보

왜 이런 정보가 필요할까.

- 다음 블록으로 이동하려면 현재 블록 크기를 알아야 한다.
- 빈 블록인지 사용 중 블록인지 알아야 재사용할 수 있다.
- 인접 블록과 합칠 수 있는지 판단해야 한다.

# 5. 정렬(alignment)은 왜 중요한가

할당기는 요청 크기만큼 딱 맞춰 주지 않고, 보통 일정 단위로 반올림한다.

예를 들어 8바이트 정렬을 사용하면:

- `malloc(1)`도 실제로는 8바이트 이상 할당될 수 있다.
- `malloc(13)`도 16바이트 또는 그 이상으로 반올림될 수 있다.

정렬이 필요한 이유는 다음과 같다.

- 하드웨어가 특정 크기 경계에 맞는 접근을 더 자연스럽게 처리한다.
- 다양한 타입의 데이터를 안전하게 저장하기 쉽다.
- 블록 레이아웃을 계산하기 쉬워진다.

Malloc Lab starter code에서도 `ALIGNMENT 8` 같은 상수를 볼 수 있다.
즉, 이 과제는 "요청 크기를 그냥 그대로 넘겨주는 문제"가 아니다.

비전공자가 가장 자주 헷갈리는 부분은 아래 차이다.

| 구분 | 의미 |
|------|------|
| 요청 크기 | 사용자가 `malloc(13)`처럼 직접 요청한 크기 |
| 정렬 후 payload 크기 | 정렬 조건을 맞추기 위해 반올림된 사용자 영역 크기 |
| 실제 블록 크기 | payload에 metadata까지 더한 allocator 내부 관리 단위 |

예를 들어 8바이트 정렬이라고 하면:

| 항목 | 예시 |
|------|------|
| 요청 크기 | 13 bytes |
| 정렬 후 payload 크기 | 16 bytes |
| 실제 블록 크기 | 16보다 큼 |

왜 마지막이 16보다 클까?

- allocator는 payload만 저장하는 것이 아니라
- header/footer 같은 metadata도 같이 관리해야 하기 때문이다

정확한 실제 블록 크기는 구현에서 metadata를 어떻게 두느냐에 따라 달라진다.
하지만 **"사용자 요청 크기 < 실제 블록 크기일 수 있다"** 는 감각은 꼭 필요하다.

---

# 6. 단편화(fragmentation)는 왜 생기는가

동적 메모리 할당기에서 가장 중요한 문제 중 하나가 단편화다.

## 6.1 내부 단편화

요청보다 더 큰 블록을 주어서 생기는 낭비다.

예:

- 사용자는 13바이트를 원했다.
- 할당기는 정렬 때문에 16바이트 또는 그 이상을 준다.
- 남는 부분이 내부 단편화다.

## 6.2 외부 단편화

전체 빈 공간의 합은 충분하지만, 붙어 있는 큰 빈 공간이 없어서 할당 실패 또는 비효율이 생기는 경우다.

예:

```text
[사용중 16][빈 8][사용중 16][빈 8][사용중 16]
```

빈 공간은 총 16바이트지만, 16바이트 연속 블록이 없으면 큰 요청을 바로 처리하기 어렵다.

그래서 할당기는 단순히 "빈 블록이 있는가"만 보면 안 되고, **어떤 크기의 빈 블록이 어떻게 흩어져 있는가**도 봐야 한다.

---

# 7. implicit free list란 무엇인가

이번 주의 핵심 자료구조는 **implicit free list** 다.

이름이 어렵지만 뜻은 단순하다.

- 힙 전체를 앞에서부터 순서대로 훑어 본다.
- 각 블록의 header를 읽는다.
- 빈 블록인지 확인한다.
- 조건에 맞는 블록을 찾으면 거기에 할당한다.

즉, 별도의 연결 리스트 포인터가 있는 것이 아니라, **힙에 있는 블록들을 순서대로 따라가며 free block을 찾는 방식**이다.

## 7.1 왜 implicit라고 부르는가

free block만 따로 연결한 명시적 리스트가 없다.
블록들의 물리적 순서 자체가 탐색 경로 역할을 한다.

즉:

- explicit free list: free block끼리 포인터로 직접 연결
- implicit free list: 힙 전체를 순서대로 스캔

이번 주 기본 구현은 implicit free list를 이해하는 데 집중하면 된다.

## 7.2 장점과 단점

장점:

- 구현이 단순하다.
- 블록 구조를 이해하기 좋다.
- split, coalesce 같은 기본 원리를 익히기 좋다.

단점:

- free block을 찾기 위해 힙 전체를 오래 훑을 수 있다.
- 요청이 많아질수록 느려질 수 있다.

즉, implicit free list는 **교육용으로는 매우 좋지만 성능은 제한적일 수 있는 기본 방식**이라고 보면 된다.

---

# 8. 블록을 할당할 때 무슨 일이 일어나는가

`mm_malloc` 또는 `malloc`이 호출되면 할당기는 대체로 아래 순서로 움직인다.

## 8.1 요청 크기 보정

먼저 요청 크기를 바로 쓰지 않는다.

- 너무 작은 요청이면 최소 블록 크기 이상으로 맞춘다.
- 정렬 단위에 맞게 반올림한다.
- header/footer 같은 메타데이터 공간도 고려한다.

## 8.2 빈 블록 찾기

그다음 free block 중에서 들어갈 수 있는 블록을 찾는다.

여기서 쓰는 정책이 fit policy다.

- first fit: 처음으로 맞는 블록 사용
- next fit: 이전 탐색 위치부터 계속 찾기
- best fit: 가장 잘 맞는 블록 찾기

비전공자 기준으로 이번 주에는 **first fit을 이해하는 것**이 가장 중요하다.

## 8.3 필요하면 블록 나누기(split)

찾은 free block이 너무 크다면 한 덩어리를 통째로 쓰는 대신 둘로 나눌 수 있다.

예:

```text
원래 free block: [      64 bytes      ]
요청 후 사용:    [ 24 alloc ][ 40 free ]
```

이렇게 하면 남는 공간을 다시 free block으로 유지할 수 있다.

## 8.4 적당한 블록이 없으면 힙 확장

기존 free block으로 해결이 안 되면 힙을 더 늘린다.

즉:

- free block 탐색 실패
- `mem_sbrk`로 힙 확장
- 새 공간을 free block처럼 다루고 다시 할당

## 8.5 한 번에 보는 할당 흐름 예시

처음 배우는 사람은 split과 free block 재사용이 따로따로 보이면 이해가 끊기기 쉽다.
그래서 한 번에 이어서 보면 좋다.

처음 상태:

```text
[ free 64 ]
```

`malloc(20)` 요청:

- 정렬과 메타데이터를 고려해 24 정도가 필요하다고 하자.
- 64짜리 free block을 찾았다.
- 너무 크므로 split한다.

결과:

```text
[ alloc 24 ][ free 40 ]
```

다음에 `malloc(16)` 요청:

- 앞의 `alloc 24`는 이미 사용 중이다.
- 뒤의 `free 40`을 사용한다.

결과:

```text
[ alloc 24 ][ alloc 16 ][ free 24 ]
```

이 흐름이 바로 `implicit free list`에서
"앞에서부터 훑으며 free block을 찾고, 남으면 나눈다"는 뜻이다.

---

# 9. 블록을 반납할 때 무슨 일이 일어나는가

`free(ptr)`는 겉보기에는 단순하지만, 실제로는 단편화 문제와 직접 연결된다.

## 9.1 일단 free 표시

가장 먼저 하는 일은 해당 블록을 "이제 사용 중이 아님"이라고 표시하는 것이다.

## 9.2 인접 빈 블록과 병합(coalesce)

바로 옆 블록들도 free 상태라면, 하나의 큰 블록으로 합칠 수 있다.

예:

```text
병합 전: [ free 16 ][ free 24 ]
병합 후: [ free 40 ]
```

왜 중요할까.

- 작은 free block 여러 개가 흩어져 있으면 큰 요청을 처리하기 어렵다.
- 병합을 하면 외부 단편화를 줄일 수 있다.

coalescing은 9.9절과 malloclab에서 반드시 이해해야 하는 핵심 개념이다.

## 9.3 한 번에 보는 반납과 병합 예시

아까와 이어서 아래 상태를 생각해 보자.

```text
[ alloc 24 ][ alloc 16 ][ free 24 ]
```

여기서 가운데 블록을 `free`하면:

```text
[ alloc 24 ][ free 16 ][ free 24 ]
```

이제 뒤쪽에 free block이 바로 붙어 있으므로 병합할 수 있다.

```text
[ alloc 24 ][ free 40 ]
```

만약 앞뒤가 모두 free였다면,
세 블록을 한 번에 더 큰 free block으로 합칠 수도 있다.

이것이 coalescing이 필요한 이유다.
병합을 하지 않으면 free 공간이 잘게 쪼개진 채로 남아 큰 요청을 처리하기 어려워진다.

# 10. header와 footer는 왜 필요한가

많은 기본 allocator 구현은 header와 footer를 사용한다.

개념적으로는 다음처럼 생각하면 된다.

```text
+---------+------------------+---------+
| header  | payload          | footer  |
+---------+------------------+---------+
```

실제로는 아래처럼 생각하면 더 이해하기 쉽다.

```text
[ header(size, alloc) | payload(사용자 영역) | footer(size, alloc) ]
                       ^
                       bp 또는 malloc이 준 포인터
```

header에는 보통 현재 블록 크기와 할당 여부가 들어간다.
footer는 비슷한 정보를 블록 끝에도 두는 방식이다.

footer가 있으면 좋은 점:

- 이전 블록 정보를 거꾸로 확인하기 쉬워진다.
- coalescing 구현이 단순해질 수 있다.

하지만 비용도 있다.

- 메타데이터 공간이 늘어난다.
- 작은 블록이 많을수록 오버헤드가 커질 수 있다.

이번 주 학습에서는 "왜 footer가 필요한가"를 이해하는 것이 중요하다.
항상 가장 최적의 방식인지까지 고민하는 것은 그다음 단계다.

비전공자가 여기서 자주 막히는 부분이 포인터 산술이다.

allocator 코드에서는 종종 이런 생각을 하게 된다.

- "현재 블록의 header는 어디 있지?"
- "다음 블록은 어디서 시작하지?"

이때 중요한 감각은 아래다.

- payload 포인터가 있으면 header는 그 바로 앞 metadata 영역에 있다
- 현재 블록 크기를 알면 다음 블록 시작 위치로 이동할 수 있다
- allocator는 바이트 단위 계산을 위해 `char *`로 캐스팅해서 포인터를 움직이는 경우가 많다

즉, `(char *)bp + size` 같은 코드는
"현재 위치에서 size 바이트만큼 앞으로 이동"이라는 뜻으로 읽으면 된다.

아주 단순하게 보면:

```text
현재 블록 시작 ----(현재 블록 크기만큼 이동)----> 다음 블록 시작
```

추가로 한 가지 더 구분하면 좋다.

- CS:APP 9.9에서 설명하는 allocator 구조는 **학습용으로 정리된 전형적인 구조**다.
- 실제 starter `mm.c`는 이런 구조가 완성된 상태가 아니라, 매우 단순한 출발점일 수 있다.

즉, 책은 "어떤 구조를 구현해야 하는가"를 설명하고,
starter 코드는 "여기서부터 직접 만들어 보라"는 출발점이라고 보면 된다.

---

# 11. `mm_init`, `mm_malloc`, `mm_free`, `mm_realloc`를 어떻게 이해할 것인가

Malloc Lab에서는 보통 아래 네 함수를 구현한다.

```c
int mm_init(void);
void *mm_malloc(size_t size);
void mm_free(void *ptr);
void *mm_realloc(void *ptr, size_t size);
```

각 함수는 아래처럼 이해하면 된다.

## 11.1 `mm_init`

역할:

- 힙 관리자 초기화
- 시작용 블록 구조 세팅
- 필요 시 초기 힙 공간 확보

여기서 비전공자가 자주 헷갈리는 점:

- `mm_init`은 메모리를 "할당받는 함수"라기보다
- **할당기가 앞으로 블록을 관리할 수 있도록 힙의 시작 상태를 준비하는 함수**에 가깝다.

조금 더 쉽게 말하면,
`mm_init`은 사용자가 바로 쓸 메모리를 주는 함수라기보다
**allocator가 나중에 헷갈리지 않도록 시작 기준점을 세팅하는 함수**다.

예를 들어:

- 힙의 시작 위치를 정리해 둔다
- 첫 순회가 가능하도록 기준 블록을 둔다
- 필요하면 초기 free block을 만들어 둔다

즉, `mm_init`이 제대로 되어 있어야
이후 `mm_malloc`과 `mm_free`가 어디서부터 힙을 보고 움직여야 할지 알 수 있다.

한 문장으로:

`할당기를 사용할 준비를 하는 함수`

## 11.2 `mm_malloc`

역할:

- 요청 크기 조정
- free block 탐색
- 적절한 블록 할당
- 필요 시 split
- 필요 시 힙 확장

한 문장으로:

`요청에 맞는 블록을 찾아서 사용자에게 돌려주는 함수`

## 11.3 `mm_free`

역할:

- 블록을 free 상태로 바꿈
- 필요 시 인접 free block과 병합

한 문장으로:

`반납된 블록을 다시 쓸 수 있게 만들고, 가능하면 주변 빈 블록과 합치는 함수`

## 11.4 `mm_realloc`

역할:

- 블록 크기 변경
- 가장 단순한 경우 새 블록 할당 후 복사
- 더 나은 경우 기존 위치 유지 시도

한 문장으로:

`기존 데이터는 최대한 유지하면서 블록 크기를 바꾸는 함수`

---

# 12. `realloc`이 특히 어려운 이유

초보자 입장에서 `realloc`은 보통 제일 헷갈린다.

이유는 다음과 같다.

- 줄이는 경우와 늘리는 경우가 다르다.
- 늘릴 때 바로 옆 free block을 합치면 복사를 피할 수 있다.
- 안 되면 새 블록을 만들고 데이터를 복사해야 한다.
- 기존 데이터가 손상되면 안 된다.

그래서 처음에는 아래처럼 이해하면 충분하다.

1. 가장 단순한 `realloc`부터 구현한다.
2. correctness를 먼저 확보한다.
3. 그다음 필요하면 제자리 확장 같은 최적화를 고민한다.

즉, `realloc`은 처음부터 멋지게 구현하려고 하기보다, **맞게 동작하는지**를 먼저 보는 것이 맞다.

# 16. 자주 하는 오해

## 16.1 "`free`를 하면 운영체제에 바로 반환된다"

기본 학습 단계에서는 이렇게 이해하면 안 된다.
우선은 할당기 내부에서 다시 사용할 수 있는 블록으로 바뀐다고 보는 것이 맞다.

## 16.2 "빈 블록만 있으면 어떤 요청이든 처리할 수 있다"

그렇지 않다.
빈 블록이 흩어져 있으면 큰 요청을 처리하지 못할 수 있다.
이것이 외부 단편화 문제다.

## 16.3 "`realloc`은 그냥 `malloc`과 `free`를 합친 것이다"

가장 단순한 구현은 비슷하지만, 실제로는 훨씬 더 어렵다.
기존 데이터 보존과 성능까지 고려해야 하기 때문이다.

## 16.4 "implicit free list를 알면 끝이다"

아니다.
이번 주 기본 구현을 이해하는 데는 충분하지만, 성능을 높이려면 이후 explicit free list 같은 확장이 필요할 수 있다.
다만 지금은 거기까지 욕심내기보다 기본 흐름을 먼저 잡는 것이 맞다.

# 17. 이번 주 학습의 핵심 정리

- 9.9절의 핵심은 힙 위에서 동적 메모리 할당기를 어떻게 설계하는가이다.
- 할당기는 블록 단위로 힙을 관리한다.
- 정렬, 단편화, split, coalesce는 모두 allocator 설계의 핵심 요소다.
- 기본 구현에서는 implicit free list를 통해 힙을 순차 탐색하며 free block을 찾는다.
- `mm_init`, `mm_malloc`, `mm_free`, `mm_realloc`는 각각 초기화, 할당, 반납, 크기 변경을 담당한다.
- 이번 주에는 고급 최적화보다 기본 동작 원리를 정확히 이해하는 것이 우선이다.
