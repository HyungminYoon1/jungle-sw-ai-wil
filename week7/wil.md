## 2. 시도한 접근 방식
- 먼저 `week7/learning/csapp` 아래 학습 문서를 만들어서 읽고, 이번 주 메모리 학습 범위를 좁혔다. 높은 점수를 출력하는 allocator를 처음부터 완성하는 것보다 `implicit free list`의 기본 흐름과 힙 블록 구조를 이해하는 데 우선순위를 뒀다.
- 퀴즈 리뷰 문서를 따로 두고 `데이터 세그먼트/힙/스택`, `malloc`, `mmap`, `단편화`를 짧게 설명하는 연습을 했다.
- 수요 코딩회 쪽은 우선 팀원 4명이 6주차에 작성한 SQL 처리기를 모아두고 1개를 선정하였고, 이후 구조를 점검하고 설계 문서를 다시 작성하였다. 그 다음 7주차에 사용할 기능인 `자동 ID`, `인덱스 등록`, `WHERE id = ?`, `선형 탐색 비교`, `벤치마크`를 각각 어느 계층에 붙여야 하는지 문서로 작성하고 리팩톨링을 진행하였다.
- B+ Tree는 별도의 학습자료를 만들어 읽고 유튜브 강의 동영상을 보고 학습하였다. SQL 처리기를 리팩토링하고 코드 리뷰를 할 때 `CSV는 영속 저장소`, `인덱스는 메모리`, `execution 계층이 인덱스 경로를 선택`, `storage는 CSV 읽기/쓰기만 담당`이라는 계층별 데이터 처리 흐름을 이해하려고 했다.
- 구현 결과를 README에 기록하였고 `docs/requirements.md`, `docs/architecture.md`, `docs/test-cases.md`, `make test` 와 같은 설계 및 테스트 내용도 문서로 남겼다. 이번 주 수요코딩회는 B+ Tree 개념을 학습하고, 최소 구현을 통해 팀원 모두가 프로젝트 구조를 설명할 수 있도록 하는 것에 더 신경을 썼다.

## 3. 문제와 해결 과정

### Malloc Lab에서 포인터와 블록 구조를 함께 봐야 했던 점
- **문제:** 처음에는 `mm_malloc`, `mm_free`, `mm_realloc` 같은 함수들이 단순히 "메모리를 할당하고 해제한다" 정도로 단순하게 생각했는데, 막상 공부를 해보니 힙 안의 블록 구조와 포인터 산술을 같이 이해해야 해서 더 복잡했다. 처음 학습 때 header/footer가 왜 필요한지, free block을 어떻게 찾고 왜 coalesce를 해야 하는지 감을 잡기 위해 시간을 사용하였다.
- **해결:** CSAPP 책을 보면서 `implicit free list` 코드를 작성하였고, 각각의 함수에 주석을 달아서 힙을 어떻게 순회하는지, split과 coalesce가 어떤 상황에서 필요한지 등을 정리했다. 이 과정을 거치고 나니 allocator가 힙 안의 블록 상태를 어떻게 관리하는지 파악할 수 있었고 이후 `explicit free list` 와 `segregated free list` 으로의 리팩토링이 수월해졌다.

### Malloc Lab에서 구현과 튜닝을 분리해서 봐야 했던 점
- **문제:** 처음에는 구현이 되기만 하면 바로 높은 점수가 나올 것이라고 생각했는데, 실제로는 correctness를 맞추는 것과 점수를 끌어올리는 것은 다른 문제였다. 기본 구조를 맞춘 뒤에도 어떤 free list 구조를 쓰는지, 어떤 요청 크기를 어떻게 분리하는지에 따라 성능이 꽤 달라졌다.
- **해결:** 먼저 책에서 작성된 코드를 따라하며 `implicit free list` allocator를 만들고, 그 이후에 `segregated free list` 쪽으로 구조를 확장하며 점수를 올리는 방식으로 접근했다. 일반적인 튜닝 범위에서는 80점대 후반에서 90점대 초반 정도가 나왔고, 이후 trace 성향이 강하게 반영되는 튜닝에서는 더 높은 점수도 확인할 수 있었다. 이 과정을 통해 allocator는 구현 자체도 어렵지만, 어떤 정책을 택하느냐에 따라 성능이 크게 달라지는 과제라는 점을 느꼈다.

### SQL 처리기에서 숨은 ID 정책을 정하는 일이 가장 중요했던 점
- **문제:** 7주차 SQL 처리기에서 가장 먼저 막힌 부분은 사용자 스키마에 없는 `id`를 어떻게 다뤄야 하는가였다. 사용자 컬럼으로 `id`를 추가하면 기존 구조와 충돌할 수 있고, 반대로 아무 기준 없이 내부 ID를 붙이면 조회 규칙과 저장 규칙이 헷갈릴 수 있었다.
- **해결:** 내부에서는 `__internal_id`를 사용하고, SQL 표면에서는 `WHERE id = ?`로만 접근하게 하는 방식으로 정리했다. 이렇게 정리하고 나니 "사용자 컬럼"과 "시스템이 관리하는 키"를 분리해서 생각할 수 있었고, `SELECT *`와 `SELECT id`의 동작 차이도 더 자연스럽게 설명할 수 있었다.

### 인덱스를 넣으면서도 기존 구조를 깨지 않는 것이 어려웠던 점
- **문제:** B+ Tree를 넣는다고 해서 전체 구조를 크게 바꾸면, 지난주에 만든 SQL 처리기의 흐름 자체가 흔들릴 수 있었다. 처음에는 인덱스를 넣으면 parser나 storage까지 많이 바꿔야 하는 것 아닌가 생각했는데, 그렇게 가면 과제 범위가 지나치게 커질 수 있었다.
- **해결:** 기존 `app -> sql -> execution -> storage` 흐름은 유지하고, `execution`에서만 `WHERE id = ?`를 만나면 `index` 계층으로 분기하는 방식으로 구조를 잡았다. 이렇게 하니 일반 `WHERE`는 그대로 선형 탐색을 유지하고, 인덱스 조회 경로만 따로 설명할 수 있어서 구조가 더 명확해졌다.

### 성능 비교는 '빠르다'고 말하는 것보다 측정 기준을 정하는 일이 더 중요했던 점
- **문제:** 처음에는 인덱스를 붙인 뒤 조회를 한 번 실행해서 빨라졌다고 보여주면 충분할 것 같았는데, 실제로는 인덱스 재구성 비용이 섞이면 결과 해석이 애매해질 수 있었다. 특히 CLI 첫 실행 결과만으로는 정말 조회가 빨라진 것인지, 준비 비용이 같이 섞인 것인지 분리해서 보기 어려웠다.
- **해결:** 그래서 데이터 준비와 조회 성능 측정을 분리하는 방식으로 생각을 바꿨다. `prepare`로 데이터셋을 만들고, `query-only`로 조회만 반복 측정하게 나누니 인덱스 조회와 선형 탐색의 차이를 더 설득력 있게 보여줄 수 있었다. 이 과정을 통해 성능 비교는 측정 기준까지 포함해 설계해야 하는 일이라는 점을 느꼈다.

### 테스트는 정상 동작보다 예외 상황에서 더 큰 의미가 있었던 점
- **문제:** 구현이 어느 정도 끝나면 "이제 동작한다"는 생각이 먼저 들었지만, 정상 케이스만 보면 놓치는 부분이 많았다. `WHERE id = abc` 같은 잘못된 입력, `SELECT *`에서 숨은 내부 ID를 보여주지 않아야 하는 정책, 인덱스 무효화 후 재구성 같은 조건은 직접 챙기지 않으면 빠지기 쉬웠다.
- **해결:** 이번에는 단순히 성공 케이스만 보는 것이 아니라, 정책이 드러나는 실패 케이스와 경계 조건도 같이 확인하려고 했다. 그러고 나니 테스트가 단순한 확인 절차가 아니라 "우리가 이 시스템을 어떤 방식으로 해석하고 있는가"를 드러내는 문서 역할도 한다는 점을 더 분명히 느꼈다.

## 4. 새로 배운 점
- allocator를 이해하려면 `malloc/free`를 함수 이름으로 외우는 것보다 힙 안의 블록이 어떤 레이아웃으로 놓이고, 그 블록들이 어떤 규칙으로 재사용되는지를 보는 것이 더 중요하다는 점을 배웠다.
- `implicit free list`는 단순한 시작점이지만, 전체 힙을 순회해야 한다는 구조적 한계가 분명하다. 그래서 이후 `segregated free list` 같은 구조가 왜 필요한지도 더 자연스럽게 이해할 수 있었다.
- `sbrk/mmap` 같은 키워드도 운영체제 개념으로 따로 떨어져 있는 것이 아니라, 결국 "필요한 메모리를 어떻게 더 얻는가"라는 allocator의 문제와 연결된다는 점이 더 선명해졌다.
- B+ Tree는 단순히 탐색이 빠른 트리라기보다, 키를 기준으로 데이터 위치를 빠르게 찾기 위한 인덱스 구조라는 점이 중요했다. 이번 주 과제에서는 `id -> row_offset` 매핑으로 단순화해 이해하니 훨씬 명확했다.
- 인덱스를 도입한다고 해서 모든 조회를 바꾸는 것이 항상 좋은 것은 아니다. `WHERE id = ?`만 인덱스를 타게 하는 방식처럼, 필요한 경로만 최적화하는 것이 오히려 더 현실적인 설계일 수 있다는 점을 배웠다.
- 테스트와 README는 구현이 끝난 뒤 덧붙이는 자료가 아니라, 구조와 정책을 설명하고 검증하는 기준이 될 수 있다는 점을 다시 느꼈다.
