## 2. 시도한 접근 방식
- 알고리즘 문제를 풀 때는 바로 코드를 작성하기보다, 먼저 "이 문제의 상태는 무엇인가", "이전 결과를 재사용할 수 있는가", "현재 선택이 이후 선택에 어떤 영향을 주는가"를 적어본 뒤 풀이를 시작하려고 했다.
- DP 문제는 작은 입력에서 어떤 값이 반복해서 다시 등장하는지 찾고 이를 기반으로 점화식을 세우는 방식으로 접근했다.
- Greedy 문제는 현재의 최선의 선택이 차후에도 최적해임을 보장할 수 있을 때 선택하는 방식으로 당장 최선처럼 보이는 선택이 어째서 전체 최적해로 이어지는지 생각해보고, 반례를 가정하며 검증하려고 했다.
- 코어타임 DP 문제 세트는 팀원들과 동시에 푸는 대신에 각자 미리 문제를 풀어보고 팀 코어타임 시간에 팀원들에게 풀이를 설명하는 방식으로 접근했다. 이 과정에서 설명을 할 수 있어야 내가 진짜 이해한 것으로 볼 수 있다는 것을 느꼈다.
- 수요 코딩회 학습에서는 "상태와 Hook이 무엇인지", "setState는 값 변경 외에 무엇을 예약해야 하는지", "effect는 왜 렌더 중 바로 실행되면 안 되는지" 같은 런타임 관점의 질문을 중심으로 학습하였다.
- 과제의 요구에 맞는 리액트 같은 렌더러를 설계할 때 AI의 설계 제시안으로부터 라이브러리를 `공개 API`, `런타임`, `Hook`, `VNode`, `Resolver`, `Diff`, `Patch`, `시연 앱` 계층으로 나누는 방향을 이해하려고 했다. 책임 단위로 분할하니 구현이 덜 복잡해 보이고, 각 계층의 책임이 더 선명해졌다.
- 렌더러 및 시연 앱의 구현시 AI를 활용하여 코드를 구현하였고, 이후 기본적인 개념을 학습하고 AI 작성 코드를 이해하는데 시간을 주로 사용하였다.


## 3. 문제와 해결 과정

### DP를 "공식 암기"처럼 접근하면 응용이 막히는 점
- **문제:** 처음에는 DP를 유형별 점화식 패턴만 외우는 방식으로 접근하려고 해서, 문제가 조금만 달라져도 어떤 상태를 잡아야 할지 바로 떠오르지 않았다. 특히 `동전`, `LCS`, `평범한 배낭` 문제는 대략적인 패턴을 미리 공부하기는 했으나 내가 고민하며 체득한 패턴이 아니고, 결국 금방 해결하지 못한 채 gpt의 힌트를 조금씩 얻어가면서 해결할 수 있었다.
- **해결:** "무엇을 기억하면 다음 결정을 내릴 수 있는가"를 확인해서 상태를 정의하고 그 이후에 점화식을 세워야 한다. 예를 들어 `LCS`는 두 문자열의 접두사 길이, `배낭`은 아이템 인덱스와 남은 무게, `점프`는 현재 위치까지 도달 가능한 경우의 수처럼 먼저 상태를 올바로 정의할 수 있어야 이어서 점화식이 뒤따라 나왔다. 이 과정을 거치며 DP는 공식을 꺼내는 것이 아니라 상태를 설계하는 문제라는 점을 더 분명히 느꼈다.

### Greedy는 빠르지만, 선택의 근거가 없으면 불안정한 점
- **문제:** Greedy 문제는 구현 자체는 비교적 짧았지만, 왜 그 선택이 항상 맞는지 설명하기가 더 어려웠다. 특히 `신입 사원` 문제에서 막혀서 오랜 시간 고민하였다.
- **해결:** 반례를 찾아보는 습관이 필요하다는 것을 느꼈다. "이 기준 말고 다른 기준으로 고르면 언제 틀리는가"를 먼저 생각해 보니, 어떤 속성을 기준으로 정렬해야 하는지와 왜 그리디가 성립하는지를 더 또렷하게 설명할 수 있었다.

### 코어타임 DP 문제에서 말로 설명하는 과정이 생각보다 어려웠던 점
- **문제:** `Climbing Stairs`, `House Robber`, `Word Break`, `Triangle`는 팀원과 풀이를 공유하려 하니 상태 정의와 그 선정 이유를 명확하게 설명하는 것이 쉽지 않았다.
- **해결:** 각 문제를 "현재 결과를 결정하는 최소 정보가 무엇인가"로 정리했다. `House Robber`는 이전 집 선택 여부의 영향, `Word Break`는 특정 인덱스까지 분해 가능한지 여부, `Triangle`은 바로 윗행의 두 후보 중 최적값 재사용이라는 구조로 설명했다. 코어타임은 정답을 맞히는 시간이라기보다 사고 과정을 설명하는 시간이라는 점을 이번 주에 느꼈다.

### 함수형 컴포넌트에서 상태가 유지되는 원리가 직관적이지 않았던 점
- **문제:** React-like 런타임을 공부하면서 가장 처음 막힌 지점은 "함수는 다시 실행되는데 `useState` 값은 왜 사라지지 않는가"였다. 표면적으로는 함수 호출일 뿐인데, 상태가 유지된다는 점이 직관과 어긋났다.
- **해결:** 참고 문서의 `FunctionComponent`와 `hooks` 배열 설명을 보며, Hook이 변수 이름이 아니라 호출 순서 기반 슬롯으로 저장된다는 점을 이해했다. 상태는 함수 안에 있는 것이 아니라, 루트 컴포넌트를 감싸는 런타임 객체가 보관하고 있고, 렌더 시에는 `hookCursor`를 따라 각 슬롯을 다시 연결하는 방식이라는 것을 파악했다. 이 구조를 이해하고 나니 Hook 규칙이 왜 필요한지도 자연스럽게 이어졌다.

### `useEffect`와 `useMemo`를 단순 편의 기능으로만 보았던 점
- **문제:** 전에는 `useEffect`를 "렌더 후 뭔가 하는 것", `useMemo`를 "성능 최적화용" 정도로 막연하게 이해하고 있었다. 그래서 왜 effect는 patch 이후에 commit 되어야 하고, 왜 memo는 dependency 비교가 중요한지 깊게 생각하지 못했다.
- **해결:** 학습용 문서를 읽으며 `useEffect`는 렌더 중 실행하는 것이 아니라 "이번 렌더에서 실행해야 할 effect를 기록해 두었다가 DOM 반영 후 commit 단계에서 처리한다"는 흐름을 이해했다. `useMemo` 역시 단순 캐시가 아니라, 파생 계산이 불필요하게 반복되지 않도록 의존성 비교를 기준으로 값을 재사용하는 구조라는 점을 이해했다. Hook을 API가 아니라 렌더 파이프라인의 일부로 바라보게 되었다.

### 상태 기반 렌더링과 고빈도 시각 효과를 분리해야 한다는 점
- **문제:** 시연 앱을 만들기 전에는 카드 틸트나 glare 같은 시각 효과도 모두 상태로 관리하는 것이 React같은 렌더러를 사용하는 방식이라고 막연히 생각했다. 하지만 마우스 이동처럼 초당 여러 번 발생하는 값을 루트 상태에 넣으면 전체 렌더링 비용이 커질 수 있다는 점을 충분히 체감하지 못했다.
- **해결:** GPT와의 대화를 통해 데이터 변화와 고빈도 DOM 효과를 분리하는 것이 성능상 유리하다는 것을 알게 되었으며, 모든 UI 변화를 상태에 넣는 것이 정답은 아니라는 점을 배웠다. 페이지 전환, 검색, 필터, 즐겨찾기처럼 구조와 데이터가 바뀌는 부분은 루트 상태로 관리하고, 카드 기울기처럼 즉시성과 빈도가 중요한 효과는 DOM 스타일 보조 효과로 처리하는 편이 더 적절하다는 점을 이해했다.

### AI 활용에서 "빠른 생성"과 "설명 가능성"을 함께 챙겨야 하는 점
- **문제:** 구현이 우선이었기 때문에 AI를 적극 활용하여 많은 코드를 생성하였지만, 결국 그만큼 생성한 코드를 모두 받아들일 수는 없었다.
- **해결:** 코드 전체를 모두 읽기보다 `useState` 가 사용되는 방식, `setState` 이후 update 예약, effect cleanup 시점, VDOM과 patch 연결 같은 핵심 로직을 중심으로 코드를 살펴보며 학습하였다.


## 4. 새로 배운 점
- DP는 결국 "무엇을 저장하면 다음 판단이 쉬워지는가"를 설계하는 과정이라는 점을 배웠다. 이번 주에 푼 `피보나치`, `계단 오르기`, `동전`, `LCS`, `배낭`, `Word Break`, `Triangle`은 형태는 달라도 상태 설계와 이전 결과 재사용이라는 공통 원리를 공유하고 있었다.
- Greedy는 빨리 푸는 알고리즘이 아니라, 특정 선택 기준이 항상 최적이라는 것을 설명할 수 있을 때만 성립하는 방식이라는 점을 배웠다. 구현보다 선택의 근거를 세우는 과정이 더 중요했다.
- 문제를 풀고 이후 과정을 팀원들에게 설명하는 환경이 내가 어떤 부분을 정확히 이해하지 못했는지 드러내 준다는 점을 더 분명히 느꼈다. 제대로 설명할 수 없는 부분은 결국 다시 공부해야 할 부분이라는 사실을 실감했다.
- 리액트 같은 렌더러에서 상태는 함수 안에 저장되는 것이 아니라, 루트 `FunctionComponent`가 가진 `hooks` 배열과 dispatcher, cursor를 통해 유지된다는 점을 이해했다. Hook 호출 순서가 중요한 이유도 이 구조와 연결되어 있었다.
- `setState`는 값을 바꾸는 함수가 아니라, 상태 변경 후 "새 VDOM 계산 → diff → patch → effect commit" 흐름을 다시 돌리게 만드는 출발점이라는 점을 배웠다. 즉, 상태 변경과 렌더 예약은 분리해서 생각해야 했다.
- `useEffect`는 단순한 부가 기능이 아니라, DOM이 실제로 반영된 뒤 실행되어야 하는 후처리 단계라는 점을 이해했다. cleanup이 다음 effect 전에 실행되거나 unmount 시 정리된다는 점도 함께 연결해서 볼 수 있게 되었다.
- `useMemo`는 최적화 장치가 아니라, 필터/정렬/집계 같은 파생 계산을 의존성 기준으로 재사용하기 위한 구조라는 점을 배웠다. 값 자체보다 "언제 다시 계산해야 하는가"가 더 중요했다.
- `VNode -> Resolver -> Diff -> Patch` 흐름을 별도 계층으로 분리하면, 선언형 UI를 실제 DOM 조작과 분리해서 생각할 수 있다는 점이 인상적이었다. 특히 resolver가 함수형 자식 컴포넌트를 실제 DOM 태그 VNode 트리로 전개한다는 개념이 이번 주에 새롭게 정리되었다.
- `key`는 리스트 아이템의 정체성을 유지해 patch가 더 안정적으로 동작하도록 만드는 핵심 단서라는 점을 배웠다.
- 하나의 루트 상태로 1개의 페이지를 여러 페이지처럼 보이는 SPA를 만들 수 있다는 것도 배웠다. `currentPage`를 바꾸는 방식으로 `dashboard`, `collection`, `detail`, `settings`를 전환하는 구조를 보며, "페이지 수"보다 "상태 전환"이 더 본질적인 관점일 수 있다는 점을 느꼈다.
- 상태 기반 렌더링과 고빈도 DOM 효과를 분리하는 판단이 프론트엔드 성능에서 중요하다는 점을 배웠다. 모든 것을 상태에 넣기보다, 렌더링 구조 변화와 즉각적인 시각 효과를 다르게 다뤄야 한다는 감각이 생겼다.


## 5. 다음 주 계획
- C 언어 문법에 익숙하지 않은 부분은 짧게 보완하되, 문법 자체를 오래 파기보다 과제를 진행하면서 `pointer`, `malloc/free`, Linux/gcc 사용법을 실제 코드와 함께 익히는 방향으로 학습하기
- Data Structures 과제에서 최소 `Linked List`까지는 직접 구현해야 하므로, 포인터와 동적 메모리 할당이 자료구조 구현에서 어떻게 쓰이는지 팀원들과 함께 정리해 보기
- 수요 코딩회 과제인 SQL 처리기를 대비해 `입력(SQL) -> 파싱 -> 실행 -> 파일 저장` 흐름을 먼저 머릿속에 그려 보고, `INSERT`, `SELECT`, CLI 입력 처리, 파일 기반 저장 포맷 설계를 설명할 수 있도록 준비하기
- 이번 주 C 과제는 별도 테스트케이스가 없으므로, 직접 입력 케이스를 만들어 결과를 검증하는 습관을 더 의식적으로 가져가기
