# Redis 실제 활용 시나리오

## 1. 왜 Redis를 실무에서 자주 사용하는가

Redis는 매우 빠른 응답 속도와 TTL 지원 덕분에 "짧은 시간 동안 자주 읽히는 데이터"를 다루는 문제에 특히 강하다. 실무에서는 모든 데이터를 Redis에 저장하기보다, 빠른 접근이 필요한 일부 데이터를 Redis에 두고 원본 데이터는 다른 저장소에서 관리하는 형태가 많다.

## 2. 캐시

가장 대표적인 활용 사례는 캐시이다.

예를 들어 상품 목록, 인기 게시글, 사용자 프로필처럼 자주 조회되지만 매번 데이터베이스에서 읽어오면 부담이 큰 데이터를 Redis에 저장할 수 있다.

동작 흐름은 보통 다음과 같다.

1. 클라이언트가 데이터를 요청한다.
2. 애플리케이션이 먼저 Redis에서 값을 찾는다.
3. 값이 있으면 즉시 반환한다.
4. 값이 없으면 데이터베이스에서 조회한 뒤 Redis에 저장하고 반환한다.

이 방식은 데이터베이스 부하를 줄이고 응답 속도를 높이는 데 효과적이다.

### Redis가 캐시에 적합한 이유

- 조회 속도가 빠르다.
- TTL 설정으로 자동 만료가 가능하다.
- 키 기반 접근이 단순하다.

## 3. 세션 저장소

로그인한 사용자의 세션 정보를 Redis에 저장하는 방식도 널리 사용된다.

웹 서비스에서 사용자의 로그인 상태, 권한, 마지막 활동 시간 같은 정보를 Redis에 저장하면 여러 서버가 같은 세션 정보를 공유하기 쉬워진다. 또한 세션 만료 시간을 TTL로 자연스럽게 표현할 수 있다.

### 예시

- 키: `session:user:123`
- 값: 사용자 세션 정보
- TTL: 30분

이 구조는 사용자가 일정 시간 활동하지 않으면 세션이 자동 만료되도록 만들기 좋다.

## 4. 인증 코드 및 일회성 토큰 저장

이메일 인증 코드, 비밀번호 재설정 토큰, 휴대폰 인증 번호처럼 짧은 시간만 유효한 데이터도 Redis에 잘 맞는다.

이런 데이터는 다음 조건을 가진다.

- 빠르게 저장되어야 한다.
- 빠르게 조회되어야 한다.
- 시간이 지나면 자동 삭제되어야 한다.

TTL이 있는 Redis는 이런 요구사항과 잘 맞는다.

## 5. 카운터

조회 수, 좋아요 수, 이벤트 참여 수처럼 숫자를 빠르게 증가시켜야 하는 경우에도 Redis가 많이 사용된다.

예를 들어 게시글 조회 수를 매 요청마다 데이터베이스에 기록하면 쓰기 부하가 커질 수 있다. 이때 Redis에서 먼저 카운트를 누적한 뒤, 일정 주기로 데이터베이스에 반영하는 방법을 사용할 수 있다.

### 예시

- 키: `post:100:view-count`
- 값: 조회 수
- 사용 명령: 증가 연산

## 6. 랭킹 시스템

게임 점수판, 인기 검색어, 실시간 랭킹처럼 정렬된 순위를 유지해야 하는 경우 Redis의 Sorted Set이 유용하다.

Sorted Set은 각 항목에 점수를 부여하고 점수 순으로 정렬된 상태를 유지할 수 있다. 따라서 상위 N개 조회 같은 기능을 빠르게 처리할 수 있다.

### 예시

- 사용자별 게임 점수 저장
- 점수 기준 상위 10명 조회
- 특정 사용자의 현재 순위 조회

## 7. 작업 큐

비동기 작업을 처리할 때 Redis를 큐처럼 사용할 수 있다. 예를 들어 이메일 발송, 이미지 처리, 알림 전송 같은 작업을 요청 즉시 처리하지 않고 큐에 넣은 뒤, 별도 작업자가 순서대로 소비하는 구조를 만들 수 있다.

학습 목적에서는 List 구조를 이용해 간단한 작업 큐를 이해할 수 있다.

## 8. Pub/Sub와 실시간 메시징

Redis는 발행/구독(Pub/Sub) 기능도 제공한다. 채팅, 실시간 알림, 이벤트 전달 같은 상황에서 간단한 메시지 전달 시스템으로 사용할 수 있다.

다만 메시지 내구성, 재시도, 대규모 소비자 그룹 관리가 중요하다면 더 전문적인 메시지 브로커가 적합할 수 있다. 따라서 Redis Pub/Sub는 "간단하고 빠른 전달"이 중요한 상황에 잘 맞는다.

## 9. 분산 락

여러 서버가 동시에 같은 작업을 실행하면 안 되는 경우, Redis를 이용해 간단한 분산 락을 구현하기도 한다.

예를 들어 다음과 같은 문제를 떠올릴 수 있다.

- 같은 배치 작업이 여러 서버에서 중복 실행되는 문제
- 재고 차감 로직이 동시에 실행되는 문제

이 경우 특정 키를 먼저 점유한 서버만 작업을 진행하게 만드는 방식으로 제어할 수 있다.

## 10. 이번 과제와 연결해서 볼 포인트

이번 미니 Redis 과제에서는 Redis의 모든 실무 기능을 구현할 필요는 없지만, 아래 시나리오는 학습적으로 특히 중요하다.

- 캐시: 왜 빠른 조회가 필요한가
- 세션 저장: 왜 TTL이 중요한가
- 카운터: 왜 키-값 저장이 단순하고 강력한가

즉, 미니 Redis 구현은 단순한 자료구조 실습이 아니라 실제 서비스에서 Redis가 어떤 문제를 해결하는지 이해하는 과정으로 볼 수 있다.
