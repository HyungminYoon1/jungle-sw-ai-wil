# 미니 Redis 구현 설계 초안

## 1. 설계 목표

이번 미니 Redis의 목표는 Redis의 모든 기능을 재현하는 것이 아니라, 핵심 개념을 학습할 수 있는 최소 기능을 직접 구현하는 것이다.

핵심 학습 목표는 다음과 같다.

- 키-값 저장소의 기본 동작 이해
- 빠른 조회를 위한 자료구조 선택 이해
- TTL 기반 만료 처리 이해
- 명령 기반 인터페이스 설계 경험

## 2. 구현 범위

학습용 최소 구현 범위는 아래와 같이 잡는 것이 적절하다.

### 필수 기능

- 문자열 키-값 저장
- `SET key value`
- `GET key`
- `DEL key`
- `EXPIRE key seconds`
- `TTL key`

### 선택 기능

- `SET key value EX seconds`
- `EXISTS key`
- `KEYS pattern`

처음에는 필수 기능 위주로 구현하고, 이후 여유가 있으면 선택 기능을 추가하는 방식이 좋다.

## 3. 내부 자료구조 설계

가장 단순한 형태는 해시 테이블 두 개를 사용하는 방식이다.

### 3-1. 데이터 저장소

- 역할: 실제 값을 저장
- 구조 예시: `Map<String, String>`

키를 통해 값을 빠르게 조회하기 위한 기본 저장소이다.

### 3-2. 만료 시간 저장소

- 역할: 각 키의 만료 시각 저장
- 구조 예시: `Map<String, Long>`

각 키가 언제 만료되는지를 기록한다. 값은 Unix time 기반의 만료 시각 또는 현재 시각 기준 만료 시점으로 저장할 수 있다.

이 구조를 분리하면 값 저장과 TTL 관리를 단순하게 나눌 수 있다.

## 4. 명령 처리 흐름

사용자가 입력한 문자열 명령을 파싱하고, 명령 종류에 따라 적절한 저장소 연산을 수행하는 구조로 설계할 수 있다.

### 공통 흐름

1. 명령 문자열을 입력받는다.
2. 공백 기준으로 토큰을 분리한다.
3. 첫 토큰으로 명령 종류를 판별한다.
4. 명령별 유효성 검사를 수행한다.
5. 저장소에 대한 읽기/쓰기 연산을 수행한다.
6. 결과 문자열을 반환한다.

예를 들어 `GET key1`이 들어오면 먼저 `key1`이 만료되었는지 확인하고, 만료되지 않았다면 값을 반환하는 흐름으로 처리할 수 있다.

## 5. TTL 처리 방식

TTL은 미니 Redis에서 가장 중요한 학습 포인트 중 하나이다.

### 5-1. 지연 삭제 방식

가장 단순한 방법은 키에 접근할 때마다 만료 여부를 검사하는 방식이다.

예를 들어 `GET`, `TTL`, `EXISTS` 같은 명령이 들어오면 다음 순서로 처리한다.

1. 해당 키가 만료 시간 저장소에 있는지 확인한다.
2. 현재 시각이 만료 시각을 지났는지 확인한다.
3. 만료되었다면 데이터 저장소와 만료 시간 저장소에서 모두 삭제한다.
4. 만료되지 않았다면 정상 처리한다.

이 방법은 구현이 단순하고 학습용 프로젝트에 적합하다.

### 5-2. 주기적 정리 방식

추가 확장으로는 일정 주기마다 만료 키를 정리하는 백그라운드 작업을 둘 수 있다. 하지만 학습 초기에는 동시성, 스케줄링, 스레드 관리까지 복잡도가 증가하므로 우선순위는 낮다.

## 6. 명령별 동작 설계

### SET

- 키와 값을 저장한다.
- 기존 값이 있으면 덮어쓴다.
- 기존 TTL이 있었다면 정책을 명확히 정해야 한다.

학습용 구현에서는 `SET` 수행 시 기존 TTL을 제거하는 방식이 단순하다.

### GET

- 먼저 만료 여부를 검사한다.
- 키가 없거나 만료되었으면 null 또는 nil 성격의 응답을 반환한다.
- 키가 있으면 값을 반환한다.

### DEL

- 데이터 저장소에서 키를 삭제한다.
- 만료 시간 저장소에서도 함께 삭제한다.
- 삭제 성공 여부를 반환할 수 있다.

### EXPIRE

- 키가 존재할 때만 만료 시간을 설정한다.
- 현재 시각 + 초 단위 값을 이용해 만료 시각을 계산한다.

### TTL

- 키가 없으면 특정 실패 값을 반환한다.
- 키는 있지만 만료 시간이 없으면 별도 값을 반환한다.
- 키에 TTL이 있으면 남은 시간을 초 단위로 계산해 반환한다.

## 7. 예외 및 경계 조건

구현 시 아래 상황을 명확하게 다루는 것이 중요하다.

- 존재하지 않는 키에 `GET` 요청
- 존재하지 않는 키에 `EXPIRE` 요청
- TTL이 없는 키에 `TTL` 요청
- 만료된 키에 대한 접근
- 인자 개수가 잘못된 명령 입력
- 음수 또는 0초 TTL 처리 정책

이런 경계 조건을 명확히 정리하면 테스트도 쉬워진다.

## 8. 테스트 관점

미니 Redis는 명령 수가 적기 때문에 단위 테스트 중심으로 검증하기 좋다.

핵심 테스트 항목은 다음과 같다.

- `SET` 후 `GET`이 올바른 값 반환
- `DEL` 후 값이 사라지는지 확인
- `EXPIRE` 설정 후 TTL이 감소하는지 확인
- 만료 시간이 지난 뒤 `GET` 시 값이 제거되는지 확인
- 만료된 키가 내부 저장소에서도 정리되는지 확인
- 잘못된 명령 형식에 대해 적절한 응답 반환

## 9. 단계별 구현 전략

복잡도를 낮추기 위해 아래 순서로 구현하는 것이 좋다.

1. 키-값 저장소만 구현한다.
2. `SET`, `GET`, `DEL`을 완성한다.
3. TTL 저장소를 추가한다.
4. `EXPIRE`, `TTL`을 구현한다.
5. 접근 시 만료 검사 로직을 공통화한다.
6. 단위 테스트를 보강한다.

이 순서를 따르면 한 번에 많은 기능을 넣지 않고 점진적으로 완성할 수 있다.

## 10. 결론

미니 Redis의 핵심은 복잡한 분산 시스템을 재현하는 것이 아니라, Redis의 본질적인 아이디어를 직접 구현해보는 데 있다.

즉, 아래 세 가지를 명확히 배우는 것이 중요하다.

- 해시 기반 키-값 저장
- 빠른 조회를 위한 단순한 명령 처리
- TTL을 통한 데이터 수명 관리

이 범위로 설계를 시작하면 구현 난도를 적절하게 유지하면서도 Redis의 핵심 개념을 충분히 학습할 수 있다.
