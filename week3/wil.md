## 2. 시도한 접근 방식
- 알고리즘 문제는 문제 풀이부터 바로 시작하지 않고, 동료학습을 통해 먼저 기본 문제의 유형별 핵심 개념을 학습하는 과정을 가졌다.
- 알고리즘 문제 풀이 시 입력/출력 조건을 먼저 정리하고, 한국어로 알고리즘 구현 과정을 작성한 뒤 풀이를 진행했다.
- 이분탐색과 분할정복 계열 문제는 시간복잡도를 먼저 계산하고, 왜 선형 탐색이 아니라 로그 탐색이 필요한지부터 생각한 뒤 코드로 옮겼다.
- 스택, 큐, 우선순위 큐, 링크드리스트, 해시 테이블은 자료구조의 연산 특성과 사용 시점을 중심으로 학습하고 문제에 적용했다.
- 미니 Redis는 구현에 앞서 요구사항, 아키텍처, API 명세 문서를 먼저 작성하고, 계층 분리와 TTL 처리 위치를 기준으로 설계를 이해하려고 했다.
- 팀 문서에서는 요구사항 정의서, 아키텍처 문서, API 명세서를 함께 보며 기능과 책임이 어떻게 연결되는지 확인했다.
- 미니 Redis 구현 후에는 팀원의 의견을 수렴하여, Redis 사용 전후의 차이를 시각적으로 보여줄 수 있는 간단한 벤치마크 앱도 추가로 개발하였다.


## 3. 문제와 해결 과정

### 알고리즘 문제에서 개념과 구현이 분리되어 있지 않았던 점
- **문제:** 초반에 개념을 공부하지 않고 바로 문제 풀이부터 접근하는 습관이 있어, 머지 정렬에 대한 기본 문제를 풀 때까지 오랜 시간동안 고민하며 어려움을 겪었다.
- **해결:** 머지 정렬이 무엇이며, 어떻게 구현되는가를 웹에서 찾아 공부하고 다시 문제로 돌아가 구현 과정을 정리했다. 이 과정을 거치니 머지 정렬 기본문제도 곧 풀 수 있었다.

### 미니 Redis에서 시스템에 대한 깊은 이해의 중요성
- **문제:** Redis 과제 구현 시 개념 학습은 했지만, 실제 시스템을 어떤 구조로 나눠 구현할지에 대한 이해는 충분하지 않았다.
- **해결:** 요구사항 정의서, 시스템 아키텍처 설계 문서, API 명세서를 먼저 작성하고 그 문서를 기준으로 현재 구현 범위와 계층 책임을 정리했다. 명령 파싱, 검증, 서비스, 저장소, 만료 처리 계층이 분리된 구조를 먼저 문서로 이해한 뒤 구현을 진행하니 전체 흐름을 따라가기가 수월했다.  
- **과제:** 다만 문서를 기반으로 구조를 잡는 것과, 각 계층의 세부 구현을 끝까지 설명 가능한 수준으로 이해하는 것은 다른 문제라는 점을 느꼈다. 앞으로는 AI의 도움을 받더라도 내가 직접 설명할 수 있는 수준까지 구조와 동작 원리를 더 깊게 정리해야겠다.

### TTL과 만료 처리 방식 이해
- **문제:** 처음에는 TTL을 단순히 "시간이 지나면 값이 사라진다" 정도로만 생각했는데, 실제로는 언제 만료를 검사하고 어떤 계층이 책임질지 정하는 것이 중요했다.
- **해결:** 아키텍처 문서를 통해 지연 삭제와 백그라운드 주기적 정리를 함께 사용하는 이유를 이해했다. `GET`, `DEL`, `EXPIRE`, `TTL` 실행 전에 만료 검사를 수행해야 한다는 점과, 만료된 키를 값 저장소와 TTL 저장소에서 함께 제거해야 한다는 점이 중요하다는 것을 느꼈다. 추가적으로 Redis를 사용하는 것이 Redis를 사용하지 않는 것보다 반드시 좋은 것만은 아님을 알게 되었고, 데이터의 종류에 따라 어느 주기로 메모리를 청소할지에 대한 정책을 잘 결정하는 것도 중요하다는 것을 느꼈다.

### 실제 활용 시나리오를 README에 녹이는 일의 어려움
- **문제:** Redis의 캐시, 세션, 카운터, 랭킹 같은 실제 활용 시나리오는 따로 정리했지만, 이를 README 수준의 짧고 명확한 설명으로 연결하는 작업은 충분히 하지 못했다.
- **해결/개선:** 기술 문서와 별개로, "이 프로젝트가 실제로 어떤 문제를 해결하는 구조인지"를 README 첫 부분에 한두 문장으로 설명하는 습관이 필요하다는 점을 배웠다. 다음에는 설계 문서뿐 아니라 소개 문서에서도 비즈니스 관점을 함께 드러내고 싶다.

### AI 활용의 한계와 가능성
- **문제:** AI를 학습 보조 도구로 사용하는 원칙은 세웠지만, 실제 발표나 협업 과정에서 AI가 제안한 코드/설계의 원리를 충분히 이해하지 못하였고, 발표 간 설명이 어려워 아쉬움이 남았다.
- **해결/개선:** AI에게 답을 받는 것보다 "왜 이렇게 설계했는지", "다른 선택지는 무엇인지"를 다시 질문하며 검증하는 방식이 더 중요하다는 점을 느꼈다. 다음부터는 AI가 만든 결과를 그대로 이용하기보다, 내가 이해하고 설명할 수 있는 범위에서 리뷰와 비교, 보완의 도구로 활용해야겠다는 교훈을 얻었다.


## 4. 새로 배운 점
- 이분탐색은 단순히 정렬된 배열에서 값을 찾는 기법이 아니라, 조건을 만족하는 경계값을 찾는 사고방식으로도 활용할 수 있다는 점을 배웠다.
- 분할정복은 문제를 나누는 기준이 명확해야 하며, 퀵 정렬과 머지 정렬은 같은 O(n log n)의 시간 복잡도를 갖는 정렬이라도 분할 방식과 병합 방식이 다르다는 점을 비교하며 이해할 수 있었다. 또한 퀵 정렬은 피벗을 어떤 것으로 지정하느냐에 따라 성능이 달라질 수 있다는 것을 배웠다.
- 스택과 큐는 구현 자체보다 어떤 데이터에 어떤 자료구조를 모델링하는지 이해하는 것이 중요했다. 우선순위 큐는 먼저 들어온 순서가 아니라 우선순위 기준으로 처리된다는 점이 핵심이었다.
- 연결 리스트와 해시 테이블은 직접 다루어 보면서, 삽입/삭제 효율과 조회 효율이 자료구조 선택에 큰 영향을 준다는 점을 다시 체감했다.
- 미니 Redis 개발을 통해 네트워크 처리, 명령 파싱, 비즈니스 로직, 저장소, TTL 관리가 분리되어야 유지보수가 쉬워진다는 점을 구조적으로 이해했다. 또한 AI를 이용할 때 처음부터 완전 구현을 요청하는 것이 아니라, 개발자가 이해 및 감독이 가능한 범위에서 모듈 단위로 사용해야겠다는 교훈을 얻었다.
- RESP3, HELLO 3, 세션 처리, 지속 연결 지원 같은 요소를 보면서 "로컬에서 동작하는 코드"와 "프로토콜을 가진 서버 프로그램"의 설계 관점이 다르다는 점을 배웠다.
- 테스트 코드는 단순히 시스템이 동작하는지 여부를 확인하는 수단이 아니라, 구현한 시스템이 요구사항을 문서대로 만족하고 있는지 검증하는 기준이라는 점을 더 분명히 느꼈다.
