# 1. 먼저 큰 그림: 이번 주에 실제로 만드는 것은 무엇인가

이번 주 수요 코딩회의 결과물은 `내부 DB 엔진을 가진 API 서버`다.

구조를 두 층으로 나누면 이해가 쉽다.

- 바깥: 외부 요청을 받고 응답을 주는 `API 서버`
- 안쪽: SQL을 실행하고 데이터를 읽고 쓰는 `내부 DB 엔진`

7주차에 만든 SQL 처리기와 B+ 트리 인덱스는 이 중 `안쪽 엔진`에 해당한다.
8주차에는 그 엔진을 네트워크 요청 뒤에 놓고, 여러 요청을 처리할 수 있는 서버 구조로 감싸는 것이 핵심이다.

한 문장으로 줄이면 아래와 같다.

`이번 주는 7주차 SQL 처리기를 네트워크 뒤에 올리고, 여러 요청을 안전하게 받을 수 있는 API 서버 구조로 감싸는 주차다.`

# 2. 공지의 `그대로 활용`은 무엇을 뜻하는가

공지의 `그대로 활용`을 곧바로 `7주차 코드를 수정 없이 그대로 여러 스레드에서 호출한다`로 이해하면 위험하다.

7주차 SQL 처리기의 강점은 아래와 같다.

- SQL 파싱 흐름이 이미 있다.
- `INSERT`, `SELECT`, `WHERE id = ...` 흐름이 있다.
- CSV 저장과 B+ 트리 인덱스 접근 구조가 있다.
- 계층 분리가 어느 정도 잡혀 있다.

즉 `내부 DB 엔진의 코어`로는 충분히 재사용할 수 있다.

하지만 7주차 엔진에는 아래 한계도 있다.

- 동시성 제어는 원래 제외 범위였다.
- 전역 registry 같은 공유 상태가 있다.
- CSV append와 index update는 멀티스레드 안전성을 전제로 설계되지 않았다.
- MVCC, snapshot, transaction 같은 고급 일관성 모델이 없다.

따라서 `그대로 활용`의 정확한 의미는 아래에 가깝다.

- 기존 파서, 실행기, 저장소, 인덱스 로직은 재사용한다.
- API 서버와 엔진 사이에 adapter를 둔다.
- 동시성 보호는 이번 주 서버 구조에서 새로 설계한다.

즉 `재사용`은 맞지만 `무수정 그대로 멀티스레드 호출`은 아니다.

# 3. 7주차 SQL 처리기를 그대로 붙여도 괜찮은가

결론부터 말하면 아래가 맞다.

`내부 DB 엔진의 기반으로 쓰는 것은 맞지만, 멀티스레드 API 서버에 아무 보호 없이 그대로 붙이는 것은 위험하다.`

왜냐하면 이번 주 공지는 `thread pool`과 `SQL 요청 병렬 처리`를 요구하고 있기 때문이다.
반면 7주차 엔진은 그런 환경을 직접 전제로 설계된 것은 아니다.

그래서 이번 주의 현실적인 목표는 아래처럼 잡는 것이 맞다.

- 엔진은 최대한 재사용한다.
- 엔진이 원래 보장하지 않던 동시성은 서버/adapter 경계에서 보수적으로 보호한다.
- API 서버와 DB 엔진의 역할을 분리한다.
- 지원하는 SQL 범위는 엔진이 안정적으로 처리 가능한 수준으로 제한한다.

# 4. API 서버와 내부 DB 엔진은 무엇이 다른가

## 4.1 API 서버의 역할

API 서버는 바깥 세상과 대화하는 층이다.

주요 책임:

- 소켓을 열고 요청을 받는다.
- HTTP 요청을 읽고 파싱한다.
- 메서드, 경로, JSON body를 검증한다.
- 유효하지 않은 요청은 엔진까지 보내지 않고 즉시 에러 응답을 만든다.
- 내부 DB 엔진을 호출한다.
- 결과를 HTTP 응답으로 변환한다.

API 서버가 직접 하지 말아야 할 일:

- SQL 파서 구현
- CSV 파일 직접 조작
- B+ 트리 직접 탐색
- 데이터 저장 정책 결정

## 4.2 내부 DB 엔진의 역할

내부 DB 엔진은 데이터를 실제로 다루는 층이다.

주요 책임:

- SQL 파싱
- 실행 경로 분기
- 저장소 접근
- 인덱스 사용
- 결과 생성

DB 엔진이 직접 하지 말아야 할 일:

- HTTP 파싱
- 소켓 관리
- worker thread 관리
- HTTP 상태코드 결정

## 4.3 왜 이 둘을 분리해야 하는가

이 경계를 분리하지 않으면 아래 문제가 생긴다.

- HTTP 코드와 DB 코드가 섞인다.
- API 형식이 바뀌면 엔진까지 흔들린다.
- 동시성 문제가 어디서 났는지 찾기 어렵다.
- 테스트가 어려워진다.

그래서 구조는 아래처럼 보는 것이 가장 자연스럽다.

```text
client
  -> API server
  -> request parser
  -> handler
  -> DB service / adapter
  -> week7 SQL engine
  -> response formatter
  -> client
```

# 5. thread pool은 무엇인가

thread pool은 `요청이 올 때마다 새 thread를 만드는 대신, 미리 준비된 worker thread들이 작업 큐에서 요청을 가져가 처리하는 구조`다.

## 5.1 왜 필요한가

요청마다 thread를 새로 만들면 아래 문제가 생긴다.

- thread 생성/종료 비용이 누적된다.
- 요청이 많아질수록 문맥 전환 비용이 커진다.
- 자원 관리가 어렵다.

thread pool은 이를 줄이기 위한 구조다.

- worker thread를 미리 만든다.
- 요청은 작업 큐에 넣는다.
- 쉬고 있던 worker가 큐에서 요청을 꺼내 처리한다.

## 5.2 이번 과제에서 왜 중요한가

이번 주 공지는 여러 요청을 동시에 받아 SQL 요청을 병렬로 처리하라고 말한다.
즉 단순한 단일 요청 서버가 아니라 `여러 요청을 동시에 수용하는 구조`가 필요하다.

다만 여기서 꼭 구분해야 하는 것이 있다.

- `서버 수준 병렬성`: 여러 요청을 동시에 받는 것
- `엔진 수준 병렬성`: 여러 요청이 내부 DB 상태를 동시에 다루는 것

이번 주에서 반드시 필요한 것은 첫 번째다.
두 번째는 엔진 특성을 고려해 안전하게 설계해야 한다.

# 6. 작업 큐와 worker는 어떻게 동작하는가

thread pool은 보통 아래 흐름으로 이해하면 된다.

1. 서버 시작 시 worker thread N개를 만든다.
2. main thread는 소켓에서 연결을 받는다.
3. 받은 요청 또는 연결 정보를 작업 큐에 넣는다.
4. worker thread가 큐에서 작업을 꺼낸다.
5. worker가 요청을 읽고 파싱한다.
6. 내부 DB 엔진을 호출한다.
7. 응답을 보내고 연결을 닫는다.

```text
accept loop
  -> task queue push(connfd or request)

worker thread 1
worker thread 2
worker thread 3
  -> task queue pop()
  -> handle request
  -> call DB engine
  -> send response
```

핵심은 `accept`와 `실제 처리`를 분리하는 것이다.

# 7. 이번 주에 꼭 알아야 하는 공유 상태

문제는 여러 요청을 동시에 받는 것 자체보다, 여러 worker가 같은 상태를 동시에 건드릴 때 생긴다.

이번 과제에서 공유 상태가 될 수 있는 것은 아래와 같다.

- 전역 인덱스 registry
- 테이블별 B+ 트리
- CSV 파일
- 테이블별 `next_id`
- 캐시나 스키마 메타정보

이 공유 상태를 보호하지 않으면 아래 문제가 생길 수 있다.

- 같은 id 중복 생성
- 파일 기록과 인덱스 갱신 불일치
- rebuild 중 중간 상태 노출
- race condition으로 인한 잘못된 조회

# 8. 동시성 보호는 왜 필요한가

가장 단순한 예시는 동시에 두 개의 `INSERT`가 들어오는 경우다.

- 두 worker가 동시에 같은 `next_id`를 읽는다.
- 둘 다 같은 값을 새 id로 선택한다.
- 둘 다 CSV에 기록을 시도한다.
- 인덱스 갱신 순서가 꼬일 수 있다.

또는 한 worker가 인덱스를 rebuild하는 동안 다른 worker가 `SELECT`를 하면 아래 문제가 생길 수 있다.

- 아직 완성되지 않은 인덱스를 읽는다.
- registry가 중간 상태를 가진다.
- 일부만 반영된 결과를 읽는다.

즉 `thread pool이 있다`는 사실만으로 엔진이 자동으로 안전해지지는 않는다.

# 9. 어떤 락 전략이 가장 현실적인가

이번 주 현실적인 선택지는 보통 세 가지다.

## 9.1 전역 mutex

엔진 전체를 하나의 큰 lock으로 감싼다.

장점:

- 가장 단순하다.
- 디버깅이 쉽다.

단점:

- 한 테이블의 긴 작업이 모든 테이블 요청을 막는다.
- 공지의 `병렬 SQL 처리`와는 거리가 있다.

따라서 전역 mutex는 fallback 또는 디버깅용으로는 의미가 있지만, 이번 주 기본 전략으로는 아쉽다.

## 9.2 테이블 단위 lock

테이블마다 별도의 lock을 둔다.

장점:

- 같은 테이블은 보호하면서 다른 테이블은 병렬 처리할 수 있다.
- 7주차 엔진을 크게 뜯지 않고도 부분적인 엔진 병렬성을 설명할 수 있다.
- 이번 과제 요구사항과 엔진 현실 사이의 타협안으로 가장 적절하다.

단점:

- 대상 테이블을 해석해야 한다.
- lock manager가 필요하다.
- 같은 테이블의 긴 write는 여전히 병목이다.

## 9.3 같은 테이블 안에서의 read/write lock

같은 테이블 안에서도 read-read는 병렬, write는 배타적으로 처리하는 방식이다.

이론적으로는 더 효율적일 수 있다.
하지만 지금 엔진은 snapshot이나 MVCC가 없기 때문에 처음부터 여기까지 들어가면 설계가 빠르게 어려워진다.

그래서 이번 주 추천 순서는 아래와 같다.

1. `테이블 단위 lock`
2. 필요하면 이후 단계에서 같은 테이블 안의 read/write 분리 검토

# 10. 왜 `테이블 단위 lock`이 최선의 추천인가

이번 주 엔진은 고급 DB처럼 동작하지 않는다.

- transaction 없음
- MVCC 없음
- snapshot 없음
- commit/rollback 없음

이 상태에서 가장 설명 가능하고 구현 가능한 병렬성은 아래다.

- `같은 테이블`: 충돌 가능한 요청에서 앞선 write를 뒤 read/write가 앞지르지 않도록 보수적으로 잠근다.
- `다른 테이블`: 병렬로 처리한다.

즉 `모든 SQL을 동시에 실행한다`가 아니라 `안전하게 분리 가능한 범위까지 병렬로 처리한다`가 이번 주의 현실적인 목표다.
이 설명은 현재 지원 범위가 사실상 single-table query 중심이라는 전제에서 가장 자연스럽게 성립한다.

# 11. 같은 클라이언트와 다른 클라이언트의 읽기/쓰기 순서는 어떻게 봐야 하는가

## 11.1 같은 클라이언트의 경우

예를 들어 같은 클라이언트가 아래 요청을 순서대로 보냈다고 하자.

1. `INSERT`
2. `SELECT`

이 경우 사용자는 당연히 두 번째 `SELECT`에서 방금 넣은 결과를 기대한다.
따라서 앞선 `INSERT`보다 뒤의 `SELECT`가 먼저 지나가면 안 된다.

즉 같은 클라이언트의 순차 요청에서는 최소한 `read-after-write` 기대를 깨지 않는 설계가 필요하다.
다만 이 보장은 테이블 lock만 있다고 자동으로 생기는 것은 아니고, 같은 클라이언트가 앞선 요청의 완료를 기준으로 다음 요청을 보내거나, 서버가 요청 순서를 무시하지 않는 흐름이 함께 맞아야 자연스럽게 성립한다.

## 11.2 다른 클라이언트의 경우

다른 클라이언트의 `SELECT`는 반드시 즉시 나가야 하는 것은 아니다.

현재 엔진 수준에서는 아래 선택지 중 하나를 가져야 한다.

- 앞선 write가 끝날 때까지 기다린다.
- snapshot 같은 고급 모델을 구현해 이전 상태를 읽게 한다.

하지만 지금 엔진에는 snapshot, MVCC가 없다.
따라서 같은 테이블에 대한 write가 진행 중일 때 다른 클라이언트의 read도 대기시키는 편이 더 안전하다.

이것은 불편해 보일 수 있지만, 중간 상태를 읽게 하는 것보다 낫다.

# 12. 대량 INSERT가 있으면 다른 클라이언트 SELECT도 막아야 하는가

질문을 두 경우로 나눠야 한다.

## 12.1 같은 테이블에 대한 요청

클라이언트 A가 `users` 테이블에 대량 `INSERT`를 수행하는 동안,
클라이언트 B가 같은 `users` 테이블에 `SELECT`를 보내면 현재 엔진 수준에서는 대기시키는 것이 더 안전하다.

이유:

- CSV와 인덱스가 중간 상태일 수 있다.
- snapshot이 없다.
- 일부만 반영된 결과를 보여주면 더 큰 혼란이 생긴다.

즉 같은 테이블에 대해서는 `즉시성`보다 `정합성`이 우선이다.

## 12.2 다른 테이블에 대한 요청

클라이언트 A가 `users` 테이블에 대량 `INSERT`를 하는 동안,
클라이언트 B가 `orders` 테이블에 `SELECT`를 보내는 경우라면 굳이 같이 막을 이유가 없다.

바로 여기서 `테이블 단위 lock`이 의미를 가진다.

- `users`는 기다림
- `orders`는 진행 가능

즉 불편을 최소화하는 현실적인 방법은 `같은 테이블은 보수적으로 대기`, `다른 테이블은 병렬 허용`이다.

# 13. 서버 경계와 동시성 경계는 무슨 뜻인가

## 13.1 서버 경계

`서버 경계`는 어디까지가 API 서버의 책임이고 어디부터가 DB 엔진의 책임인지 나누는 선이다.

예:

- API 서버: 요청 수신, HTTP 파싱, 검증, 응답 생성
- DB 엔진: SQL 실행, CSV 접근, 인덱스 조회

## 13.2 동시성 경계

`동시성 경계`는 어디까지는 여러 worker가 동시에 들어와도 되고, 어디서부터는 lock으로 보호해야 하는지 나누는 선이다.

예:

- 요청 읽기, 파싱, 검증: 병렬 가능
- 대상 테이블 결정 후 엔진 호출: 해당 테이블 lock 필요

즉 이 두 경계를 분리해서 생각해야 구조가 단순해진다.

# 14. 권장 요청 흐름

이번 주에 가장 설명하기 쉬운 요청 흐름은 아래다.

1. client가 HTTP 요청을 보낸다.
2. accept loop가 연결을 받는다.
3. 작업 큐에 요청 정보를 넣는다.
4. worker가 요청을 꺼낸다.
5. 요청을 파싱하고 검증한다.
6. 유효하지 않으면 엔진까지 가지 않고 즉시 에러 응답을 만든다.
7. 대상 테이블을 해석한다.
8. 해당 테이블 lock을 잡는다.
9. 7주차 SQL 엔진을 호출한다.
10. 결과를 로컬 버퍼나 구조체로 복사한다.
11. lock을 푼다.
12. 응답을 직렬화하고 보낸다.

```text
client
  -> API server
  -> parse / validate
  -> resolve target table
  -> lock(table)
  -> week7 SQL engine
  -> copy result out
  -> unlock(table)
  -> response
```

# 15. 이번 주 구현 목표를 어떻게 잡으면 좋은가

이번 주 목표는 아래처럼 잡는 것이 현실적이다.

1. thread pool이 있는 서버를 만든다.
2. `GET /health`, `POST /query`를 만든다.
3. 7주차 SQL 처리기를 adapter를 통해 호출한다.
4. table-level lock을 붙인다.
5. 같은 테이블은 충돌 가능한 요청에서 앞선 write를 보수적으로 앞지르지 않게 한다.
6. 다른 테이블은 병렬 허용한다.
7. 테스트와 README로 정합성과 구조를 설명한다.

이 정도면 아래 요구사항을 충분히 설명할 수 있다.

- thread pool
- 병렬 요청 처리
- 내부 DB 엔진 재사용
- API 구조
- 테스트와 README

# 16. 이번 주에 조심해야 할 오해

## 16.1 `thread pool이 있으면 DB도 자동으로 안전하다`

아니다.
thread pool은 요청 처리 구조일 뿐, DB 공유 상태 보호는 별도 문제다.

## 16.2 `그대로 활용`이니까 수정 없이 붙이면 된다

아니다.
엔진은 재사용하되, 서버와 엔진 사이 경계와 락 전략은 이번 주에 새로 설계해야 한다.

## 16.3 병렬 처리라면 모든 SQL이 동시에 실행돼야 한다

지금 단계에서는 그렇지 않다.
같은 테이블은 정합성을 위해 보수적으로 잠그고, 다른 테이블은 병렬 허용하는 구조가 더 현실적이다.

## 16.4 다른 클라이언트의 SELECT는 무조건 즉시 나가야 한다

그렇지 않다.
현재 엔진에는 snapshot이 없으므로, 같은 테이블의 write 중에는 다른 클라이언트 read도 기다리게 하는 편이 더 안전하다.

# 17. 개념 학습의 핵심 정리

- 이번 주는 `API 서버`와 `내부 DB 엔진`을 연결하는 주차다.
- 7주차 SQL 처리기는 재사용할 수 있지만, 멀티스레드 환경에 무수정 그대로 붙이면 위험하다.
- 가장 중요한 설계 포인트는 `서버 경계`와 `동시성 경계`를 나누는 것이다.
- 이번 주 가장 현실적인 락 전략은 `테이블 단위 lock`이다.
- `같은 테이블은 정합성 우선`, `다른 테이블은 병렬 허용`이 현재 엔진 수준에서 가장 설명 가능하고 구현 가능한 병렬성이다.
- 같은 테이블의 긴 write는 뒤 요청을 기다리게 할 수 있지만, 다른 테이블 요청까지 막지 않게 하는 것이 중요하다.
