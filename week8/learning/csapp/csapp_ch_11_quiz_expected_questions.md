## 1. 클라이언트와 서버의 차이는 무엇인가?

### 답변

- 클라이언트는 요청을 보내는 쪽이다.
- 서버는 요청을 기다리다가 받고, 처리한 뒤 응답을 보내는 쪽이다.

웹브라우저는 대표적인 클라이언트이고, 웹서버 프로그램은 대표적인 서버다.
중요한 차이는 누가 먼저 연결을 시도하고 요청을 보내는가에 있다.

### 헷갈리기 쉬운 포인트

클라이언트와 서버는 "항상 특정 프로그램 이름"이 아니라 역할이다.
예를 들어 proxy는 브라우저에게는 서버처럼 보이지만, origin 서버에게는 클라이언트처럼 동작한다.

---

## 2. IP 주소와 포트 번호는 왜 함께 필요한가?

### 답변

IP 주소는 어떤 컴퓨터인지 알려 주고, 포트 번호는 그 컴퓨터 안의 어떤 프로그램인지 알려 준다.

즉, 네트워크 연결은 보통 아래 두 정보를 함께 알아야 가능하다.

- IP 주소: 어느 호스트인지
- 포트 번호: 그 호스트 안의 어느 서비스인지

예를 들어 `127.0.0.1:8000`은 로컬 컴퓨터의 `8000`번 포트에서 실행 중인 서버를 뜻한다.

### 헷갈리기 쉬운 포인트

IP 주소만 알면 컴퓨터까지는 찾을 수 있지만, 그 안에서 어떤 서버 프로그램과 통신할지는 알 수 없다.
그래서 포트 번호가 꼭 필요하다.

---

## 3. DNS는 무엇이며 왜 필요한가?

### 답변

DNS는 사람이 읽기 쉬운 도메인 이름을 컴퓨터가 통신에 사용하는 IP 주소로 바꿔 주는 시스템이다.

예를 들어:

```text
www.example.com
-> DNS 조회
-> 93.184.216.34
```

브라우저는 보통 도메인 이름을 바로 연결에 쓰지 않고, DNS를 통해 IP 주소를 알아낸 뒤 그 IP 주소와 포트로 연결한다.

### 헷갈리기 쉬운 포인트

DNS는 이름을 IP 주소로 바꾸기 위한 프로토콜이자 시스템이다.
이번 주 관점에서는 실제 데이터 전달에 앞서 "어디로 가야 하는지 찾는 단계"로 이해하면 된다.

---

## 4. `getaddrinfo`는 왜 사용하는가?

### 답변

`getaddrinfo`는 호스트 이름과 서비스(포트) 정보를 실제 소켓 연결이나 바인딩에 사용할 수 있는 주소 구조로 바꿔 주는 함수다.

즉, 아래 과정을 코드 수준에서 처리해 준다.

- 도메인 이름 또는 문자열 형태의 주소 해석
- 포트 번호 해석
- IPv4/IPv6 등 실제 연결에 사용할 주소 구조 준비

### 헷갈리기 쉬운 포인트

`getaddrinfo`는 단순히 DNS만 하는 함수라고 보기보다,
"호스트 이름과 포트 문자열을 실제 연결에 쓸 수 있는 주소 리스트로 바꿔 주는 함수"
라고 이해하는 것이 더 정확하다.

---

## 5. TCP는 무엇인가?

### 답변

TCP는 연결을 먼저 맺고, 순서대로, 안정적으로 데이터를 주고받게 해 주는 전송 계층 프로토콜이다.

이번 주 기준으로 꼭 기억할 점은 아래와 같다.

- HTTP 요청/응답은 보통 TCP 위에서 동작한다.
- 서버와 클라이언트는 TCP 연결을 맺은 뒤 데이터를 읽고 쓴다.
- tiny 웹서버와 proxy 서버도 TCP 연결 위에서 동작한다.

### 헷갈리기 쉬운 포인트

TCP는 "데이터 형식"이 아니라 "안정적으로 보내는 통로"에 가깝다.
HTTP와 역할이 다르다.

---

## 6. TCP와 HTTP의 차이는 무엇인가?

### 답변

- TCP: 데이터를 안정적으로 주고받기 위한 전송 방식
- HTTP: 요청과 응답이 어떤 형식으로 오갈지를 정한 규칙

즉:

- TCP = 통로
- HTTP = 형식

브라우저가 서버에 `GET /index.html HTTP/1.0` 같은 요청을 보낸다는 것은,
HTTP 형식의 메시지를 TCP 연결 위에서 보낸다는 뜻이다.

---

## 7. file descriptor는 무엇인가?

### 답변

file descriptor는 운영체제가 열린 파일, 소켓 같은 입출력 대상을 관리하기 위해 붙이는 정수 번호다.

예를 들어:

- 파일을 열어도 file descriptor가 생긴다.
- socket을 만들어도 file descriptor가 생긴다.

그래서 소켓도 `read`, `write`, `close` 같은 흐름으로 다룰 수 있다.

### 헷갈리기 쉬운 포인트

socket은 특별한 무언가처럼 느껴지지만,
운영체제 입장에서는 읽고 쓰고 닫을 수 있는 파일 비슷한 입출력 대상으로 다뤄진다.

---

## 8. socket은 무엇인가?

### 답변

socket은 프로그램이 네트워크 통신을 하기 위해 운영체제로부터 받는 통신용 입출력 창구다.

`socket()` 호출이 성공하면 결국 file descriptor 하나를 돌려받고,
프로그램은 그 번호를 통해 네트워크 입출력을 수행한다.
즉, 아주 짧게 말하면 socket은 네트워크 통신용 file descriptor라고 이해할 수 있다.

즉:

- 클라이언트는 socket을 통해 서버에 연결하고
- 서버는 socket을 통해 요청을 받고
- 양쪽은 socket을 통해 데이터를 읽고 쓴다

### 헷갈리기 쉬운 포인트

socket은 HTTP 전용이 아니다.
HTTP, FTP, SSH 같은 여러 프로토콜이 socket 위에서 동작할 수 있다.
또 `socket`, `bind`, `listen`, `accept`, `connect` 같은 함수 묶음을 보통 BSD socket 인터페이스라고 부른다.

---

## 9. 서버 쪽 소켓 함수 흐름을 순서대로 설명하시오.

### 답변

보통 서버는 아래 순서로 동작한다.

```text
socket
-> bind
-> listen
-> accept
-> read / write
-> close
```

각 함수의 의미는 아래와 같다.

- `socket`: 통신용 소켓 생성
- `bind`: 특정 주소와 포트에 소켓 연결
- `listen`: 들어오는 연결을 받을 준비
- `accept`: 실제 클라이언트 연결 수락
- `read/write`: 요청과 응답 처리
- `close`: 연결 종료

---

## 10. `bind`, `listen`, `accept`의 역할을 각각 설명하시오.

### 답변

- `bind`: 서버 소켓을 특정 IP 주소와 포트 번호에 묶는다.
- `listen`: 이 소켓을 연결 요청을 기다리는 passive socket으로 만든다.
- `accept`: 실제로 들어온 연결 하나를 받아 새로운 연결 소켓을 반환한다.

### 헷갈리기 쉬운 포인트

`accept`가 반환하는 소켓은 리스닝 소켓 그 자체가 아니라,
"이번 연결을 위한 별도의 통신용 소켓"이다.

---

## 11. `listenfd`와 `connfd`의 차이는 무엇인가?

### 답변

- `listenfd`: 서버가 특정 포트에서 새 연결을 기다릴 때 사용하는 리스닝 소켓
- `connfd`: `accept`가 반환하는 실제 통신용 연결 소켓

서버는 보통 `listenfd`를 계속 열어 두고 반복해서 `accept`를 수행한다.
클라이언트 하나하나와 실제 데이터를 주고받는 것은 `connfd`로 한다.

### 헷갈리기 쉬운 포인트

초심자는 서버가 `listenfd` 하나로 곧바로 `read`, `write`한다고 오해하기 쉽다.
실제 통신은 보통 `accept` 이후 얻은 `connfd`에서 일어난다.

---

## 12. 클라이언트 쪽 소켓 함수 흐름을 순서대로 설명하시오.

### 답변

클라이언트는 보통 아래 순서로 동작한다.

```text
socket
-> connect
-> write
-> read
-> close
```

- `socket`: 통신용 소켓 생성
- `connect`: 서버의 IP:포트에 연결 시도
- `write`: 요청 전송
- `read`: 응답 수신
- `close`: 연결 종료

---

## 13. `accept`와 `connect`의 차이는 무엇인가?

### 답변

- `connect`는 클라이언트가 먼저 서버에 연결을 시도할 때 사용한다.
- `accept`는 서버가 기다리다가 들어온 연결을 받아들일 때 사용한다.

즉:

- 클라이언트 입장 핵심 함수: `connect`
- 서버 입장 핵심 함수: `accept`

---

## 14. HTTP 요청 메시지의 기본 구조를 설명하시오.

### 답변

HTTP 요청 메시지는 보통 아래 구조를 가진다.

```text
request line
headers
빈 줄
body(필요한 경우)
```

예:

```http
GET /index.html HTTP/1.0
Host: example.com
User-Agent: ...

```

여기서 첫 줄은 request line이고, 그 뒤는 헤더들이다.
빈 줄은 헤더가 끝났다는 표시다.

---

## 15. HTTP 응답 메시지의 기본 구조를 설명하시오.

### 답변

HTTP 응답 메시지는 보통 아래 구조를 가진다.

```text
status line
headers
빈 줄
body(필요한 경우)
```

예:

```http
HTTP/1.0 200 OK
Content-Type: text/html
Content-Length: 123

<html>...</html>
```

즉, 서버는 파일 내용만 보내는 것이 아니라,
먼저 HTTP 형식에 맞는 상태 줄과 헤더를 보내고, 그 뒤에 본문을 보내야 한다.

---

## 16. HTTP 메시지에서 빈 줄이 왜 중요한가?

### 답변

빈 줄은 "헤더가 여기서 끝난다"는 표시다.
서버는 이 빈 줄을 기준으로 요청 헤더 읽기를 멈추고,
그 뒤의 데이터를 본문으로 이해하거나 다음 단계로 넘어간다.
즉, HTTP 메시지에서 헤더 전체는 보통 `\r\n\r\n`을 기준으로 끝난다고 이해하면 된다.

### 헷갈리기 쉬운 포인트

tiny 웹서버의 `read_requesthdrs`는 바로 이 빈 줄이 나올 때까지 요청 헤더를 읽는 흐름과 연결된다.

---

## 17. `Host` 헤더는 왜 중요한가?

### 답변

`Host` 헤더는 클라이언트가 어떤 호스트를 요청하는지 서버에게 알려 준다.
특히 프록시나 가상 호스팅 환경에서는 매우 중요하다.
또 HTTP/1.1에서는 `Host` 헤더가 사실상 필수이므로, 요청을 정확히 구분하는 데 중요하다.

예:

```http
GET /home.html HTTP/1.1
Host: localhost:8000
```

proxy 과제에서는 origin 서버로 요청을 다시 만들 때 `Host` 헤더를 적절히 전달하거나 직접 생성해야 한다.

---

## 18. `tiny` 웹서버는 왜 교육용 예제로 좋은가?

### 답변

`tiny`는 아래 요소를 한 번에 보여 주기 때문이다.

- 소켓 기반 서버 구조
- HTTP 요청 읽기
- 정적 콘텐츠 응답
- 동적 콘텐츠 응답
- 간단한 에러 응답

즉, 네트워크 요청을 받아 실제 응답으로 돌려주는 전체 흐름을 비교적 작은 코드로 보여 준다.

---

## 19. 정적 콘텐츠와 동적 콘텐츠의 차이는 무엇인가?

### 답변

- 정적 콘텐츠: 이미 저장된 파일을 그대로 보내는 경우
- 동적 콘텐츠: 프로그램을 실행하거나 계산한 결과를 응답으로 보내는 경우

예를 들어:

- `home.html` 같은 파일을 읽어 보내면 정적 콘텐츠
- `cgi-bin/adder`를 실행해서 결과를 보내면 동적 콘텐츠

---

## 20. `tiny`의 `parse_uri`는 왜 중요한가?

### 답변

`parse_uri`는 요청 URI를 보고

- 정적 콘텐츠인지 동적 콘텐츠인지 판단하고
- 파일 경로와 CGI 인자를 나누는 역할을 한다

CS:APP tiny에서는 보통 `cgi-bin` 포함 여부로 정적/동적을 구분한다.
또 URI가 `/`로 끝나면 `home.html`을 붙이는 흐름도 여기서 처리한다.

---

## 21. `tiny`에서 `QUERY_STRING`과 `Dup2`가 왜 필요한가?

### 답변

동적 콘텐츠 실행 시:

- `QUERY_STRING`은 CGI 프로그램이 사용할 인자를 환경변수로 넘기기 위해 필요하다.
- `Dup2(fd, STDOUT_FILENO)`는 CGI 프로그램의 출력이 소켓으로 가게 만들기 위해 필요하다.

즉, `cgi-bin/adder` 같은 프로그램이 표준 출력으로 내보낸 결과가 그대로 클라이언트 응답이 되도록 연결하는 것이다.

### 헷갈리기 쉬운 포인트

`Execve`만 호출한다고 응답이 저절로 소켓으로 가지 않는다.
표준 출력을 소켓으로 바꿔 주는 `Dup2`가 필요하다.

---

## 22. `read_requesthdrs`는 무엇을 하는 함수인가?

### 답변

요청 라인 다음에 오는 HTTP 헤더들을 한 줄씩 읽다가,
빈 줄(`\r\n`)이 나오면 멈추는 함수다.

tiny의 기본 구현에서는 요청 헤더를 깊게 해석하지 않고,
일단 끝까지 읽고 버리는 수준이어도 충분하다.

---

## 23. `proxy`는 왜 단순 복사기가 아닌가?

### 답변

proxy는 브라우저가 보낸 요청을 그대로 복사만 하는 프로그램이 아니다.
보통 아래 과정을 수행한다.

- 클라이언트 요청 읽기
- URI 파싱
- origin 서버와 새 연결 생성
- origin 서버에 맞는 요청 문자열 재구성
- 응답 수신 후 다시 클라이언트에 전달

즉, 중간 전달자이지만 요청과 응답의 구조를 모두 이해하고 다뤄야 한다.

---

## 24. proxy에서 absolute URI와 path는 어떻게 다르게 쓰이는가?

### 답변

브라우저는 proxy에 보통 absolute URI를 포함한 요청을 보낸다.

예:

```http
GET http://localhost:8000/home.html HTTP/1.1
```

하지만 origin 서버에 보낼 때는 보통 request line을 아래처럼 바꿔야 한다.

```http
GET /home.html HTTP/1.0
```

즉, proxy는 absolute URI에서

- host
- port
- path

를 분리한 뒤, origin 서버에 맞는 요청 형식으로 다시 만들어야 한다.

---

## 25. proxy에서 `Connection: close`와 `Proxy-Connection: close`를 넣는 이유는 무엇인가?

### 답변

기본 구현에서 연결 관리를 단순하게 하기 위해서다.
이 헤더들을 `close`로 두면 응답이 끝난 뒤 연결이 닫히고,
proxy는 EOF까지 읽는 단순한 방식으로 relay를 구현하기 쉬워진다.
또 proxy가 origin 서버 쪽 요청을 `HTTP/1.0` 기반의 비지속 연결로 단순화해서 다루기 쉬워진다.

### 헷갈리기 쉬운 포인트

이 설정이 없으면 persistent connection 같은 더 복잡한 상황을 다뤄야 할 수 있다.
proxylab 기본 구현에서는 연결을 짧게 가져가는 쪽이 훨씬 단순하다.
또 `Proxy-Connection`은 표준 HTTP 헤더라기보다 proxy 구현 편의를 위해 자주 쓰이는 비표준 헤더라는 점도 알아두면 좋다.

---

## 26. 왜 proxy 응답 relay에서는 `Rio_readlineb`만으로 처리하면 안 되는가?

### 답변

응답 본문에는 텍스트뿐 아니라 바이너리 데이터가 올 수 있기 때문이다.
예를 들어 이미지 파일이나 실행 파일은 줄 단위 텍스트처럼 다루면 안 된다.

그래서 proxy는 보통:

```c
while ((n = Rio_readnb(&server_rio, buf, MAXBUF)) > 0) {
    Rio_writen(clientfd, buf, n);
}
```

처럼 바이트 단위로 안전하게 relay한다.

### 헷갈리기 쉬운 포인트

`Rio_readlineb`는 헤더 읽기에는 편하지만,
본문 전체 relay까지 그것만으로 처리하면 binary-safe 하지 않을 수 있다.
즉, 응답 헤더는 줄 단위로 읽기 쉽지만, 응답 본문은 바이너리까지 안전하게 전달하려면 바이트 단위 처리가 더 적절하다.

---

## 27. blocking I/O란 무엇이며 왜 중요한가?

### 답변

blocking I/O는 어떤 입출력 호출이 끝날 때까지 현재 실행 흐름이 기다리는 방식을 뜻한다.

예를 들어 순차 서버가 한 클라이언트의 요청을 오래 처리하고 있으면,
다른 클라이언트는 그동안 기다려야 할 수 있다.

이 개념이 중요한 이유는 다음과 같다.

- tiny는 기본적인 순차 서버 구조를 보여 준다.
- proxy는 한 요청이 막혀도 다른 요청을 처리할 수 있어야 한다.
- 그래서 동시성 구조가 필요해진다.

---

## 28. 왜 proxy에서는 concurrency가 필요한가?

### 답변

한 요청이 느린 서버와 통신 중이거나 응답이 오래 걸리면,
순차 proxy는 그동안 다른 요청을 처리하지 못할 수 있다.

예를 들어:

- 첫 번째 요청이 멈춰 있는 origin 서버로 감
- proxy가 순차 처리만 함
- 두 번째 요청은 정상적인 tiny 서버 요청이어도 기다리게 됨

이 문제를 줄이기 위해 thread-per-connection 같은 동시성 구조를 사용한다.

---

## 29. `rio_*` 함수는 왜 쓰는가?

### 답변

`Rio_readlineb`, `Rio_readnb`, `Rio_writen` 같은 함수는 CS:APP에서 제공하는 robust I/O 래퍼다.
즉, 운영체제나 C 표준 라이브러리의 기본 함수가 아니라, 책에서 예제 구현을 쉽게 하기 위해 제공하는 보조 함수들이다.

이 함수들이 도움이 되는 이유는 아래와 같다.

- 줄 단위 읽기가 편하다
- 원하는 바이트 수를 더 안정적으로 처리할 수 있다
- 네트워크 입출력 코드를 단순하게 만들 수 있다

초심자 기준으로는 "네트워크 입출력을 더 안전하고 편하게 다루는 보조 함수"라고 이해하면 충분하다.

---

## 30. 아래 코드를 보고 왜 문제가 될 수 있는지 설명하시오.

```c
while (1) {
    int connfd = Accept(listenfd, (SA *)&clientaddr, &clientlen);
    pthread_create(&tid, NULL, thread, &connfd);
}
```

### 답변

`connfd`는 루프 안의 지역변수인데, 그 주소를 그대로 thread에 넘기면 다음 반복에서 값이 바뀔 수 있다.
그 결과 여러 thread가 같은 주소를 보거나 잘못된 파일 디스크립터 값을 읽을 수 있다.

안전한 방식은 보통 아래와 같다.

```c
int *connfdp = Malloc(sizeof(int));
*connfdp = connfd;
pthread_create(&tid, NULL, thread, connfdp);
```

### 헷갈리기 쉬운 포인트

이 문제는 네트워크 개념 자체라기보다,
동시성 환경에서 지역변수 주소를 넘길 때 생길 수 있는 C 코드상의 대표적인 실수다.

---

## 31. 아래 서버 코드 흐름을 말로 설명하시오.

```c
listenfd = Open_listenfd(argv[1]);
while (1) {
    connfd = Accept(listenfd, (SA *)&clientaddr, &clientlen);
    doit(connfd);
    Close(connfd);
}
```

### 답변

이 코드는 서버가 특정 포트에서 연결을 기다리다가,
클라이언트 연결이 들어오면 하나 받아서 `doit(connfd)`로 요청을 처리하고,
처리가 끝나면 그 연결을 닫는 순차 서버 흐름이다.

즉:

1. 리스닝 소켓 생성
2. 무한 루프 진입
3. 클라이언트 연결 하나 수락
4. 해당 연결에 대한 요청/응답 처리
5. 연결 종료
6. 다시 다음 연결 대기

---

## 32. 브라우저에서 URL을 입력한 뒤 응답을 받기까지의 전체 흐름을 설명하시오.

### 답변

아래 흐름으로 설명할 수 있다.

```text
사용자
-> 브라우저에 URL 입력
-> DNS가 도메인 이름을 IP 주소로 변환
-> 브라우저가 서버의 IP:포트로 TCP 연결
-> 브라우저가 HTTP 요청 전송
-> 서버 프로그램이 socket으로 요청을 읽음
-> 서버가 파일 또는 내부 로직 처리
-> 서버가 HTTP 응답 전송
-> 브라우저가 결과 표시
```

이 문제는 이번 주 학습 전체를 한 번에 묻는 대표적인 서술형 문제다.
`DNS`, `IP`, `TCP`, `socket`, `HTTP`, `서버 처리`가 모두 연결된다는 점을 보여 줄 수 있어야 한다.

---

## 33. TCP/IP 계층을 큰 흐름 중심으로 설명하시오.

### 답변

입문 단계에서는 TCP/IP 계층을 보통 아래처럼 크게 이해하면 충분하다.

- 응용 계층: HTTP 같은 애플리케이션 프로토콜
- 전송 계층: TCP, UDP 같은 데이터 전달 방식
- 인터넷 계층: IP를 통한 호스트 간 전달
- 링크 계층: 실제 네트워크 장치를 통한 프레임 전달

예를 들어 웹 요청은 아래처럼 볼 수 있다.

```text
HTTP
-> TCP
-> IP
-> 링크 계층
```

즉, HTTP는 "무슨 형식으로 말할지", TCP는 "어떻게 안정적으로 보낼지", IP는 "어느 호스트로 갈지"를 맡는다고 보면 된다.

### 헷갈리기 쉬운 포인트

HTTP와 TCP를 같은 수준의 개념으로 섞어 설명하면 안 된다.
HTTP는 응용 계층, TCP는 전송 계층이다.

## 34. Datagram Socket과 Stream Socket의 차이는 무엇인가?

### 답변

- Datagram Socket은 보통 UDP와 연결된다.
- Stream Socket은 보통 TCP와 연결된다.

차이를 간단히 정리하면 아래와 같다.

- Datagram Socket: 연결 없이 메시지 단위로 주고받는 방식
- Stream Socket: 연결을 맺고 바이트 스트림 형태로 주고받는 방식

이번 주 웹서버와 proxy 서버는 보통 **Stream Socket(TCP)** 을 사용한다.

### 헷갈리기 쉬운 포인트

웹 요청/응답은 보통 순서와 안정성이 중요하므로,
이번 주 범위에서는 Datagram보다 Stream Socket이 더 직접적으로 연결된다.
또 UDP 기반 Datagram은 메시지 경계를 유지하지만, 연결을 따로 맺지 않고 순서나 재전송을 보장하지 않는다.
반면 TCP 기반 Stream은 연결을 맺고, 순서와 신뢰성을 보장하면서 연속된 바이트 흐름처럼 데이터를 다룬다.

---

## 35. 웹서버는 어떤 기능들의 모음이라고 볼 수 있는가?

### 답변

웹서버는 단순히 파일 하나를 보내는 프로그램이 아니라,
아래 기능들이 연결된 프로그램이라고 볼 수 있다.

- 클라이언트 연결을 기다리는 기능
- HTTP 요청을 읽는 기능
- 요청 경로와 메서드를 해석하는 기능
- 정적 파일을 찾고 보내는 기능
- 필요하면 동적 프로그램을 실행하는 기능
- HTTP 응답 헤더와 본문을 만들어 보내는 기능
- 에러 상황에서 적절한 상태코드를 보내는 기능

즉, 웹서버는 "네트워크 연결 + 요청 해석 + 콘텐츠 준비 + 응답 생성"의 모음이다.

---

## 36. MIME type은 무엇이며 왜 필요한가?

### 답변

MIME type은 서버가 보내는 콘텐츠의 종류를 알려 주는 문자열이다.

예:

- `text/html`
- `text/plain`
- `image/jpeg`
- `image/gif`

브라우저는 이 정보를 보고 응답 본문을 어떻게 해석할지 결정한다.
그래서 웹서버는 `Content-Type` 헤더에 올바른 MIME type을 넣어 주는 것이 중요하다.

### 헷갈리기 쉬운 포인트

파일 확장자와 MIME type은 연결되지만 완전히 같은 개념은 아니다.
웹서버는 확장자를 참고해서 MIME type을 결정해 응답 헤더에 넣는다.

---

## 37. CGI는 무엇이며 동적 콘텐츠와 어떻게 연결되는가?

### 답변

CGI(Common Gateway Interface)는 웹서버가 외부 프로그램을 실행하고,
그 결과를 HTTP 응답으로 돌려주기 위한 방식이다.

tiny에서는 `cgi-bin/adder` 같은 프로그램이 예시다.
흐름은 보통 아래와 같다.

1. 클라이언트가 동적 요청을 보냄
2. 웹서버가 해당 프로그램을 실행함
3. 프로그램의 표준 출력이 소켓으로 연결됨
4. 그 결과가 클라이언트 응답이 됨

즉, CGI는 "동적 콘텐츠를 만들기 위한 실행 인터페이스"라고 볼 수 있다.
tiny 기준으로는 서버가 먼저 상태 줄을 보내고, CGI 프로그램이 나머지 헤더와 본문을 출력하는 흐름으로 이해하면 더 정확하다.

### 헷갈리기 쉬운 포인트

정적 콘텐츠는 파일 자체를 읽어 보내지만,
동적 콘텐츠는 프로그램을 실행한 결과를 보내는 점이 다르다.

---

## 38. HTTP 메서드는 무엇이며 대표적으로 어떤 것들이 있는가?

### 답변

HTTP 메서드는 클라이언트가 서버에게 "무슨 종류의 요청을 하는지"를 나타낸다.

대표적인 예는 아래와 같다.

- `GET`: 리소스를 가져오고 싶다는 요청
- `POST`: 데이터를 서버에 보내 처리하라는 요청
- `HEAD`: 본문 없이 헤더만 받고 싶다는 요청

tiny의 기본 예제는 보통 `GET` 위주로 동작한다.

### 헷갈리기 쉬운 포인트

메서드는 URL과 별개다.
같은 경로라도 `GET`, `POST`, `HEAD`는 서로 다른 의미를 가질 수 있다.

---

## 39. `HEAD` 메서드는 `GET`과 어떻게 다른가?

### 답변

`HEAD`는 `GET`과 거의 같은 요청이지만,
응답에서 **본문(body)은 보내지 않고 헤더만 보내는 메서드**다.

즉:

- `GET`: 헤더 + 본문
- `HEAD`: 헤더만

`HEAD`는 리소스의 존재 여부, 크기, 타입 같은 정보를 본문 없이 빠르게 확인할 때 유용하다.
즉, 서버 입장에서는 `GET`과 비슷하게 처리하되 본문만 보내지 않는 메서드라고 이해하면 된다.

### 헷갈리기 쉬운 포인트

`HEAD`는 "아무 응답도 없는 요청"이 아니다.
본문만 빠질 뿐, 상태 줄과 헤더는 여전히 중요하다.

---

## 40. HTTP 상태코드에서 자주 나오는 예를 설명하시오.

### 답변

상태코드는 서버가 요청을 어떻게 처리했는지 나타내는 숫자다.

대표적인 예:

- `200 OK`: 정상 처리
- `301 Moved Permanently`: 다른 위치로 영구 이동
- `403 Forbidden`: 접근 권한 없음
- `404 Not Found`: 요청한 리소스를 찾을 수 없음
- `500 Internal Server Error`: 서버 내부 오류

`301`은 보통 클라이언트가 이동할 새 주소를 알 수 있도록 `Location` 헤더와 함께 사용된다.

범주로 묶어 보면 아래처럼 이해할 수 있다.

- `2xx`: 성공
- `3xx`: 리다이렉션
- `4xx`: 클라이언트 요청 문제
- `5xx`: 서버 내부 문제

tiny에서는 없는 파일이나 권한 없는 파일에 대해
적절한 상태코드를 포함한 에러 응답을 보내야 한다.

### 헷갈리기 쉬운 포인트

상태코드는 단순 숫자 암기보다,
"이 상황에서 서버가 클라이언트에게 어떤 의미를 알려 주는가"로 이해하는 것이 좋다.

---

## 41. HTTP 헤더는 왜 필요한가?

### 답변

HTTP 헤더는 요청과 응답에 대한 부가 정보를 전달한다.

예를 들면:

- `Host`: 어떤 호스트를 요청하는지
- `User-Agent`: 어떤 클라이언트인지
- `Content-Type`: 본문의 종류가 무엇인지
- `Content-Length`: 본문 크기가 얼마인지
- `Connection`: 연결을 어떻게 관리할지

즉, 헤더는 단순 장식이 아니라
"본문을 어떻게 해석하고 연결을 어떻게 다룰지"를 정하는 중요한 메타데이터다.

---

## 42. 프록시 서버는 무엇이며 왜 사용하는가?

### 답변

프록시 서버는 클라이언트와 원본 서버 사이의 중간에서 요청과 응답을 대신 전달하는 서버다.

흐름은 아래와 같다.

```text
브라우저 -> 프록시 -> 원본 서버
```

프록시를 사용하는 이유는 아래처럼 정리할 수 있다.

- 요청을 대신 전달할 수 있다
- 응답을 캐시해서 재사용할 수 있다
- 여러 요청을 관리하거나 필터링할 수 있다
- 클라이언트와 서버 사이에 중간 제어 지점을 둘 수 있다

이번 주 proxy 과제는 단순 전달뿐 아니라
동시성 처리와 캐시까지 확장해 보는 점이 핵심이다.

### 헷갈리기 쉬운 포인트

프록시는 단순한 복사기가 아니다.
클라이언트 요청을 이해하고, 원본 서버용 요청으로 재구성하고, 응답을 다시 전달하는 구조를 가진다.

## 마지막 정리

이번 8주차 쪽지시험은 단순 용어 암기만 물을 가능성보다,
아래 두 가지를 같이 볼 가능성이 높다.

1. 네트워크 키워드를 정확히 설명할 수 있는가
2. 그 키워드가 tiny 웹서버와 proxy 흐름에서 어떻게 연결되는지 설명할 수 있는가

즉, 문제를 풀 때도 따로따로 외우기보다 아래처럼 연결해서 생각하는 것이 좋다.

```text
DNS
-> IP:포트
-> TCP 연결
-> socket
-> HTTP 요청/응답
-> tiny 웹서버
-> proxy와 concurrency
```

이 흐름으로 설명할 수 있으면, 개념형 문제와 서술형 문제 모두 대응하기 훨씬 쉬워진다.
