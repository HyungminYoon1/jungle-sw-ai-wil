# CS:APP 11장 기본 학습자료

## 주제: 비전공자를 위한 네트워크 프로그래밍과 Web/Proxy Server 입문

---

## 이 자료의 목표

이 문서는 **CS:APP 11장 네트워크 프로그래밍**을 처음 보는 학습자도 큰 흐름을 잡을 수 있도록 쉽게 풀어 쓴 기본 학습자료다.

이번 주 팀 학습의 목표는 다음과 같다.

- 이번 주에 왜 `Web/Proxy Server`를 배우는지 설명할 수 있다.
- `BSD socket`, `IP`, `TCP`, `HTTP`, `file descriptor`, `DNS`가 각각 무엇인지 큰 흐름으로 이해할 수 있다.
- 클라이언트의 요청이 서버에 도착하고, 응답이 다시 돌아가는 전체 과정을 설명할 수 있다.
- `echo server -> tiny 웹서버 -> proxy 서버`가 왜 이런 순서로 이어지는지 이해할 수 있다.
- 수요 코딩회의 `미니 DBMS API 서버`가 왜 이번 학습과 연결되는지 설명할 수 있다.

이 자료는 **CS:APP 11장 전체를 세세하게 해설하는 문서가 아니다.**
이번 주 과제와 직접 연결되는 범위를 우선 이해하도록 돕는 문서다.

이번 주 기준으로 아래 범위를 먼저 이해하면 충분하다.

- 클라이언트-서버 모델
- IP 주소와 포트
- DNS
- TCP 연결
- socket과 file descriptor
- HTTP 요청/응답
- tiny 웹서버의 역할
- proxy 서버의 역할

즉, 이번 주에는 네트워크 이론을 깊게 파고들기보다,
**요청이 어떻게 이동하고, 서버 프로그램이 그것을 어떻게 읽고 응답하는지**
를 먼저 잡는 것이 중요하다.

# 1. 왜 이번 주에 CS:APP 11장을 공부해야 하는가

지난주까지는 메모리, 포인터, SQL 처리기, B+ Tree처럼 프로그램 내부 구조를 많이 다뤘다.
이번 주에는 시야를 조금 넓혀서, **외부 클라이언트의 요청을 서버가 받아 처리하는 구조**를 배운다.

공지 기준으로 이번 주 학습 흐름은 아래와 같다.

```text
소켓 이해
-> 간단한 client/server 요청-응답
-> tiny 웹서버 구현
-> proxy 서버 도전
-> 수요 코딩회 API 서버 구현
```

이 흐름이 중요한 이유는 다음과 같다.

- 이제 프로그램은 혼자 실행되는 것이 아니라, 네트워크를 통해 다른 프로그램과 통신해야 한다.
- 서버는 단순 계산만 하는 것이 아니라, 외부 요청을 받아 적절한 응답을 돌려줘야 한다.
- 수요 코딩회 API 서버도 결국은 네트워크 요청을 받아 내부 DB 엔진으로 연결하는 구조다.

즉, 이번 주 CS:APP 11장은
**네트워크 요청이 실제 서버 코드로 연결되는 과정**을 이해하기 위한 기본 토대라고 보면 된다.

- 어떤 형식으로 요청을 보낼지 알아야 한다.

- 처리 결과를 응답으로 보내야 한다.

CS:APP 11장은 바로 이 과정을 설명한다.

# 3. 먼저 큰 그림: 웹 요청은 어떻게 오가는가

브라우저에서 어떤 주소에 접속한다고 생각해 보자.

```text
브라우저
  -> DNS로 서버 주소 확인
  -> 서버와 TCP 연결
  -> HTTP 요청 전송
  -> 서버가 요청 읽기
  -> 서버가 파일 또는 프로그램 실행 결과 준비
  -> HTTP 응답 전송
  -> 브라우저가 응답 표시
```

이 흐름 안에 이번 주 키워드가 거의 모두 들어 있다.

- `DNS`: 이름을 IP 주소로 바꾸는 과정
- `IP`: 어느 컴퓨터로 갈 것인가
- `TCP`: 안정적으로 연결하고 데이터를 주고받는 방식
- `socket`: 프로그램이 네트워크 통신을 다루는 창구
- `HTTP`: 요청과 응답의 형식

즉, 이번 주 키워드는 서로 따로 놀지 않는다.
**하나의 요청이 이동하는 과정 안에서 각자 맡은 역할이 다를 뿐이다.**

# 4. 네트워크를 처음 볼 때 필요한 최소 개념

## 4.1 클라이언트와 서버

가장 먼저 잡아야 할 개념은 클라이언트-서버 모델이다.

- **클라이언트(client)**: 요청을 보내는 쪽
- **서버(server)**: 요청을 받고 응답하는 쪽

예를 들어:

- 웹브라우저 -> 클라이언트
- 웹서버 프로그램 -> 서버

여기서 중요한 점은 "누가 먼저 요청을 보내는가"다.
보통 클라이언트가 먼저 요청하고, 서버는 기다리다가 요청을 받아 처리한다.

## 4.2 IP 주소와 포트

네트워크에서 어떤 서버와 통신하려면, 어디로 가야 하는지 알아야 한다.
이때 필요한 것이 **IP 주소**와 **포트 번호**다.

- **IP 주소**: 어떤 컴퓨터인지 알려주는 주소
- **포트 번호**: 그 컴퓨터 안에서 어떤 프로그램인지 알려주는 번호

비유하면:

- IP 주소 = 건물 주소
- 포트 번호 = 건물 안의 호수

즉, 서버 프로그램에 연결한다는 것은
**특정 IP 주소의 특정 포트에 연결한다**는 뜻이다.

## 4.3 DNS

우리는 보통 `www.example.com` 같은 이름을 입력하지,
`93.184.216.34` 같은 숫자 IP를 외우지는 않는다.

이때 필요한 것이 **DNS(Domain Name System)** 다.

DNS는
**사람이 읽기 쉬운 이름을 컴퓨터가 통신에 쓰는 IP 주소로 바꾸는 시스템**
이라고 이해하면 된다.

즉:

```text
www.example.com
-> DNS 조회
-> 93.184.216.34
```

브라우저는 보통 이 과정을 먼저 거친 뒤 서버에 연결한다.

### 코드에서는 어떻게 주소를 다루는가

브라우저나 프로그램은 DNS 결과를 문자열로만 들고 있지 않는다.
C 코드에서는 보통 `getaddrinfo`를 사용해 호스트 이름과 포트 번호를
실제로 `connect`나 `bind`에 사용할 수 있는 주소 정보로 바꾼다.

- `host`: 예: `www.example.com`
- `service`: 예: `80`, `8080`
- `getaddrinfo`: host와 service를 실제 연결에 사용할 수 있는 주소 정보로 변환
- `sockaddr`: 운영체제가 이해하는 공통 주소 구조

즉, 이번 주 코드에서 `getaddrinfo`가 보인다면
"도메인 이름과 포트 번호를 소켓 연결에 사용할 수 있는 주소로 바꾸는 단계"
라고 이해하면 된다.

## 4.4 TCP

네트워크로 데이터를 보낼 때는 여러 방식이 있지만,
이번 주 웹서버와 프록시 서버 학습에서는 **TCP**가 핵심이다.

TCP는 아주 단순하게 말하면:

- 연결을 먼저 맺고
- 순서대로
- 안정적으로
- 데이터를 주고받게 해 주는 방식

이라고 이해하면 된다.

초심자 기준으로 이번 주에 꼭 기억할 점은 아래 정도면 충분하다.

- HTTP 요청과 응답은 보통 TCP 위에서 오간다.
- 서버는 클라이언트와 TCP 연결을 맺은 뒤 데이터를 읽고 쓴다.
- tiny 웹서버와 proxy 서버도 이 흐름 위에서 동작한다.

## 4.5 file descriptor

C에서 파일을 읽고 쓸 때 `read`, `write`, `close` 같은 함수를 본 적이 있을 수 있다.
네트워크 소켓도 비슷하게 다룬다.

이때 중요한 개념이 **file descriptor**다.

file descriptor는 운영체제가 관리하는 열린 대상에 붙이는 번호라고 생각하면 된다.

예를 들어:

- 파일을 열어도 file descriptor가 생길 수 있고
- 소켓을 만들어도 file descriptor가 생길 수 있다

즉, 운영체제 입장에서는
파일도, 소켓도, 여러 입출력 대상도
**읽고 쓰고 닫을 수 있는 대상**으로 다룬다.

그래서 socket에서도 `read`, `write`, `close` 같은 흐름이 자연스럽게 등장한다.

## 4.6 socket

socket은 초심자가 가장 많이 헷갈리는 단어 중 하나다.

이번 주 기준으로는 아래처럼 이해하면 충분하다.

> socket은 프로그램이 네트워크 통신을 하기 위해 운영체제로부터 받는 통신용 입출력 창구다.

즉:

- 클라이언트는 socket을 통해 서버에 연결하고
- 서버는 socket을 통해 요청을 받고
- 양쪽 모두 socket을 통해 데이터를 읽고 쓴다

## 4.7 HTTP

HTTP는
**클라이언트와 서버가 어떤 형식으로 요청과 응답을 주고받을지 정한 규칙**
이다.

예를 들어 브라우저는 이런 요청을 보낼 수 있다.

```http
GET /index.html HTTP/1.0
Host: example.com

```

그리고 서버는 이런 응답을 돌려줄 수 있다.

```http
HTTP/1.0 200 OK
Content-Type: text/html
Content-Length: 123

```

여기서 중요한 것은,
TCP가 "데이터를 안정적으로 보내는 통로"라면
HTTP는 "그 데이터의 의미와 형식"에 가깝다는 점이다.

즉:

- TCP = 어떻게 안정적으로 주고받을 것인가
- HTTP = 무엇을 어떤 형식으로 주고받을 것인가

추가로 초심자가 꼭 기억하면 좋은 점이 하나 있다.
HTTP 메시지는 보통 아래 구조를 가진다.

```text
start line
headers
빈 줄
body(필요한 경우)
```

여기서 **빈 줄**은 "헤더가 여기서 끝난다"는 표시다.
tiny 웹서버가 요청을 읽거나, 직접 요청 문자열을 만들어 볼 때 이 구조를 알고 있어야 흐름이 덜 헷갈린다.

# 5. socket 관련 함수는 왜 중요한가

CS:APP 11장을 읽을 때는 네트워크 이론만 보는 것이 아니라,
실제 서버 코드가 어떤 함수들로 연결되는지도 함께 봐야 한다.

이번 주에 자주 보게 될 함수 흐름은 대략 아래와 같다.

## 5.1 서버 쪽 큰 흐름

```text
socket
-> bind
-> listen
-> accept
-> read / write
-> close
```

각 함수의 의미를 아주 간단히 보면:

- `socket`: 통신용 소켓 생성
- `bind`: 이 소켓을 특정 주소와 포트에 연결
- `listen`: 들어오는 연결을 받을 준비
- `accept`: 실제 클라이언트 연결 수락
- `read / write`: 요청과 응답 처리
- `close`: 연결 종료

여기서 초심자가 꼭 구분해야 할 것이 두 가지 소켓이다.

- `listenfd`: 서버가 특정 포트에서 새 연결을 기다릴 때 사용하는 리스닝 소켓
- `connfd`: `accept`가 반환하는 실제 통신용 연결 소켓

즉, 서버는 보통 `listenfd`로 계속 기다리고,
클라이언트가 들어오면 `accept`로 새로운 `connfd`를 받아
그 소켓에 대해 `read`와 `write`를 수행한다.

## 5.2 클라이언트 쪽 큰 흐름

```text
socket
-> connect
-> write
-> read
-> close
```

즉:

- 서버는 기다리는 쪽
- 클라이언트는 먼저 연결하는 쪽

이라는 차이가 있다.

## 5.3 왜 순차 서버가 느려질 수 있는가

기본적인 `accept`, `read`, `write`는 **blocking I/O**로 동작하는 경우가 많다.
즉, 작업이 끝날 때까지 현재 흐름이 기다릴 수 있다는 뜻이다.

예를 들어 순차 서버가 한 클라이언트의 요청을 오래 읽거나,
응답을 천천히 보내고 있으면 그동안 다른 요청은 기다려야 할 수 있다.

이 개념이 중요한 이유는 다음과 같다.

- tiny 웹서버는 기본적인 요청/응답 구조를 이해하는 데 집중하는 예제다.
- proxy 과제에서는 여러 요청을 더 잘 처리하기 위해 병렬 처리가 중요해진다.
- 수요 코딩회 API 서버에서도 thread pool과 병렬 요청 처리가 필요한 이유가 여기서 이어진다.

즉, `blocking I/O`를 이해하면
"왜 순차 처리만으로는 한계가 생기고, 왜 동시성 구조가 필요한가"
를 훨씬 자연스럽게 이해할 수 있다.

## 5.4 CS:APP에서 `rio_*`가 왜 나오는가

CS:APP 예제 코드를 보다 보면 `Rio_readlineb`, `Rio_writen` 같은 함수가 나온다.
이것은 책에서 제공하는 **robust I/O 래퍼**라고 보면 된다.

초심자 기준으로는 아래 정도만 이해하면 충분하다.

- 네트워크에서는 한 번의 `read`나 `write`로 원하는 길이만큼 항상 깔끔하게 처리된다고 기대하면 안 된다.
- 줄 단위로 읽고 싶은 경우도 있고, 정해진 바이트 수만큼 안정적으로 보내고 싶은 경우도 있다.
- `rio_*` 함수는 이런 상황을 조금 더 다루기 쉽게 만든 도우미다.

즉, tiny 웹서버 코드에서 `rio_*`가 보이면
"네트워크 입출력을 더 안전하고 편하게 처리하기 위한 보조 함수"
라고 이해하면 된다.

# 6. 왜 echo server부터 시작하는가

공지에 나온 흐름을 보면 바로 웹서버부터 하지 않고,
먼저 socket 기반의 간단한 요청/응답을 해 보라고 되어 있다.

이유는 단순하다.

웹서버는 생각보다 한 번에 많은 개념이 들어간다.

- 네트워크 연결
- 요청 읽기
- HTTP 형식 이해
- 파일 읽기
- 응답 만들기

그래서 처음에는 더 단순한 형태인 **echo server**가 도움이 된다.

echo server는 보통:

- 클라이언트가 데이터를 보내면
- 서버가 그 데이터를 읽고
- 그대로 다시 돌려주는

아주 단순한 구조다.

이 과정을 통해 먼저 아래를 익힐 수 있다.

- 연결이 어떻게 만들어지는가
- 서버는 어디서 기다리는가
- 데이터를 어떻게 읽고 쓰는가
- 클라이언트와 서버가 주고받는 흐름이 무엇인가

즉, echo server는
**HTTP 없이 순수한 요청/응답 흐름을 먼저 익히는 단계**
라고 보면 된다.

---

# 7. tiny 웹서버는 무엇을 배우기 위한 예제인가

CS:APP의 tiny 웹서버는 실무용 완성 서버가 아니라,
**웹서버의 핵심 동작을 배우기 위한 교육용 예제**다.

tiny 웹서버를 통해 배우는 것은 아래와 같다.

## 7.1 HTTP 요청 읽기

서버는 클라이언트가 보낸 요청 라인과 헤더를 읽는다.

예를 들어:

- 어떤 경로를 요청했는지
- 어떤 메서드를 사용했는지
- 어떤 버전의 HTTP인지

같은 정보를 확인한다.

## 7.2 정적 콘텐츠와 동적 콘텐츠

웹서버는 요청에 따라 다른 방식으로 응답할 수 있다.

- **정적 콘텐츠(static content)**:
  이미 저장된 파일을 그대로 보내는 경우
- **동적 콘텐츠(dynamic content)**:
  프로그램을 실행하거나 계산한 결과를 보내는 경우

`tiny/cgi-bin/adder.c`가 등장하는 이유도 여기와 연결된다.
즉, 서버는 단순 파일 전송만 하는 것이 아니라,
프로그램 실행 결과도 응답으로 만들 수 있다는 점을 보여준다.

## 7.3 HTTP 응답 만들기

서버는 단순히 파일 내용만 보내는 것이 아니라,
HTTP 응답 형식에 맞는 헤더와 본문을 구성해야 한다.

즉, tiny 웹서버는
**네트워크 통신 + HTTP 형식 + 파일/프로그램 처리**
가 함께 들어가는 예제다.

# 8. proxy 서버는 왜 tiny 다음에 오는가

proxy 서버는 브라우저와 웹서버 사이의 중간에 위치한다.

```text
브라우저 -> 프록시 -> 웹서버
```

브라우저는 프록시에 요청을 보내고,
프록시는 그 요청을 웹서버에 전달하고,
웹서버 응답을 다시 브라우저로 전달한다.

이 구조가 중요한 이유는 다음과 같다.

## 8.1 요청을 대신 전달하는 구조를 배운다

proxy는 단순 서버가 아니라,
**한쪽에서는 클라이언트처럼 동작하고 다른 쪽에서는 서버처럼 동작하는 프로그램**
이다.

즉, 프록시를 구현하면

- 클라이언트 요청 읽기
- 다른 서버에 새 연결 만들기
- 요청 전달하기
- 응답 다시 받기
- 응답 다시 전달하기

를 모두 다루게 된다.

## 8.2 순차 처리와 병렬 처리 차이를 배운다

공지에도 나와 있듯이 proxy 과제는 다음으로 확장된다.

- 순차적 요청 처리
- 병렬적 요청 처리
- 요청 캐시

즉, 단순히 네트워크만 배우는 것이 아니라,
**서버가 여러 요청을 어떻게 다룰 것인가**까지 이어진다.

이 부분은 수요 코딩회의 API 서버와도 연결된다.
API 서버도 결국 외부 요청을 동시에 받아야 하기 때문이다.

## 8.3 캐시가 왜 필요한지도 배운다

같은 요청이 반복된다면,
매번 원본 서버에 다시 요청하기보다
중간에서 저장해 둔 결과를 재사용할 수 있다.

즉, proxy 과제를 통해
단순 전달을 넘어서 성능과 구조까지 생각하게 된다.

# 9. 이번 주 수요 코딩회 API 서버와는 어떻게 연결되는가

이번 주 수요 코딩회는 `미니 DBMS - API 서버` 구현이다.

이 과제는 표면적으로는 tiny 웹서버와 달라 보이지만,
기본 구조는 연결된다.

## 9.1 공통점

- 외부 클라이언트 요청을 받는다
- 요청을 읽고 해석한다
- 내부 로직에 연결한다
- 결과를 응답으로 돌려준다

즉, tiny 웹서버에서는 파일이나 CGI 프로그램으로 연결했다면,
API 서버에서는 내부 DB 엔진으로 연결한다고 보면 된다.

## 9.2 이번 주 학습이 API 서버에 주는 도움

- socket과 요청/응답 흐름을 이해하면 서버 시작점이 보인다
- HTTP 형식을 이해하면 API 요청 처리 구조를 생각하기 쉽다
- proxy와 병렬 처리 개념은 thread pool 기반 서버 설계와 연결된다

즉, 학습 과제와 수요 코딩회는 따로 노는 것이 아니라,
**작은 서버를 이해한 뒤 더 실제적인 서버 구조로 가는 연결 단계**
라고 볼 수 있다.

# 10. 초심자 기준으로 꼭 구분해야 하는 것

이번 주에는 비슷해 보이는 단어들이 많이 나온다.
아래 차이를 구분하면 흐름이 훨씬 쉬워진다.

## 10.1 IP와 DNS

- IP: 실제로 통신할 컴퓨터의 주소
- DNS: 이름을 IP로 바꾸는 시스템

## 10.2 TCP와 HTTP

- TCP: 연결을 맺고 데이터를 안정적으로 보내는 방식
- HTTP: 요청과 응답의 형식

## 10.3 socket과 HTTP

- socket: 프로그램이 네트워크 통신을 위해 사용하는 입출력 창구
- HTTP: 그 socket 위에서 주고받는 데이터 형식

## 10.4 서버와 프록시

- 서버: 요청을 직접 처리해 응답하는 프로그램
- 프록시: 요청을 중간에서 전달하고 경우에 따라 가공·저장하는 프로그램

---

# 11. 한 번에 보는 전체 요청 흐름

아래 그림을 이해하면 이번 주 큰 흐름은 거의 잡은 것이다.

```text
사용자
-> 브라우저에 URL 입력
-> DNS가 도메인 이름을 IP 주소로 변환
-> 브라우저가 서버의 IP:포트로 TCP 연결
-> 브라우저가 HTTP 요청 전송
-> 서버 프로그램이 socket으로 요청을 읽음
-> 서버가 파일 또는 내부 로직 처리
-> 서버가 HTTP 응답 전송
-> 브라우저가 결과 표시
```

proxy가 중간에 있으면 아래처럼 된다.

```text
브라우저
-> proxy 연결
-> proxy가 요청을 읽음
-> proxy가 원본 서버에 새 연결 생성
-> proxy가 요청 전달
-> 원본 서버 응답 수신
-> proxy가 응답 전달
-> 브라우저가 결과 표시
```

즉, proxy는 요청 흐름을 한 번 더 직접 다루게 만드는 예제다.

# 15. 자주 하는 오해

## 15.1 "HTTP를 배우면 네트워크를 다 이해한 것이다"

아니다.
HTTP는 요청과 응답 형식이고,
그 아래에는 DNS, IP, TCP, socket 같은 층위가 있다.

## 15.2 "socket은 HTTP 전용이다"

아니다.
socket은 네트워크 통신을 위한 일반적인 인터페이스다.
HTTP는 그 위에서 자주 쓰이는 프로토콜 중 하나다.

## 15.3 "서버는 요청을 그냥 함수처럼 바로 받는다"

실제로는 연결을 기다리고,
socket을 만들고,
요청을 읽고,
응답을 쓰는 흐름이 있다.

## 15.4 "proxy는 단순 복사기다"

아니다.
proxy는 중간에서 요청을 전달할 뿐 아니라,
병렬 처리, 캐시, 필터링 같은 확장 포인트를 가진다.

## 15.5 "tiny 웹서버를 이해하면 바로 실무 서버를 이해할 수 있다"

tiny는 교육용 예제다.
하지만 요청/응답 흐름, HTTP 처리, 정적/동적 응답의 큰 그림을 배우기에는 매우 좋다.

# 16. 이번 주 학습의 핵심 정리

- 이번 주는 네트워크 요청을 서버 프로그램이 어떻게 받아 처리하는지를 배우는 주차다.
- `DNS`, `IP`, `TCP`, `socket`, `HTTP`는 각각 다른 역할을 맡지만 하나의 요청 흐름 안에서 연결된다.
- `echo server`는 요청/응답의 가장 단순한 형태를 익히는 단계다.
- `tiny 웹서버`는 socket, HTTP, 정적/동적 응답을 함께 배우는 교육용 예제다.
- `proxy 서버`는 요청 전달, 병렬 처리, 캐시까지 확장되는 단계다.
- 이 학습은 수요 코딩회의 `미니 DBMS API 서버` 구현과 직접 연결된다.
