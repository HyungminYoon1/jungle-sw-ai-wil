## 1. 왜 C 기초를 따로 정리해야 하는가

이번 주 C 학습에서 중요한 점은 "문법을 전부 외우는 것"이 아니다.  
공지 문서에서도 C 언어 전체를 완벽히 익히는 것이 목표가 아니라, 과제를 진행하면서 필요한 개념을 익히는 방향을 권장하고 있다.

그렇다면 무엇을 우선적으로 익혀야 할까.  
자료구조 구현과 SQL 처리기 흐름을 고려하면 특히 아래 주제가 중요하다.

- 변수: 값이 메모리에 어떻게 저장되는지 이해하기 위해
- 함수: 입력과 반환, 호출 흐름을 명확히 하기 위해
- 구조체: 여러 데이터를 하나로 묶어 다루기 위해
- 포인터: 노드 연결, 동적 메모리, 함수 인자 전달을 이해하기 위해
- `malloc` / `free`: 런타임에 필요한 메모리를 직접 관리하기 위해

즉, 이 문서는 "문법 사전"이 아니라 "자료구조와 시스템 프로그래밍으로 연결되는 C 기초"에 초점을 둔다.

## 2. 변수: 값이 저장되는 가장 기본 단위

변수는 데이터를 저장하기 위한 이름 붙은 공간이다.  
하지만 C에서는 단순히 "이름표"로만 생각하면 부족하다. 타입과 메모리 관점까지 같이 봐야 한다.

예시:

```c
int age = 20;
char grade = 'A';
double score = 95.5;
```

### 2.1 타입이 중요한 이유

C는 정적 타입 언어다. 즉, 변수는 선언할 때 타입이 정해지고, 그 타입에 따라 메모리 크기와 해석 방식이 달라진다.

예를 들어:

- `int`: 정수
- `char`: 문자 하나
- `double`: 실수

자료구조 구현에서는 특히 다음 감각이 중요하다.

- 타입이 다르면 메모리 크기도 다를 수 있다.
- 같은 비트 패턴도 타입에 따라 다르게 해석될 수 있다.
- 포인터 연산은 "가리키는 대상 타입 크기"를 기준으로 움직인다.

### 2.2 선언과 초기화

```c
int x;        // 선언
int y = 10;   // 선언 + 초기화
```

초기화되지 않은 지역 변수는 위험하다.  
지역 변수는 자동으로 0이 들어간다고 기대하면 안 된다.

예:

```c
int x;
printf("%d\n", x);
```

이 코드는 쓰레기값을 출력할 수 있다.

### 2.3 상수와 리터럴

```c
const int MAX_SIZE = 100;
```

`const`는 "이 변수 값을 바꾸지 않겠다"는 의도를 코드에 드러낸다.  
코드를 읽는 사람 입장에서 의미가 분명해지고, 실수도 줄어든다.

## 3. 연산자와 표현식

C에서는 연산자 우선순위와 타입 변환 때문에 예상과 다른 결과가 나올 수 있다.

예:

```c
int a = 5;
int b = 2;
printf("%d\n", a / b);
```

결과는 `2`다.  
정수 나눗셈이기 때문에 소수점 이하는 버려진다.

이런 감각은 나중에 인덱스 계산, 크기 계산, 포인터 오프셋 계산에서 중요하다.

### 3.1 자주 쓰는 연산자

- 산술: `+`, `-`, `*`, `/`, `%`
- 비교: `==`, `!=`, `<`, `>`, `<=`, `>=`
- 논리: `&&`, `||`, `!`
- 대입: `=`
- 주소 관련: `&`, `*`

특히 `=`와 `==`를 헷갈리지 않도록 주의해야 한다.

```c
if (x = 0) {
    ...
}
```

이 코드는 비교가 아니라 대입이다.  
많이 나오는 실수다.

## 4. 함수: 일을 나누고 흐름을 분리하는 도구

함수는 하나의 작업을 이름 붙여 분리한 것이다.

예시:

```c
int add(int a, int b) {
    return a + b;
}
```

구성 요소:

- 반환 타입: `int`
- 함수 이름: `add`
- 매개변수: `int a, int b`
- 반환값: `a + b`

### 4.1 함수 선언과 정의

선언:

```c
int add(int a, int b);
```

정의:

```c
int add(int a, int b) {
    return a + b;
}
```

헤더 파일에는 보통 선언을 두고, `.c` 파일에는 정의를 둔다.

예:

- `list.h`: 함수 원형 선언
- `list.c`: 함수 실제 구현

이 분리는 자료구조 프로젝트에서 매우 중요하다.

### 4.2 값 전달의 의미

C 함수는 기본적으로 값을 복사해서 전달한다.

```c
void change(int x) {
    x = 100;
}
```

```c
int main(void) {
    int a = 10;
    change(a);
    printf("%d\n", a); // 여전히 10
}
```

이유는 `a`의 값이 복사되어 `x`로 전달되기 때문이다.

그래서 원본을 바꾸고 싶다면 포인터를 사용해야 한다.

```c
void change(int *x) {
    *x = 100;
}
```

```c
int main(void) {
    int a = 10;
    change(&a);
    printf("%d\n", a); // 100
}
```

이 개념은 연결 리스트 함수에서 특히 중요하다.

예:

- 노드 값을 수정하기
- 헤드 포인터를 갱신하기
- 구조체 내부 필드를 바꾸기

### 4.3 함수를 나누는 기준

좋은 함수는 보통 역할이 분명하다.

예를 들어 SQL 처리기라면:

- 입력 읽기
- SQL 파싱
- 명령 실행
- 파일 저장

자료구조라면:

- 생성
- 삽입
- 삭제
- 탐색
- 출력

이렇게 나누면 디버깅도 쉬워지고 테스트도 쉬워진다.

## 5. 배열과 문자열 기초

자료구조 구현과 SQL 처리기에서는 배열과 문자열을 피할 수 없다.

### 5.1 배열

```c
int numbers[5] = {1, 2, 3, 4, 5};
```

배열의 특징:

- 같은 타입의 데이터를 연속된 메모리에 저장
- 인덱스로 접근
- 범위를 넘어가도 C가 자동으로 막아주지 않음

예:

```c
numbers[10] = 99;
```

이 코드는 매우 위험하다.  
실행은 될 수 있지만 다른 메모리를 망가뜨릴 수 있다.

### 5.2 문자열

C에서 문자열은 보통 `char` 배열로 표현한다.

```c
char name[] = "jungle";
```

문자열 끝에는 보이지 않는 종료 문자 `'\0'`가 들어간다.

즉, 메모리에는 실제로 다음처럼 들어 있다.

```text
'j' 'u' 'n' 'g' 'l' 'e' '\0'
```

이 종료 문자가 없으면 문자열 함수들이 어디까지 읽어야 하는지 알 수 없어서 위험해진다.

SQL 파서나 파일 입력 처리에서는 문자열 끝 처리와 버퍼 크기 관리가 매우 중요하다.

## 6. 구조체: 관련 있는 데이터를 묶는 방법

구조체는 서로 관련 있는 여러 값을 하나의 타입으로 묶는다.

예시:

```c
typedef struct {
    int id;
    char name[32];
} User;
```

이제 `User`는 `id`와 `name`을 함께 가지는 하나의 사용자 정의 타입이다.

사용 예시:

```c
User user;
user.id = 1;
```

### 6.1 왜 구조체가 중요한가

자료구조 구현에서는 한 개의 값만 다루는 경우가 드물다.

예:

- 연결 리스트 노드: 값 + 다음 노드 주소
- 트리 노드: 값 + 왼쪽 자식 + 오른쪽 자식
- SQL 명령 구조: 명령 종류 + 테이블 이름 + 컬럼 목록 + 값 목록

이런 정보는 구조체 없이는 다루기 매우 불편하다.

### 6.2 구조체 안에 구조체, 구조체 포인터

예를 들어 연결 리스트 노드는 아래처럼 생긴다.

```c
typedef struct Node {
    int value;
    struct Node *next;
} Node;
```

핵심:

- `value`: 현재 노드가 가진 데이터
- `next`: 다음 노드를 가리키는 포인터

즉, 자료구조는 "값"과 "연결 정보"를 함께 다루는 구조다.

## 7. 포인터: C 학습의 핵심

포인터는 메모리 주소를 저장하는 변수다.

예:

```c
int x = 10;
int *p = &x;
```

의미:

- `x`: 정수 값 10을 저장
- `&x`: `x`의 주소
- `p`: 그 주소를 저장하는 포인터

### 7.1 `&` 와 `*`

`&`:

- 어떤 변수의 주소를 구한다

`*`:

- 포인터가 가리키는 곳의 값을 읽거나 쓴다

예:

```c
int x = 10;
int *p = &x;

printf("%d\n", *p); // 10
*p = 20;
printf("%d\n", x);  // 20
```

즉, `p`를 통해 `x`를 간접적으로 조작할 수 있다.

### 7.2 포인터를 왜 써야 하는가

포인터가 필요한 대표 상황:

- 원본 데이터를 함수 안에서 수정해야 할 때
- 큰 구조체를 복사하지 않고 전달하고 싶을 때
- 동적 메모리를 사용할 때
- 연결 리스트, 트리처럼 노드 연결이 필요할 때

### 7.3 포인터 선언 읽는 법

```c
int *p;
char *name;
Node *head;
```

읽는 방식:

- `p`는 `int`를 가리키는 포인터
- `name`은 `char`를 가리키는 포인터
- `head`는 `Node`를 가리키는 포인터

포인터 자체의 타입이 왜 중요한가:

- 역참조 시 어떤 크기와 구조로 읽을지 결정된다
- 포인터 연산 단위가 달라진다

### 7.4 포인터와 구조체 접근

구조체 변수일 때:

```c
user.id
```

구조체 포인터일 때:

```c
user_ptr->id
```

`->`는 사실 `(*user_ptr).id`의 축약이다.

자료구조 구현에서 이 문법은 계속 나온다.

### 7.5 포인터 실수 예시

#### 1) 초기화하지 않은 포인터

```c
int *p;
*p = 10;
```

`p`가 어디를 가리키는지 정해지지 않았기 때문에 매우 위험하다.

#### 2) `NULL` 확인 없이 역참조

```c
Node *node = NULL;
printf("%d\n", node->value);
```

실행 중 프로그램이 죽을 수 있다.

#### 3) 포인터와 값 혼동

```c
int x = 10;
int *p = &x;

printf("%d\n", p);   // 잘못된 형식의 출력
printf("%d\n", *p);  // 값 출력
```

주소를 출력할 때는 보통 `%p`를 쓴다.

```c
printf("%p\n", (void *)p);
```

## 8. 이중 포인터가 필요한 순간

초반에는 어려워 보이지만, 특정 상황에서는 꼭 필요하다.

예를 들어 함수 안에서 `head` 자체를 바꾸고 싶다고 하자.

```c
void push_front(Node **head, int value);
```

왜 `Node *`가 아니라 `Node **`일까.

- `Node *head`는 "노드를 가리키는 포인터"
- `Node **head`는 "그 포인터 변수 자체의 주소"

즉, 함수 안에서 호출자의 `head`를 바꾸려면 `head`의 주소를 받아야 한다.

이 개념은 연결 리스트 첫 노드 삽입, 삭제 후 헤드 갱신에서 자주 필요하다.

## 9. 동적 메모리 할당: `malloc`

C에서는 실행 중 필요한 메모리를 직접 요청할 수 있다.  
이때 대표적으로 쓰는 함수가 `malloc`이다.

예:

```c
Node *node = malloc(sizeof(Node));
```

의미:

- `Node` 하나를 저장할 수 있는 크기만큼 메모리를 힙에서 요청
- 성공하면 그 메모리의 시작 주소 반환
- 실패하면 `NULL` 반환 가능

그래서 보통 바로 `NULL` 체크를 하는 습관이 좋다.

```c
Node *node = malloc(sizeof(Node));
if (node == NULL) {
    return;
}
```

### 9.1 왜 동적 메모리가 필요한가

자료구조에서는 실행 전에 노드 개수를 미리 알기 어려운 경우가 많다.

예:

- 연결 리스트에 몇 개를 넣을지 모름
- 트리가 얼마나 커질지 모름
- SQL 입력 결과 컬럼 개수나 값 개수가 실행 중 정해질 수 있음

이럴 때 실행 중에 필요한 만큼 메모리를 만들어야 한다.

### 9.2 `sizeof`를 같이 쓰는 이유

```c
malloc(sizeof(Node))
```

처럼 쓰면 타입 크기가 바뀌어도 안전하게 대응할 수 있다.  
매직 넘버로 바이트 수를 직접 쓰는 것보다 훨씬 낫다.

---

## 10. 메모리 해제: `free`

`malloc`으로 할당한 메모리는 사용이 끝나면 직접 반납해야 한다.

```c
free(node);
```

왜 중요한가:

- 해제하지 않으면 메모리 누수(memory leak)가 생긴다.
- 프로그램이 커질수록 낭비가 누적된다.
- 장시간 실행되는 프로그램에서는 더 치명적이다.

### 10.1 해제 후 포인터 처리

보통 아래처럼 처리하는 습관이 좋다.

```c
free(node);
node = NULL;
```

이유:

- 이미 해제한 메모리를 다시 참조하는 실수를 줄일 수 있다.

### 10.2 자주 하는 `free` 실수

#### 1) 해제한 메모리 다시 사용

```c
free(node);
printf("%d\n", node->value);
```

위험하다.  
이 메모리는 더 이상 유효하지 않다.

#### 2) 같은 메모리를 두 번 해제

```c
free(node);
free(node);
```

이 역시 심각한 오류를 만들 수 있다.

#### 3) 스택 메모리를 `free`하려고 함

```c
Node node;
free(&node);
```

이건 잘못이다.  
`free`는 `malloc`, `calloc`, `realloc` 등으로 동적 할당한 메모리에만 써야 한다.

## 11. 스택과 힙 관점으로 보기

초반 C 학습에서 꼭 잡아두면 좋은 관점이다.

### 11.1 스택(stack)

대체로 함수 호출과 함께 자동으로 관리되는 메모리 영역이다.

예:

```c
void foo(void) {
    int x = 10;
}
```

여기서 `x`는 보통 스택 영역에 놓인다.  
함수가 끝나면 수명이 끝난다.

### 11.2 힙(heap)

`malloc`으로 직접 요청하는 메모리 영역이다.

예:

```c
int *p = malloc(sizeof(int));
```

힙 메모리는 함수가 끝나도 자동으로 사라지지 않는다.  
직접 `free`해야 한다.

### 11.3 왜 이 구분이 중요한가

자료구조 노드를 함수 안에서 만들고 함수 밖에서도 계속 쓰고 싶다면, 스택 지역 변수로는 부족한 경우가 많다.

예:

```c
Node *create_node(int value) {
    Node node;
    node.value = value;
    return &node; // 잘못된 예
}
```

이 코드는 함수가 끝나면 `node`의 수명이 끝나므로 위험하다.

올바른 방식은 동적 할당이다.

```c
Node *create_node(int value) {
    Node *node = malloc(sizeof(Node));
    if (node == NULL) {
        return NULL;
    }

    node->value = value;
    node->next = NULL;
    return node;
}
```

## 12. 자료구조 관점에서 포인터와 메모리 이해하기

### 12.1 연결 리스트

연결 리스트는 노드들이 포인터로 이어진 구조다.

```c
typedef struct Node {
    int value;
    struct Node *next;
} Node;
```

핵심 질문:

- 현재 노드는 어떤 값을 가지는가
- 다음 노드는 어디를 가리키는가
- 마지막 노드의 `next`는 `NULL`인가

즉, 연결 리스트는 "값의 나열"이 아니라 "주소의 연결"로 이해해야 한다.

### 12.2 트리

트리는 각 노드가 여러 자식을 가질 수 있다.  
이진 트리라면 보통 왼쪽, 오른쪽 포인터 두 개를 가진다.

```c
typedef struct TreeNode {
    int value;
    struct TreeNode *left;
    struct TreeNode *right;
} TreeNode;
```

자료구조 문제를 풀 때는 값 비교보다도 "포인터 연결이 올바른가"를 자주 점검해야 한다.

### 12.3 SQL 처리기 구조와도 연결됨

SQL 처리기에서 파싱 결과를 구조체로 담는다고 생각해 보자.

예:

```c
typedef struct {
    int type;
    char *table_name;
    char **columns;
    int column_count;
} SqlCommand;
```

이 경우도 핵심은 같다.

- 어떤 데이터가 어디에 저장되는가
- 문자열은 복사했는가, 주소만 들고 있는가
- 언제 해제해야 하는가

즉, 포인터와 메모리 수명 개념은 자료구조 과제와 SQL 처리기 모두에 공통으로 적용된다.

## 13. 함수, 구조체, 포인터를 함께 보는 예제

아래는 연결 리스트 앞쪽 삽입의 축약 예시다.

```c
typedef struct Node {
    int value;
    struct Node *next;
} Node;

Node *create_node(int value) {
    Node *node = malloc(sizeof(Node));
    if (node == NULL) {
        return NULL;
    }

    node->value = value;
    node->next = NULL;
    return node;
}

int push_front(Node **head, int value) {
    Node *new_node = create_node(value);
    if (new_node == NULL) {
        return 0;
    }

    new_node->next = *head;
    *head = new_node;
    return 1;
}
```

이 예제에서 함께 봐야 할 포인트:

- 구조체 `Node`가 데이터와 연결 정보를 묶는다.
- `create_node`는 힙에 새 노드를 만든다.
- `push_front`는 호출자의 `head`를 바꿔야 하므로 `Node **head`를 받는다.
- 새 노드의 `next`는 이전 헤드를 가리킨다.
- 마지막에는 `free`로 정리할 책임이 생긴다.

이 한 예제 안에 이번 주 핵심 문법이 거의 다 들어 있다.

## 14. C 학습에서 자주 나오는 오해

### 오해 1. 포인터는 그냥 어려운 문법이다

포인터는 문법 장식이 아니라 "메모리 주소를 다루는 방법"이다.  
자료구조 구현에서는 거의 필수 개념이다.

### 오해 2. `malloc`만 알면 된다

중요한 것은 `malloc` 자체보다도 메모리 수명 관리다.

질문:

- 누가 할당했는가
- 누가 해제할 책임이 있는가
- 언제까지 유효한가

### 오해 3. 구조체는 객체지향 클래스 같은 것이다

비슷하게 묶는 역할은 있지만, C 구조체는 기본적으로 데이터 묶음이다.  
동작은 별도의 함수로 분리하는 경우가 많다.

### 오해 4. 컴파일만 되면 일단 괜찮다

C는 런타임 메모리 오류가 매우 많다.  
포인터, 배열, 해제된 메모리 사용 문제는 실행 후에 드러나는 경우가 많다.

### 변수

- 이 값의 타입은 무엇인가
- 몇 바이트인지 대략 감이 오는가
- 초기화되었는가

### 함수

- 입력은 무엇인가
- 반환값은 무엇인가
- 원본을 바꾸는가, 복사본만 다루는가

### 구조체

- 이 구조체는 어떤 정보를 묶고 있는가
- 각 필드는 어떤 역할을 하는가

### 포인터

- 이 변수는 값인가 주소인가
- 역참조가 안전한가
- `NULL` 가능성이 있는가

### 메모리

- 이 메모리는 스택인가 힙인가
- 누가 할당했는가
- 언제 해제해야 하는가

## 18. 꼭 기억할 핵심 문장

- 변수는 이름표일 뿐 아니라 타입과 메모리 해석 방식을 함께 가진다.
- 함수는 값을 복사해서 받으므로 원본 수정이 필요하면 포인터가 필요하다.
- 구조체는 관련 있는 데이터를 묶어 자료구조와 명령 객체를 표현하는 데 유용하다.
- 포인터는 주소를 저장하며, C 자료구조의 핵심 도구다.
- `malloc`은 실행 중 필요한 메모리를 힙에 할당하고, `free`는 그 메모리를 반납한다.
- 자료구조 구현의 많은 버그는 문법보다 포인터 연결과 메모리 수명 관리에서 나온다.
