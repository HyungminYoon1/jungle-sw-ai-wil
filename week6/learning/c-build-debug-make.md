## 2. 왜 Linux 환경과 gcc를 먼저 익혀야 하는가

자료구조 과제는 단순히 코드를 쓰는 것으로 끝나지 않는다. 작성한 코드를 직접 빌드하고, 실행하고, 출력 결과를 확인하고, 필요하면 다시 수정하는 반복이 계속된다.

이때 아래 능력이 필요하다.

- 현재 어느 디렉터리에서 작업 중인지 확인하기
- 소스 파일이 어디 있는지 찾기
- 컴파일 명령을 실행하기
- 실행 파일을 실행하기
- 에러 메시지를 읽고 다시 수정하기

즉, 과제 수행의 가장 기본 단위는 아래 흐름이다.

```text
파일 위치 확인 -> 소스 수정 -> gcc로 컴파일 -> 실행 -> 결과 확인 -> 다시 수정
```

이 흐름이 익숙하지 않으면 포인터나 자료구조 개념을 알아도 과제를 밀고 나가기가 어렵다.

## 3. Ubuntu/Linux 환경이란 무엇인가

이번 주 공지에서는 `Ubuntu 22.04 LTS (x86_64)` 이상 환경을 사용하라고 안내한다.  
Ubuntu는 Linux 배포판 중 하나이고, 터미널 명령을 통해 파일과 프로그램을 다루는 방식이 매우 중요하다.

Windows에 익숙한 사람이라면 처음에는 다음 차이가 낯설 수 있다.

- 경로 구분에 `/`를 사용한다.
- 실행 파일을 현재 폴더에서 실행할 때 `./program`처럼 입력한다.
- 터미널에서 파일 탐색과 실행을 많이 한다.
- 빌드 도구와 컴파일러 사용이 명령행 중심이다.

이번 주에는 Linux 자체를 깊게 공부하는 것이 목표는 아니지만, 최소한 "과제를 수행할 수 있는 정도"의 명령 사용은 익혀두는 편이 좋다.

## 4. 가장 먼저 익힐 Linux 기본 명령

### 4.1 현재 위치 확인: `pwd`

```bash
pwd
```

의미:

- 현재 내가 어느 디렉터리에 있는지 출력한다.

처음에는 "지금 내가 어디에서 명령을 치고 있는지" 감이 없는 경우가 많다.  
그럴 때 가장 먼저 `pwd`를 보는 습관이 유용하다.

### 4.2 파일 목록 보기: `ls`

```bash
ls
```

자주 쓰는 형태:

```bash
ls -l
ls -a
```

의미:

- `ls`: 현재 폴더 파일 목록 보기
- `ls -l`: 자세한 정보까지 보기
- `ls -a`: 숨김 파일까지 보기

### 4.3 디렉터리 이동: `cd`

```bash
cd data-structures
cd ..
cd ~
```

의미:

- `cd data-structures`: 해당 폴더로 이동
- `cd ..`: 상위 폴더로 이동
- `cd ~`: 홈 디렉터리로 이동

### 4.4 파일 내용 보기: `cat`

```bash
cat main.c
```

짧은 파일을 빠르게 확인할 때 유용하다.

### 4.5 폴더 만들기: `mkdir`

```bash
mkdir build
```

### 4.6 파일 복사와 이동: `cp`, `mv`

```bash
cp main.c backup_main.c
mv old.c new.c
```

### 4.7 파일 삭제: `rm`

```bash
rm main
rm *.o
```

주의:

- Linux의 `rm`은 휴지통으로 가지 않고 바로 삭제된다.
- 특히 `rm -r`, `rm -rf`는 매우 조심해서 써야 한다.

이번 주 학습에서는 `pwd`, `ls`, `cd`, `cat` 정도만 먼저 익혀도 과제 진행에 큰 도움이 된다.

## 5. C 프로젝트에서 자주 보는 파일들

자료구조 과제나 C 프로젝트에서는 주로 아래 파일들을 보게 된다.

- `.c`: 실제 구현 파일
- `.h`: 함수 선언, 구조체 선언 등을 담는 헤더 파일
- 실행 파일: 컴파일 결과물
- 경우에 따라 `Makefile`: 빌드 규칙 파일

예시:

```text
main.c
list.c
list.h
stack.c
stack.h
```

보통 역할은 다음과 같이 나뉜다.

- `main.c`: 프로그램 시작점
- `list.c`: 연결 리스트 함수 구현
- `list.h`: 연결 리스트 관련 선언

이 구조를 이해해야 `gcc`로 여러 파일을 함께 빌드하는 이유가 보인다.

## 6. `gcc`란 무엇인가

`gcc`는 GNU Compiler Collection의 C 컴파일러다.  
쉽게 말하면 `.c` 파일을 실행 가능한 프로그램으로 바꾸는 도구다.

가장 단순한 예시는 아래와 같다.

```c
#include <stdio.h>

int main(void) {
    printf("Hello, C!\n");
    return 0;
}
```

이 파일이 `main.c`라면, 아래처럼 컴파일할 수 있다.

```bash
gcc main.c -o main
```

의미:

- `gcc`: 컴파일러 실행
- `main.c`: 입력 파일
- `-o main`: 출력 실행 파일 이름을 `main`으로 지정

그리고 실행은 보통 이렇게 한다.

```bash
./main
```

Linux에서는 현재 디렉터리의 프로그램을 실행할 때 `./`를 붙이는 점이 중요하다.

## 7. `gcc` 기본 명령부터 익히기

### 7.1 한 파일 컴파일

```bash
gcc main.c -o main
```

가장 먼저 익혀야 할 형태다.

### 7.2 경고를 함께 보기

```bash
gcc -Wall -Wextra main.c -o main
```

의미:

- `-Wall`: 자주 발생하는 중요한 경고 표시
- `-Wextra`: 추가 경고 표시

C는 "컴파일은 되지만 위험한 코드"도 허용하는 편이라, 경고를 함께 보는 습관이 중요하다.

### 7.3 디버깅 정보를 넣어 컴파일하기

```bash
gcc -g main.c -o main
```

이 옵션은 나중에 `gdb` 같은 디버거를 사용할 때 도움이 된다.  
다만 이번 문서의 핵심은 디버깅 자체보다 `gcc` 사용법이므로, 일단 "`-g`는 디버깅용 정보 추가" 정도로 이해해도 충분하다.

### 7.4 자주 쓰는 학습용 조합

```bash
gcc -Wall -Wextra -g main.c -o main
```

이번 주처럼 학습과 실습 중심일 때 무난한 조합이다.

## 8. 여러 파일을 함께 컴파일하는 이유

자료구조 과제는 파일 하나로 끝나지 않는 경우가 많다.

예를 들어:

- `main.c`
- `list.c`
- `list.h`

이렇게 구성되어 있다면 보통 아래처럼 컴파일한다.

```bash
gcc main.c list.c -o app
```

여기서 중요한 점은 헤더 파일 자체를 직접 컴파일하는 것이 아니라, 구현이 들어 있는 `.c` 파일들을 함께 넣는다는 것이다.

### 8.1 왜 헤더 파일이 필요한가

예:

```c
// list.h
void list_push_back(int value);
```

```c
// list.c
void list_push_back(int value) {
    ...
}
```

`main.c`는 `list.h`를 보고 "이 함수가 있구나"를 알 수 있고, 실제 구현은 `list.c`에서 찾는다.

### 8.2 구현 파일을 빼먹으면 어떤 일이 생기나

예를 들어 `main.c`에서 `list_push_back()`을 호출하는데 `list.c`를 컴파일 대상에 넣지 않으면, 보통 링크 에러가 발생한다.

예시 메시지:

```text
undefined reference to `list_push_back`
```

이 메시지는 대개 "선언은 알겠는데 실제 구현을 찾지 못했다"는 뜻이다.

## 9. 컴파일, 오브젝트 파일, 링크를 아주 간단히 이해하기

초반에는 아래 정도로만 이해해도 충분하다.

1. `.c` 파일을 컴파일한다.
2. 각 파일은 중간 결과물로 바뀔 수 있다.
3. 마지막에 여러 결과물을 묶어 실행 파일을 만든다.

직접 나눠서 보면 아래와 같다.

```bash
gcc -c main.c
gcc -c list.c
gcc main.o list.o -o app
```

의미:

- `-c`: 실행 파일까지 만들지 않고 오브젝트 파일까지만 생성
- `main.o`, `list.o`: 중간 결과물
- 마지막 줄: 오브젝트 파일들을 묶어 최종 실행 파일 생성

이번 주에는 이 과정을 깊게 외울 필요는 없지만, 링크 에러를 이해하는 데 도움이 된다.

## 10. 실행 파일 실행하기

Linux에서 현재 폴더에 있는 프로그램은 보통 아래처럼 실행한다.

```bash
./app
```

명령행 인자가 있다면 이렇게 실행할 수 있다.

```bash
./app input.txt
```

즉:

- `app`은 프로그램 이름
- `input.txt`는 프로그램에 전달되는 인자

이 감각은 나중에 SQL 처리기처럼 파일 인자를 받는 프로그램을 만들 때도 그대로 이어진다.

## 11. 컴파일 에러와 링크 에러를 구분해서 보기

### 11.1 컴파일 에러

소스 코드 문법이나 타입 문제 때문에 실행 파일을 만들 수 없는 경우다.

예:

```c
int x = 10
printf("%d\n", x);
```

가능한 메시지:

```text
error: expected ';' before 'printf'
```

이 경우는 문법 수준 문제다.

### 11.2 링크 에러

각 파일을 따로는 처리했지만, 마지막에 필요한 함수 구현을 찾지 못하는 경우다.

예:

```text
undefined reference to `push`
```

점검 질문:

- 구현 파일을 컴파일 명령에 포함했는가
- 함수 이름 철자가 일치하는가
- 선언과 구현이 같은 형태인가

이 둘을 구분하면 에러를 훨씬 빨리 고칠 수 있다.

## 12. 자주 보는 `gcc` 관련 문제

### 12.1 세미콜론 누락

가장 흔한 문법 오류다.

### 12.2 헤더 파일 경로 문제

예:

```text
fatal error: list.h: No such file or directory
```

이 경우:

- 파일 위치를 확인하거나
- 필요하면 include 경로를 추가해야 한다.

예:

```bash
gcc -Iinclude main.c list.c -o app
```

### 12.3 함수 선언과 구현 불일치

예를 들어 헤더와 구현의 매개변수 형태가 다르면 경고나 오류가 생길 수 있다.

### 12.4 구현 파일 누락

`main.c`만 컴파일하고 `list.c`를 빼먹는 경우가 대표적이다.

### 12.5 경고를 무시함

처음에는 실행만 되면 넘어가고 싶지만, C에서는 경고가 실제 버그로 이어지는 경우가 많다.

## 14. 자료구조 과제에서 Linux/gcc 관점으로 특히 중요한 것

이번 주 과제는 단순 사용이 아니라 `기본 자료구조를 직접 구현`하는 데 목적이 있다.  
즉, 이미 만들어진 프로그램을 실행하는 것이 아니라, 스스로 코드를 수정하고 빌드하고 검증해야 한다.

그래서 아래가 중요하다.

- 어떤 파일을 고쳐야 하는지 찾기
- 변경 후 다시 컴파일하기
- 경고와 에러 메시지 읽기
- 실행 결과가 맞는지 직접 확인하기

공지 문서에도 이번 주 C 과제에는 별도의 테스트케이스가 없으므로, 다양한 입력을 넣어 스스로 확인하라고 되어 있다.  
즉, `gcc 사용법`은 단순 도구 사용이 아니라, 직접 검증하며 구현을 밀고 가기 위한 기반이다.

## 15. 과제 진행 순서를 공지와 연결해서 보기

공지 추천 순서는 아래와 같다.

```text
C언어 공부
-> Linux/gcc 사용법 익히기
-> C Pointer(&, * 연산자) 공부
-> malloc/free 사용법 파악
-> 과제 구현
```

이 순서를 실천 가능한 형태로 바꾸면 아래처럼 볼 수 있다.

1. Linux에서 파일 이동과 확인 명령을 익힌다.
2. `gcc main.c -o main` 수준의 기본 컴파일을 익힌다.
3. 경고를 보며 간단한 수정과 재컴파일을 반복한다.
4. 여러 파일을 함께 빌드하는 방법을 익힌다.
5. 그 다음 포인터와 `malloc/free`를 학습하며 자료구조 구현으로 넘어간다.

즉, `gcc`는 별도 이론 과목이 아니라 "포인터와 자료구조 실습으로 들어가기 전 준비 단계"로 이해하면 된다.

---

## 16. 추가로 알아두면 좋은 내용 1: 아주 기초적인 디버깅 감각

이번 문서의 핵심 목표는 아니지만, 최소한 아래 정도는 알고 있으면 좋다.

- 에러 메시지가 난 줄만 보지 말고 바로 위 코드도 같이 보기
- 프로그램이 죽으면 입력과 수정 지점을 다시 확인하기
- `printf`로 중간 값 출력해 보기

예:

```c
printf("value=%d\n", value);
```

포인터를 다루기 시작하면 나중에 `gdb` 같은 디버거가 더 중요해지지만, 이번 주 초반에는 일단 `컴파일 -> 실행 -> 출력 확인 -> 수정` 루프에 익숙해지는 것이 더 우선이다.

---

## 17. 추가로 알아두면 좋은 내용 2: `make`는 무엇인가

`make`는 여러 파일을 빌드할 때 반복 명령을 줄여 주는 도구다.

예를 들어 아래 명령을 자주 반복한다고 하자.

```bash
gcc -Wall -Wextra -g main.c list.c stack.c -o app
```

파일이 많아질수록 이 명령은 길어지고 실수하기 쉬워진다.  
이럴 때 `Makefile`에 규칙을 적어 두면 `make` 한 번으로 빌드할 수 있다.

간단한 예:

```make
CC = gcc
CFLAGS = -Wall -Wextra -g
TARGET = app

all:
	$(CC) $(CFLAGS) main.c list.c stack.c -o $(TARGET)
```

실행:

```bash
make
```

다만 이번 주 공지의 직접 학습 목표는 `Linux/gcc 사용법`이므로, `make`는 우선순위가 더 높은 핵심 주제라기보다 "작업이 늘어날 때 도움이 되는 자동화 도구" 정도로 이해하면 충분하다.

---

## 18. 꼭 기억할 핵심 문장

- 이번 주 공지 기준의 직접 학습 목표는 `Linux/gcc 사용법`이다.
- Ubuntu/Linux 환경에서는 터미널 명령으로 파일 위치 확인, 이동, 실행을 자주 하게 된다.
- `gcc main.c -o main`은 가장 기본적인 C 컴파일 명령이다.
- 여러 파일 프로젝트에서는 필요한 `.c` 파일을 함께 컴파일해야 한다.
- `undefined reference`는 대개 링크 단계에서 구현을 못 찾았다는 뜻이다.
- `make`는 유용하지만, 이번 문서에서는 핵심 목표가 아니라 보조 도구다.

## 19. 체크리스트

- `pwd`, `ls`, `cd` 명령의 역할을 설명할 수 있다.

- `undefined reference` 메시지가 왜 나오는지 설명할 수 있다.
