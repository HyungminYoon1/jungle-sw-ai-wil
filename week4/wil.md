## 2. 시도한 접근 방식
- 알고리즘 문제는 무작정 풀이부터 시작하기보다, 먼저 트리와 그래프가 어떤 구조인지, BFS와 DFS가 어떤 상황에서 자연스러운 선택인지 개념을 정리한 뒤 문제에 적용하려고 했다.
- basic 문제를 풀 때는 입력과 출력 형식만 보는 것이 아니라, 문제를 어떤 자료구조로 표현해야 하는지 먼저 생각하려고 했다. 트리 문제는 노드 간 부모-자식 관계, 그래프 문제는 연결 관계와 방문 여부를 기준으로 정리한 뒤 코드를 작성했다.
- BFS와 DFS는 둘 다 탐색 알고리즘이지만, 큐를 이용해 가까운 것부터 넓게 확인할지, 재귀나 스택을 이용해 한 방향으로 깊게 들어갈지를 문제별로 비교해 보며 선택했다.
- 위상정렬은 단순히 알고리즘 이름을 외우기보다, "선후관계가 있는 작업을 어떤 순서로 처리해야 하는가"라는 관점으로 이해하려고 했다.
- 수요 코딩회 준비에서는 React 사용법 자체보다, 브라우저 DOM이 왜 트리 구조인지, Virtual DOM을 왜 별도 데이터 구조로 두는지, Diff가 왜 필요한지부터 문서로 정리했다.
- Virtual DOM 구현은 실제 DOM을 바로 조작하는 방식과 비교하면서 이해했다. 먼저 DOM을 순회해 자바스크립트 객체 트리로 바꾸고, 그다음 이전 트리와 새 트리를 비교해 변경점만 실제 DOM에 반영하는 흐름을 중심으로 학습했다.
- 팀 협업에서는 README와 설명 문장을 같이 챙기는 방식을 의식했다. 단순히 기능을 구현하는 것에서 멈추지 않고, 발표할 때 핵심 로직을 설명할 수 있을 정도로 구조와 테스트 포인트를 정리하려고 했다.


## 3. 문제와 해결 과정

### 알고리즘 유형을 문제마다 새로 해석해야 했던 점
- **문제:** 이번 주에는 트리, 그래프, BFS, DFS, 위상정렬처럼 서로 연결된 개념이 많았는데, 처음에는 각 개념이 따로따로 존재하는 것처럼 느껴져 문제를 볼 때마다 어떤 방식을 선택해야 할지 헷갈렸다.
- **해결:** 문제를 읽고 바로 코드를 쓰기보다, 먼저 "이 입력은 트리인가 그래프인가", "순서가 중요한가", "가까운 것부터 봐야 하나, 끝까지 내려가야 하나"를 질문하는 습관을 들였다. 이 기준이 생기니 BFS/DFS 선택이나 방문 처리 방식이 조금 더 선명해졌다.

### 트리와 그래프 개념이 프론트엔드 과제와 연결되지 않았던 점
- **문제:** 수요 코딩회 과제는 알고리즘 문제에서 접한 트리 구조와 순회 알고리즘을 적용하는 단계라는 것을 인지하고 있었고, 검색을 통해 DFS를 적용해야 한다는 것은 알았으나, 어째서 DFS가 제일 효율적인 알고리즘인지는 정확히 알지 못한 채 깊이있는 질문을 던지지 않고 코드를 구현했다.
- **해결:** React 학습 문서와 Virtual DOM 구현 가이드를 다시 보면서, DFS가 항상 가장 효율적인 알고리즘이라서 선택된 것이 아니라 DOM과 Virtual DOM이 부모-자식으로 연결된 트리 구조이기 때문에 현재 노드를 처리한 뒤 자식 노드를 재귀적으로 내려가며 순회하기에 가장 자연스러운 방식이라는 점을 이해했다. 또한 BFS로 구현할 경우 레벨 순회는 가능하지만, DOM 변환과 Diff의 부모-자식 처리 흐름과는 다소 맞지 않고 구현이 더 장황해질 수 있어 이번 과제에는 적합하지 않다는 점도 정리할 수 있었다.

### Diff를 'React 내부의 어려운 기술' 정도로만 생각했던 점
- **문제:** 처음에는 Diff 알고리즘을 React 내부에서 자동으로 처리되는 복잡한 구현이라고만 생각해서, 내가 직접 설계하고 설명할 수 있는 범위로 내려오지 않았다.
- **해결:** 학습용 구현에서는 Create, Remove, Replace, Text Update, Props Update처럼 설명 가능한 핵심 케이스로 범위를 줄이는 것이 중요하다는 점을 배웠다. React 전체를 복제하려 하지 않고, "무엇이 바뀌었는지 찾고 필요한 부분만 반영한다"는 핵심 원리를 기준으로 단순화해 보니 구조가 훨씬 이해되었다.

### 실제 DOM 조작이 왜 부담이 큰지 체감이 부족했던 점
- **문제:** Virtual DOM이 왜 필요한지 대략 알고 있었지만, 그냥 Diff 알고리즘이 DOM을 직접 바꾸어도 되는 것 아닌가 하는 의문이 있었다.
- **해결:** 문서를 보며 실제 DOM은 브라우저 렌더링 파이프라인과 연결되어 있고, 변경이 많아질수록 reflow와 repaint 비용이 발생한다는 점을 다시 정리했다. 그 뒤 Virtual DOM이 단순한 '가짜 DOM'이 아니라, 변경 사항을 먼저 계산하기 위한 중간 표현이라는 점을 더 분명히 이해하게 되었다.

### AI 활용에서 '코드 생성'보다 '검증 질문'이 더 중요하다는 점
- **문제:** 지난주에 AI에게 구현 방법을 물어보면 빠르게 답을 받을 수는 있었지만, AI가 작성한 코드를 내가 설명할 수 없고, 핵심 개념에 대한 이해도 부족했다는 문제가 있었다.
- **해결:** 이번 주에는 AI에게 가상 DOM + Diff 알고리즘을 이용한 리액트 같은 렌더러의 최소 구현을 요청하였고, 이어서 시스템 기능을 개선하기보다는 코드를 하나하나 확인하며 "이 모듈이 왜 필요하고 어떻게 사용되는지"를 검증하는 질문을 중심으로 학습했다. 이 방식이 속도는 조금 느릴 수 있어도, 결과적으로 리액트를 이해하고 발표를 준비하는 데 더 도움이 되었다.


## 4. 새로 배운 점
- 트리는 단순히 알고리즘 문제에서만 등장하는 자료구조가 아니라, HTML 문서 구조와 DOM 자체를 이해하는 데에도 직접 연결된다는 점을 배웠다. 브라우저가 HTML을 트리 형태로 해석한다는 사실을 알고 나니, 프론트엔드 화면도 결국 자료구조 위에서 움직인다는 감각이 생겼다.
- Virtual DOM은 실제 DOM을 그대로 다루기 전에 화면 상태를 자바스크립트 객체 트리로 표현한 중간 계층이라는 점을 알게 되었다. 이 구조 덕분에 브라우저와 분리된 상태에서 비교, 테스트, 변경 계산을 수행할 수 있고, 그 후에 필요한 부분만 실제 DOM에 반영할 수 있다는 흐름을 이해했다.
- Diff 알고리즘은 복잡하고 알 수 없는 프레임워크 내부 기술이 아니라, 두 트리를 비교해 "어디가 달라졌는가"를 찾는 비교 알고리즘이라는 관점으로 바라볼 수 있게 되었다. 추가, 삭제, 교체, 텍스트 변경, 속성 변경 같은 핵심 케이스로 나누어 생각하면 구현 범위를 설명 가능한 수준으로 관리할 수 있다는 점을 배웠다.
- DFS는 트리나 그래프 알고리즘 문제에서만 쓰는 탐색 기법이 아니라, DOM을 Virtual DOM으로 변환하는 과정에서도 매우 자연스럽게 사용된다는 점이 흥미로웠다. 현재 노드를 처리한 뒤 자식 노드를 재귀적으로 내려가는 흐름이 DOM 순회와 잘 맞아떨어졌고, 알고리즘에서 배운 개념이 실제 UI 구조 해석으로 이어질 수 있다는 점을 체감했다.
- BFS와 DFS는 단순히 둘 중 하나를 외워서 적용하는 것이 아니라, 문제의 성격에 따라 탐색 순서를 결정하는 도구라는 점을 다시 느꼈다. 가까운 경로를 먼저 확인해야 하는지, 한 경로를 끝까지 내려가며 구조를 파악해야 하는지에 따라 탐색 방식이 달라지고, 이 차이가 코드 구조에도 직접 반영된다는 점을 경험했다.
- 그래프 문제를 풀면서 방문 처리의 중요성을 배웠다. 노드 사이 연결만 보는 것이 아니라, 이미 방문한 노드를 어떻게 관리하느냐가 무한 반복을 막고, 탐색의 효율성과 정확도를 높이는 핵심이라는 점을 배웠다.  
- 위상정렬은 결국 선후관계가 있는 작업을 올바른 순서로 정리하는 방법이라는 점을 이해했다. 방향이 있고, 사이클이 없는 그래프에서만 가능하며, 의존성이 있는 작업 흐름을 다룰 때 사용할 수 있다는 점에서 의미가 있었다.
- 실제 DOM 조작 비용을 줄이기 위해 Virtual DOM과 Diff가 필요하다는 설명을 접하면서, 프론트엔드에서도 성능 문제를 자료구조와 알고리즘 관점으로 해석할 수 있다는 점을 배웠다. 또한 Patch를 단순한 DOM 조작이 아니라 Diff 결과를 실제 화면에 반영하는 별도의 단계로 분리해 생각해야 구조가 더 깔끔해지고 테스트 포인트도 명확해진다는 점을 이해했다.
- 히스토리 관리는 단순한 부가 기능이 아니라 상태 관리 관점에서 중요한 학습 포인트였다. 이전 상태와 현재 상태를 배열과 인덱스로 관리하는 방식은 UI에서도 결국 상태 전이가 핵심이라는 점을 보여주었고, 이를 통해 알고리즘 공부와 프론트엔드 구현이 생각보다 가까이 연결되어 있다는 점을 더 분명히 느낄 수 있었다.
- AI를 활용하더라도 무엇보다 이해 여부에 중점을 두는 것이 중요하다는 점을 다시 확인했다. 특히 Virtual DOM과 Diff처럼 개념 설명이 중요한 과제에서는, AI가 준 코드를 해석하면서 그 구조를 검증하고 추가적인 질문을 던지는 것이 훨씬 학습 효과가 컸다.
