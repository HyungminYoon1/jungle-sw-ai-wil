# Virtual DOM + Diff 구현 가이드

## 1. 문서 목적

이 문서는 4주차 수요 코딩회에서 요구하는 Virtual DOM과 Diff 알고리즘 구현을 위한 학습용 설계 문서이다.
목표는 React의 내부를 완전히 복제하는 것이 아니라, 브라우저 DOM을 읽어 Virtual DOM 트리로 바꾸고, 두 트리의 차이를 계산해 필요한 변경만 실제 DOM에 반영하는 흐름을 직접 구현해 보는 것이다.

또한 이번 구현은 4주차 학습 주제인 트리, 알고리즘, BFS, DFS와 밀접하게 연결된다.
특히 DOM을 트리로 이해하고 DFS 방식으로 순회하며, 두 트리를 비교하는 재귀 알고리즘을 설계하는 것이 핵심이다.

## 2. 요구사항 해석

수요 코딩회 요구사항을 구현 관점에서 정리하면 다음과 같다.

1. 브라우저의 실제 DOM을 읽어서 Virtual DOM 트리로 변환해야 한다.
2. 이전 Virtual DOM과 현재 Virtual DOM을 비교하는 Diff 알고리즘이 필요하다.
3. 변경된 부분만 실제 영역에 반영하는 Patch 과정이 있어야 한다.
4. 실제 영역과 테스트 영역, 그리고 Patch / 뒤로가기 / 앞으로가기 버튼이 있는 검증용 페이지가 필요하다.
5. Patch 이후 새 상태를 history에 저장하고, 뒤로가기와 앞으로가기로 이전 상태를 복원할 수 있어야 한다.
6. README 또는 발표에서 핵심 로직을 설명할 수 있을 정도로 구조가 명확해야 한다.

## 3. 전체 동작 흐름

전체 흐름은 아래와 같이 잡으면 구현과 설명이 모두 쉬워진다.

1. 페이지 로드 시 실제 영역의 샘플 HTML을 읽는다.
2. 이를 Virtual DOM 트리로 변환한다.
3. 변환된 트리를 이용해 테스트 영역을 초기 렌더링한다.
4. 사용자가 테스트 영역을 수정한다.
5. Patch 버튼을 누르면 테스트 영역의 현재 DOM을 다시 Virtual DOM으로 변환한다.
6. 이전 Virtual DOM과 새로운 Virtual DOM을 Diff 한다.
7. Diff 결과를 실제 영역에 Patch 한다.
8. 새 Virtual DOM을 history에 저장한다.
9. 뒤로가기 / 앞으로가기를 누르면 history의 특정 상태를 다시 실제 영역과 테스트 영역에 렌더링한다.

## 4. 추천 자료구조

학습용 구현에서는 Virtual DOM 노드를 단순한 객체로 두는 것이 좋다.

```js
{
  type: "element",
  tagName: "div",
  props: {
    id: "app",
    class: "container"
  },
  children: [
    {
      type: "element",
      tagName: "p",
      props: {},
      children: [
        {
          type: "text",
          value: "hello"
        }
      ]
    }
  ]
}
```

텍스트 노드는 별도 타입으로 분리하면 비교가 쉬워진다.

```js
{
  type: "text",
  value: "hello"
}
```

이렇게 설계하면 다음 장점이 있다.

- 태그 노드와 텍스트 노드를 명확히 구분할 수 있다.
- props 비교와 children 비교를 분리할 수 있다.
- 재귀 함수로 순회, Diff, Patch를 구현하기 쉽다.

## 5. DOM -> Virtual DOM 변환

이 단계는 실제 DOM 트리를 읽어 같은 구조의 자바스크립트 객체 트리로 바꾸는 과정이다.
주로 DFS 방식의 재귀 함수로 구현하면 자연스럽다.

예시 흐름은 다음과 같다.

1. 현재 DOM 노드의 타입을 확인한다.
2. 텍스트 노드면 `type: "text"` 객체를 만든다.
3. 요소 노드면 태그 이름과 속성을 읽는다.
4. 자식 노드들을 순회해 각각 Virtual DOM으로 변환한다.
5. 변환한 자식들을 children 배열에 담아 반환한다.

주의할 점은 다음과 같다.

- 공백만 있는 텍스트 노드를 유지할지 제거할지 기준을 먼저 정해야 한다.
- 비교와 렌더링이 쉬워지도록 불필요한 노드는 최대한 단순하게 정리하는 편이 좋다.
- attribute 수집 시 `id`, `class`, `value` 같은 주요 속성이 빠지지 않도록 확인해야 한다.

## 6. Diff 알고리즘의 핵심 케이스

학습용 구현에서는 모든 최적화보다 "설명 가능한 단순한 규칙"이 더 중요하다.
아래 5가지 케이스를 우선 처리하면 요구사항 설명과 데모가 쉬워진다.

### 1) 생성(Create)

이전 노드는 없고 새 노드만 있으면 새 노드를 추가한다.

### 2) 삭제(Remove)

이전 노드는 있는데 새 노드가 없으면 기존 노드를 제거한다.

### 3) 교체(Replace)

노드 타입이 다르거나 태그 이름이 다르면 기존 노드를 새 노드로 교체한다.
예를 들어 `div`가 `section`으로 바뀌면 통째로 교체하는 것이 가장 단순하다.

### 4) 텍스트 변경(Text Update)

텍스트 노드의 값이 다르면 텍스트 내용만 수정한다.

### 5) 속성 및 자식 변경(Update Children)

같은 태그라면 속성을 비교하고, 자식 노드 배열을 인덱스 기준으로 순회하면서 재귀적으로 Diff 한다.

이 5가지 케이스만으로도 대부분의 데모 시나리오를 설명할 수 있다.
수요 코딩회 안내문에 나온 "최소 변경을 찾기 위한 핵심 케이스"도 이 틀 안에서 정리하면 발표에 적합하다.

## 7. Patch 형식 예시

Diff 결과를 바로 DOM에 반영해도 되지만, 중간에 patch 목록을 만들면 검증과 디버깅이 쉬워진다.

예를 들면 다음과 같은 형식을 사용할 수 있다.

```js
[
  { type: "REPLACE", path: [0, 1], node: newVNode },
  { type: "TEXT", path: [2, 0], value: "updated text" },
  { type: "PROPS", path: [1], props: { class: "active" } },
  { type: "REMOVE", path: [3] },
  { type: "CREATE", path: [4], node: createdVNode }
]
```

여기서 `path`는 루트에서부터 몇 번째 자식으로 내려가는지를 나타내는 배열이다.
예를 들어 `[2, 0]`은 루트의 세 번째 자식, 그 아래 첫 번째 자식을 의미한다.

이 방식의 장점은 다음과 같다.

- 어떤 노드가 바뀌었는지 사람이 읽기 쉽다.
- 테스트 케이스를 작성할 때 예상 patch를 비교할 수 있다.
- 실제 DOM 반영 함수와 Diff 함수를 분리할 수 있다.

## 8. 실제 DOM 반영 전략

Patch 단계에서는 path를 이용해 실제 DOM의 대상 노드를 찾고, patch 타입에 따라 필요한 작업만 수행한다.

기본 전략은 다음과 같다.

1. `CREATE`: 새 DOM 노드를 만들어 해당 위치에 삽입한다.
2. `REMOVE`: 대상 노드를 제거한다.
3. `REPLACE`: 새 노드를 만들고 기존 노드를 교체한다.
4. `TEXT`: `textContent`를 수정한다.
5. `PROPS`: 변경된 속성만 추가, 수정, 삭제한다.

발표와 구현 단순성을 고려하면, 동일 레벨의 자식 비교는 우선 인덱스 기준으로 처리하는 것이 적절하다.
React처럼 key 기반 재배치까지 다루려 하면 복잡도가 크게 올라가므로, 이번 과제에서는 범위를 명확히 제한하는 편이 좋다.

## 9. History 설계

뒤로가기와 앞으로가기 기능을 위해서는 상태 이력을 관리해야 한다.
가장 단순한 방식은 `history` 배열과 현재 위치를 가리키는 `currentIndex`를 두는 것이다.

예시:

```js
const state = {
  history: [],
  currentIndex: -1
};
```

동작 규칙은 다음과 같이 정리할 수 있다.

1. 초기 Virtual DOM을 history에 저장한다.
2. Patch로 새 상태가 만들어지면 현재 인덱스 뒤쪽 이력을 잘라낸 뒤 새 상태를 push 한다.
3. 뒤로가기는 `currentIndex - 1`로 이동한다.
4. 앞으로가기는 `currentIndex + 1`로 이동한다.
5. 특정 이력으로 이동할 때는 그 시점의 Virtual DOM을 실제 영역과 테스트 영역 모두에 다시 렌더링한다.

이 구조는 브라우저 히스토리와 비슷한 개념이어서 설명하기 좋고, 상태 관리 연습에도 도움이 된다.

## 10. 테스트 포인트

수요 코딩회 결과물은 구현뿐 아니라 검증 과정도 중요하다.
따라서 아래와 같은 테스트 케이스를 준비하면 좋다.

- 텍스트만 변경되는 경우
- 속성 하나만 변경되는 경우
- 자식 노드 하나가 추가되는 경우
- 자식 노드 하나가 삭제되는 경우
- 태그가 통째로 교체되는 경우
- 여러 단계 깊이의 자식 노드가 변경되는 경우
- 뒤로가기 / 앞으로가기 시 실제 영역과 테스트 영역이 함께 동기화되는 경우

발표에서는 "어떤 입력에 대해 어떤 patch가 나왔는지", "실제 DOM이 어느 부분만 바뀌는지"를 함께 보여주면 설득력이 높아진다.

## 11. 구현 우선순위 제안

이번 수요 코딩회는 "학습"보다 "구현"이 우선이므로, 아래 순서로 작게 나누어 개발하는 것이 좋다.

1. 샘플 HTML을 Virtual DOM으로 변환하는 기능
2. Virtual DOM을 다시 DOM으로 렌더링하는 기능
3. 텍스트 변경만 처리하는 가장 단순한 Diff
4. 속성 변경과 자식 추가/삭제 처리
5. patch 목록 출력 또는 시각화
6. history와 뒤로가기 / 앞으로가기 기능
7. 엣지 케이스 테스트 및 데모 정리

이렇게 진행하면 빠르게 동작하는 최소 기능을 만든 뒤, 품질을 단계적으로 높일 수 있다.

## 12. 발표 때 설명하면 좋은 포인트

- DOM을 왜 트리로 봐야 하는가
- Virtual DOM을 어떤 객체 구조로 표현했는가
- DOM -> Virtual DOM 변환에 DFS를 어떻게 사용했는가
- Diff에서 어떤 5가지 핵심 케이스를 처리했는가
- 전체 DOM 재렌더링 대신 변경 부분만 반영하도록 어떻게 설계했는가
- history를 어떤 자료구조로 관리했는가
- 어떤 테스트로 구현을 검증했는가

## 13. 정리

이번 구현의 핵심은 React 개념을 추상적으로 이해하는 데서 끝나지 않고, DOM을 트리 자료구조로 변환하고 두 트리를 비교하는 알고리즘을 직접 만드는 것이다.

즉, 4주차의 알고리즘 학습 주제와 프론트엔드 구현이 하나로 연결되는 과제라고 볼 수 있다.
Virtual DOM, Diff, Patch, History를 각각 분리된 책임으로 구현하면 구조도 깔끔해지고, README 기반 발표에서도 설명하기 쉬운 결과물을 만들 수 있다.
