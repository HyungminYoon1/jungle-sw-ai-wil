# React 학습 가이드: Virtual DOM과 Diff 알고리즘

## 1. 이번 주 학습 목표와 React 주제의 연결

4주차 수요 코딩회의 핵심은 React의 대표 개념인 Virtual DOM과 Diff 알고리즘을 직접 구현해 보는 것이다.
이 주제는 단순히 React 사용법을 익히는 것이 아니라, 브라우저 DOM을 자료구조와 알고리즘 관점에서 이해하고 직접 다뤄 보는 학습과 연결된다.

이번 주에 학습하는 트리, 알고리즘, BFS, DFS, 위상정렬 중에서 특히 다음 개념이 직접적으로 연결된다.

- 트리: HTML 문서 구조와 Virtual DOM은 부모-자식 관계를 가지는 트리 구조이다.
- DFS: DOM 트리를 순회하며 노드를 읽고 Virtual DOM으로 변환할 때 자주 사용하는 방식이다.
- BFS: 같은 깊이의 노드들을 확인하거나 레벨 단위로 구조를 분석할 때 활용할 수 있다.
- 알고리즘: 두 트리를 비교해 최소 변경 지점을 찾는 과정 자체가 알고리즘 설계 문제이다.

즉, 이번 과제는 React를 배우는 과제이면서 동시에 트리 순회와 비교 알고리즘을 실제 동작하는 프로그램으로 연결하는 과제라고 볼 수 있다.

## 2. DOM과 Virtual DOM

### DOM이란 무엇인가

DOM(Document Object Model)은 브라우저가 HTML 문서를 메모리 위에서 트리 구조로 표현한 객체 모델이다.
브라우저는 HTML을 읽은 뒤 각 태그를 노드로 만들고, 이를 부모-자식 관계로 연결하여 문서 전체를 하나의 트리처럼 다룬다.

예를 들어 아래와 같은 HTML이 있으면:

```html
<div id="app">
  <h1>Hello</h1>
  <p>world</p>
</div>
```

브라우저는 이를 대략 다음과 같은 트리로 이해한다.

```text
div#app
├─ h1
│  └─ "Hello"
└─ p
   └─ "world"
```

### 실제 DOM이 느릴 수 있는 이유

실제 DOM은 브라우저 화면과 직접 연결되어 있다.
따라서 DOM을 자주 수정하면 브라우저가 레이아웃 계산과 화면 다시 그리기를 수행해야 하고, 이 과정에서 비용이 커질 수 있다.

핵심 이유는 다음과 같다.

- Reflow: 요소 크기나 위치가 바뀌면 브라우저가 레이아웃을 다시 계산해야 한다.
- Repaint: 색상, 배경, 텍스트 등 화면 표현이 바뀌면 다시 그려야 한다.
- 실제 DOM 조작은 단순한 데이터 변경이 아니라 브라우저 렌더링 파이프라인과 연결된다.

그래서 UI 변경이 잦은 환경에서는 "무엇이 바뀌었는지"를 먼저 계산하고, 정말 필요한 부분만 실제 DOM에 반영하는 방식이 중요하다.

### Virtual DOM이란 무엇인가

Virtual DOM은 실제 DOM을 그대로 조작하기 전에, 화면 구조를 자바스크립트 객체 형태의 트리로 표현한 가상 구조이다.
React는 화면의 이전 상태와 다음 상태를 각각 Virtual DOM으로 표현하고, 두 트리의 차이를 계산한 뒤 필요한 부분만 실제 DOM에 반영한다.

학습용으로 단순화하면 Virtual DOM 노드는 아래와 같이 생각할 수 있다.

```js
{
  type: "div",
  props: { id: "app" },
  children: [
    { type: "h1", props: {}, children: ["Hello"] },
    { type: "p", props: {}, children: ["world"] }
  ]
}
```

이 구조의 장점은 다음과 같다.

- 트리 구조이므로 순회와 비교가 쉽다.
- 브라우저 DOM과 분리된 순수 데이터이므로 테스트하기 쉽다.
- 변경 사항을 계산한 뒤 필요한 부분만 실제 DOM에 반영할 수 있다.

## 3. Diff 알고리즘이 필요한 이유

사용자가 화면 일부만 수정했는데 전체 DOM을 매번 다시 그리면 비효율적이다.
따라서 이전 Virtual DOM과 새로운 Virtual DOM을 비교해 변경된 부분만 찾는 과정이 필요하다.
이 비교 과정이 Diff 알고리즘이다.

Diff 알고리즘의 목적은 다음과 같다.

- 무엇이 바뀌었는지 찾는다.
- 바뀐 부분만 실제 DOM에 반영한다.
- 전체 렌더링 비용을 줄인다.

React 내부 구현은 매우 복잡하지만, 이번 수요 코딩회에서는 학습용으로 핵심 원리를 단순화한 Diff를 구현하는 것이 목표이다.
중요한 것은 "React를 똑같이 복제하는 것"이 아니라, 트리 비교와 최소 변경 반영의 핵심 아이디어를 이해하고 설명할 수 있는 수준까지 만드는 것이다.

## 4. 이번 과제에서 이해해야 할 Diff의 핵심 케이스

학습용 Diff 알고리즘에서는 아래와 같은 핵심 케이스를 우선적으로 다루면 좋다.

### 1) 노드가 새로 추가된 경우

이전 트리에는 없고 새 트리에는 있으면 해당 노드를 생성해야 한다.

### 2) 노드가 삭제된 경우

이전 트리에는 있었지만 새 트리에는 없으면 해당 노드를 제거해야 한다.

### 3) 태그 종류가 달라진 경우

예를 들어 `p`가 `div`로 바뀌었다면 같은 노드로 보기 어렵기 때문에 교체 처리하는 것이 단순하고 안전하다.

### 4) 텍스트 내용이 바뀐 경우

텍스트 노드의 값이 다르면 텍스트만 수정하면 된다.

### 5) 속성(attribute)이 바뀐 경우

`class`, `id`, `value` 같은 속성이 바뀌면 바뀐 속성만 반영하면 된다.

### 6) 자식 노드 구조가 바뀐 경우

자식 배열을 순서대로 비교하면서 추가, 삭제, 수정 여부를 판단한다.

이 과정은 결국 두 트리를 재귀적으로 내려가며 비교하는 문제이므로, 트리 순회와 재귀적 사고가 매우 중요하다.

## 5. 트리와 DFS/BFS가 어떻게 연결되는가

### DFS 관점

DOM을 Virtual DOM으로 변환할 때는 보통 현재 노드를 읽고 자식 노드로 내려가는 깊이 우선 탐색 방식이 자연스럽다.
즉, 하나의 노드를 처리한 뒤 그 자식들을 차례로 재귀 호출하면서 트리를 만든다.

### BFS 관점

BFS는 구현의 핵심이라기보다는 트리 구조를 레벨 단위로 이해할 때 유용하다.
예를 들어 특정 깊이의 노드 개수를 세거나, 구조를 시각적으로 검증할 때 도움이 된다.

### 알고리즘 관점

이번 구현은 다음 질문에 답하는 알고리즘 문제이다.

- 트리 형태의 데이터를 어떤 구조로 표현할 것인가
- 두 트리를 어떤 순서로 비교할 것인가
- 변경 사항을 어떤 형식으로 저장할 것인가
- 실제 DOM에는 어떤 순서로 반영할 것인가

따라서 이번 과제는 React 개념 학습과 알고리즘 문제 해결이 함께 들어 있는 좋은 예시이다.

## 6. React에서 이 개념이 중요한 이유

React가 널리 사용되는 이유 중 하나는 화면을 선언적으로 표현하고, 내부적으로는 변경점을 계산해 효율적으로 반영하기 때문이다.
개발자는 "최종적으로 어떤 UI가 되어야 하는지"를 중심으로 코드를 작성하고, React는 이전 상태와 다음 상태를 비교해 업데이트를 수행한다.

이번 과제에서 Virtual DOM과 Diff를 직접 구현해 보면 다음을 더 명확하게 이해할 수 있다.

- React가 왜 상태 변화 중심으로 UI를 다루는지
- React가 왜 트리 구조를 중요하게 생각하는지
- 실제 DOM을 바로 건드리는 방식과 어떤 차이가 있는지
- 변경 사항만 반영하는 접근이 왜 성능과 유지보수 측면에서 유리한지

## 7. 이번 주차에서 특히 가져가야 할 학습 포인트

- HTML 문서는 트리 구조이며, DOM도 결국 트리라는 점
- Virtual DOM은 실제 DOM을 다루기 위한 중간 표현이라는 점
- Diff 알고리즘은 두 트리를 비교해 최소 변경을 찾는 과정이라는 점
- DFS 기반 순회와 재귀가 구현의 핵심 도구라는 점
- 자료구조와 알고리즘 학습이 프론트엔드 구현과 직접 연결될 수 있다는 점

## 8. 정리

이번 수요 코딩회 주제는 React의 개념을 외우는 활동이 아니라, 브라우저의 DOM 구조를 이해하고 이를 Virtual DOM이라는 자료구조로 바꾼 뒤, 두 트리를 비교하는 알고리즘을 직접 구현해 보는 활동이다.

즉, 4주차의 학습 주제인 트리, 알고리즘, BFS, DFS를 실제 프론트엔드 문제에 연결해 보는 것이 핵심이다.
이 과정을 통해 React의 동작 원리를 더 깊이 이해할 수 있고, 발표에서도 "왜 이런 구조로 구현했는지"를 자료구조와 알고리즘 관점에서 설명할 수 있게 된다.
